# Chapter 2. Modeling Your Code: Communicating Sequential Processes

## The Difference Between Concurrency and Parallelism

핵심 정의:

> **동시성(Concurrency)은 코드의 속성이고, 병렬성(Parallelism)은 실행 중인 프로그램의 속성이다.**

우리는 병렬 코드를 작성하는 것이 아니라, **병렬로 실행되기를 희망하는 동시성 코드**를 작성한다. 실제 병렬 실행 여부는 런타임 환경(코어 수, OS 스케줄러, 하이퍼바이저 등)에 달려 있다.

```
단일 코어 머신:          멀티 코어 머신:
                         
 goroutine A  ──┐         goroutine A  ──────────
 goroutine B  ──┤         goroutine B  ──────────
 (인터리빙)     │         (진정한 병렬)
                ▼
 A─B─A─B─A─B─A─B         A─A─A─A─A
                          B─B─B─B─B
```

병렬성은 **시간 또는 컨텍스트의 함수**다. 5초 컨텍스트에서 각 1초짜리 작업 두 개는 병렬로 보이지만, 1초 컨텍스트에서는 순차로 보인다. 원자성과 마찬가지로 컨텍스트 정의가 먼저다.

**추상화 계층과 동시성의 어려움:**

```
어려움 증가 ↑
            │  OS 스레드 + 뮤텍스   ← 대부분 언어가 여기서 멈춤
            │  프로세스 경계
            │  별도 머신
어려움 감소 ↓
```

Go는 OS 스레드 위에 다른 레이어를 추가한 게 아니라 **고루틴과 채널로 대체**했다. 덕분에 추상화 수준은 높이면서 오히려 더 단순하게 동시성을 모델링할 수 있다.

## What Is CSP?

**CSP(Communicating Sequential Processes)**는 1978년 Tony Hoare(Charles Antony Richard Hoare)가 ACM에 발표한 논문이자, 그 논문에서 제안된 프로그래밍 기법이다.

Hoare의 핵심 주장: **입출력(I/O)은 프로그래밍, 특히 동시성 코드에서 간과된 기본 프리미티브다.**

당시 프로그래밍 연구는 순차 코드 구조화(goto 논쟁, OOP 태동)에 집중되어 있었고, 동시성은 주목받지 못했다. Hoare는 이를 바로잡기 위해 프로세스 간 통신을 언어 레벨 프리미티브로 격상시켰다.

### CSP의 핵심 개념

- **프로세스(Process):** 입력을 받아 출력을 생성하는 캡슐화된 로직 단위 (현대의 함수/고루틴에 해당)
- `!` 연산자: 프로세스에 입력 전송
- `?` 연산자: 프로세스에서 출력 읽기
- **가드 명령(Guarded Command):** `조건 → 동작` 형태. 좌변이 거짓이면 우변 실행 안 함 (Go의 `select` 기원)
- **대응(Correspondence):** 한 프로세스의 출력이 다른 프로세스의 입력으로 직접 흐르는 것

CSP 논문의 예시와 Go 채널의 대응:

| CSP 표기 | 설명 | Go 대응 |
|----------|------|---------|
| `cardreader?cardimage` | cardreader에서 값 읽기 | `cardimage := <-cardreader` |
| `lineprinter!lineimage` | lineprinter로 값 전송 | `lineprinter <- lineimage` |
| `*[c:character; west?c → east!c]` | west에서 읽어 east로 전달, 반복 | `for c := range west { east <- c }` |

CSP는 이후 **프로세스 대수(Process Calculus)**로 발전해 동시성 시스템을 수학적으로 모델링하고 정확성을 증명하는 수단이 되었다. Go는 CSP의 언어적 표현(고루틴, 채널, select)에서 직접 영감을 받았다.

## How This Helps You

고루틴이 OS 스레드보다 우수한 추상화인 이유:

**스레드 기반 언어에서 웹 서버를 설계할 때 고민:**
- 언어가 스레드를 네이티브 지원하는가, 라이브러리가 필요한가?
- 스레드 경계를 어디에 설정할 것인가?
- OS별 스레드 무게(weight)는?
- 스레드 풀의 최적 크기는?

이 질문들은 **문제 자체가 아니라 병렬성의 기술적 구현**에 관한 것이다.

**Go에서의 자연스러운 매핑:**

```go
// 문제를 자연스럽게 표현: 각 연결 = 하나의 고루틴
for conn := range listener.Accept() {
    go handleConnection(conn) // 스레드 풀? 크기 제한? 고민 불필요
}
```

각 사용자 연결을 고루틴 하나로 표현하면 문제 도메인(사용자 세션)이 코드 구조에 1:1로 매핑된다. 이것이 가능한 이유는 Go의 약속: **고루틴은 가볍다.**

**고루틴 멀티플렉싱의 추가 이점:**

```
고루틴 (수천~수백만)
    │ Go 런타임이 자동 스케줄링
    ▼
OS 스레드 (수십~수백)
    │
    ▼
CPU 코어
```

- I/O 블록된 고루틴을 감지해 OS 스레드를 다른 고루틴으로 재할당 → CPU 활용도 향상
- 런타임 최적화가 프로그램 구조 변경 없이 자동 적용됨 (관심사 분리)
- 고루틴이 세밀할수록 암달의 법칙 상 병렬화 비율이 높아짐 → 자연스럽게 성능 향상

**채널과 select의 조합성:**

- 채널은 다른 채널과 조합 가능 (뮤텍스 조합은 악몽)
- `select`로 여러 채널을 균일한 무작위 방식으로 처리
- 타임아웃, 취소, 서브시스템 메시지를 채널로 자연스럽게 통합

## Go's Philosophy on Concurrency

Go는 CSP 스타일(채널)과 메모리 접근 동기화(sync 패키지) 모두 지원한다. 어느 것이 더 **표현적이고 단순한가**를 기준으로 선택한다.

### 의사결정 트리 (Decision Tree)

```
동시성 코드를 어떻게 작성할까?
│
├─ 데이터 소유권을 이전하려는가?
│   └─ YES → 채널 사용
│       (생산자-소비자 분리, 버퍼 채널로 인메모리 큐)
│
├─ 구조체의 내부 상태를 보호하려는가?
│   └─ YES → sync 프리미티브 사용
│       (내부 구현 세부사항 은닉, 호출자에게 복잡성 노출 금지)
│
├─ 여러 로직 조각을 조율해야 하는가?
│   └─ YES → 채널 사용
│       (채널은 조합 가능, 분산된 락은 악몽)
│
└─ 성능 크리티컬 섹션인가?
    └─ YES → sync 프리미티브 고려
        (채널은 내부적으로 메모리 동기화 사용 → 항상 뮤텍스보다 느림)
        (단, 프로파일링으로 병목 확인 후 적용)
```

### 내부 상태 보호: sync 프리미티브가 적합한 예

```go
type Counter struct {
    mu    sync.Mutex
    value int
}

func (c *Counter) Increment() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.value++
}
```

`Increment` 호출은 원자적으로 간주할 수 있다. **락은 타입 내부에 캡슐화**되어 있고, 호출자는 동기화를 신경 쓸 필요가 없다.

> 경고: 락을 타입 외부로 노출하면 적신호. 렉시컬 스코프를 최소화할 것.

### 채널 vs 뮤텍스 요약

| 상황 | 권장 |
|------|------|
| 데이터 소유권 이전 | 채널 |
| 구조체 내부 상태 보호 | sync (Mutex, RWMutex) |
| 여러 고루틴 조율 | 채널 + select |
| 성능 크리티컬 (프로파일링 확인 후) | sync |
| 일회성 초기화 | sync.Once |
| 고루틴 그룹 대기 | sync.WaitGroup |

### Go 동시성 철학 요약

> **단순함을 목표로 한다. 가능하면 채널을 사용한다. 고루틴을 무료 자원처럼 취급한다.**

OS 스레드에서 유효했던 패턴들(스레드 풀 등)은 고루틴 세계에서 대부분 불필요하다. 고루틴으로 문제 도메인을 직접 모델링하고, 상한에 걱정하기보다 구조 재설계를 먼저 고려하라.
