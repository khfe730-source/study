# 18장. 더티 플래그 (Dirty Flag)

> **분류**: 최적화 패턴 (Optimization Patterns)  
> **별칭**: 더티 비트 (Dirty Bit)

---

## 의도 (Intent)

결과가 필요할 때까지 불필요한 작업을 미뤄서 회피한다.

---

## 동기 (Motivation)

### 씬 그래프와 트랜스폼

많은 게임은 **씬 그래프(scene graph)**를 사용한다. 씬 그래프는 월드의 모든 오브젝트를 담는 계층적 자료구조다. 렌더링 엔진은 이를 통해 화면에 무엇을 어디에 그릴지 결정한다.

씬 그래프의 핵심은 **계층 구조(hierarchy)**다. 오브젝트는 부모를 가질 수 있으며, 자신의 트랜스폼(위치/회전/스케일)은 부모 기준의 **로컬 트랜스폼(local transform)**으로 표현된다.

```
해적선(Ship)
  └── 돛대 꼭대기 감시초소(Crow's Nest)
        └── 해적(Pirate)
              └── 앵무새(Parrot)
```

배가 움직이면 그 위의 모든 것이 자동으로 따라온다. 각 오브젝트는 부모 기준으로 상대적 위치를 가지기 때문이다.

렌더링을 위해서는 **월드 트랜스폼(world transform)** — 씬 전체 기준의 절대 위치 — 이 필요하다.

### 월드 트랜스폼 계산

특정 오브젝트의 월드 트랜스폼은 루트부터 해당 오브젝트까지 부모 체인을 따라 트랜스폼을 모두 합산해서 구한다.

```
앵무새의 월드 트랜스폼
= Ship.local × Crow's Nest.local × Pirate.local × Parrot.local
```

---

### 문제 1: 매 프레임 전체 재계산 (순진한 접근)

가장 단순한 방법은 렌더링할 때마다 씬 그래프를 순회하며 월드 트랜스폼을 즉석 계산하는 것이다. 그러나 이는 **정적인 오브젝트(움직이지 않는 지형 등)도 매 프레임 재계산**하는 낭비를 초래한다.

---

### 문제 2: 변경 즉시 전파 (eager recalculation)

반대로 로컬 트랜스폼이 바뀔 때마다 즉시 월드 트랜스폼을 갱신하면, **한 프레임에 여러 트랜스폼이 변경될 때 같은 오브젝트를 여러 번 재계산**하게 된다.

예시: 한 프레임에 Ship, Crow's Nest, Pirate, Parrot 네 오브젝트의 로컬 트랜스폼이 모두 변경되는 경우:

```
변경 순서         재계산 대상 (eager 방식)
─────────────────────────────────────────────────
Ship 이동         Ship, Crow's Nest, Pirate, Parrot  (4회)
Crow's Nest 흔들림  Crow's Nest, Pirate, Parrot       (3회)
Pirate 기울기      Pirate, Parrot                    (2회)
Parrot 점프        Parrot                            (1회)
─────────────────────────────────────────────────
합계              총 10회 계산 (실제 필요 계산: 4회)
★ Parrot 월드 트랜스폼 = 4번 계산, 실제 렌더링에 쓰이는 건 마지막 1번뿐
```

6번의 불필요한 연산이 발생한다.

---

### 해결: 지연된 재계산 (Deferred Recalculation)

로컬 트랜스폼 변경과 월드 트랜스폼 갱신을 **분리**한다. 변경 사항을 축적하고, 실제로 월드 트랜스폼이 필요한 시점(렌더링 직전)에 한 번만 계산한다.

이를 위해 각 노드에 **더티 플래그(dirty flag)** 하나를 추가한다.

- 로컬 트랜스폼이 변경되면 → 플래그를 **세운다(set)**
- 월드 트랜스폼이 필요할 때 → 플래그가 세워져 있으면 계산하고 **내린다(clear)**
- 플래그가 내려져 있으면 → 캐시된 이전 월드 트랜스폼을 그대로 사용

```
Dirty Flag이 적용된 동일 예시
─────────────────────────────────────────────────
변경 단계: Ship, Crow's Nest, Pirate, Parrot 각각 dirty 마킹만
렌더링 시: 각 오브젝트의 월드 트랜스폼 정확히 1회씩 계산
─────────────────────────────────────────────────
합계      총 4회 계산 → 최적
```

단 하나의 비트(bit)로 이것이 가능하다:
- 부모 체인에서 중복 재계산 제거
- 변경되지 않은 오브젝트의 재계산 완전 생략
- 렌더링 전에 제거된 오브젝트는 계산 자체를 건너뜀

---

## 패턴 요약

**기본 데이터(primary data)**가 시간에 따라 변한다. **파생 데이터(derived data)**는 비용이 큰 과정을 통해 기본 데이터로부터 결정된다. "더티(dirty)" 플래그는 파생 데이터가 기본 데이터와 동기화되지 않았음을 추적한다. 기본 데이터가 변경되면 플래그를 세운다. 파생 데이터가 필요할 때 플래그가 세워져 있으면 재처리하고 플래그를 내린다. 세워져 있지 않으면 이전 캐시된 파생 데이터를 사용한다.

---

## 언제 사용하는가

더티 플래그는 두 종류의 작업에 적용된다: **계산(calculation)**과 **동기화(synchronization)**. 두 경우 모두 기본 데이터에서 파생 데이터를 생성하는 과정이 시간이 많이 걸리거나 비용이 크다.

**적합한 조건**:

1. **기본 데이터가 파생 데이터 사용 빈도보다 더 자주 변경된다.** 기본 데이터 변경 후 파생 데이터가 사용되기 전에 또 변경된다면, 중간 계산을 건너뛸 수 있다.

2. **증분 업데이트(incremental update)가 어렵다.** 예를 들어 합산 같은 경우 "현재까지의 합계"를 유지하면서 변경 사항을 더하거나 빼는 것이 더 간단하다. 이런 경우 더티 플래그보다 실시간 누적 업데이트가 낫다.

---

## 주의사항 (Keep in Mind)

### 1. 지연이 너무 길어지면 비용이 한꺼번에 발생한다

파생 데이터가 필요한 순간 한 번에 계산하게 되는데, 그 계산이 무거우면 그 순간 눈에 띄는 **일시 정지(pause)**가 발생한다.

텍스트 에디터의 "저장되지 않은 변경사항" 표시도 더티 플래그다. 애플리케이션이 종료되거나 전원이 끊기면 저장이 안 된 채 소실될 수 있다. 자동 저장(auto-save)은 이 문제를 보완하는 전략이다.

```
저장 전략의 스펙트럼
───────────────────────────────────────────────────
매번 즉시 저장  ←────────────────→  종료 시 한 번 저장
(I/O 낭비)                        (데이터 손실 위험)
             ↑
         자동 저장 (중간 타협점)
───────────────────────────────────────────────────
```

### 2. 기본 데이터가 변경될 때마다 반드시 플래그를 세워야 한다

캐시 무효화(cache invalidation) 문제다. 기본 데이터를 변경하는 모든 경로에서 플래그 세팅을 빠뜨리면 **낡은(stale) 데이터**를 사용하는 버그가 발생한다. 디버깅하기 매우 어렵다.

해결책: **기본 데이터 변경을 인터페이스 뒤에 캡슐화**한다. 모든 변경이 하나의 좁은 API를 통하게 하면 거기서 플래그를 세워 누락을 방지할 수 있다.

### 3. 이전 파생 데이터를 메모리에 유지해야 한다

더티 플래그 패턴을 쓰지 않으면 필요할 때마다 계산하고 버릴 수 있다. 이 패턴을 쓰면 캐시된 결과를 항상 메모리에 보관해야 한다. **속도를 위해 메모리를 트레이드**하는 것이다.

## 샘플 코드 (Sample Code)

### 트랜스폼 클래스

```cpp
class Transform
{
public:
    static Transform origin();

    // 두 트랜스폼을 합산 (행렬 곱셈)
    Transform combine(Transform& other);
};
```

### 씬 그래프 노드 (최적화 전)

```cpp
class GraphNode
{
public:
    GraphNode(Mesh* mesh)
    : mesh_(mesh),
      local_(Transform::origin())
    {}

private:
    Transform  local_;
    Mesh*      mesh_;

    GraphNode* children_[MAX_CHILDREN];
    int        numChildren_;
};
```

비최적화 순회 — 매 프레임 모든 노드 재계산:

```cpp
// ❌ 매 프레임 모든 노드의 world transform 계산
void GraphNode::render(Transform parentWorld)
{
    Transform world = local_.combine(parentWorld);  // 항상 계산

    if (mesh_) renderMesh(mesh_, world);

    for (int i = 0; i < numChildren_; i++)
    {
        children_[i]->render(world);
    }
}

// 루트에서 시작
graph_->render(Transform::origin());
```

### 더티 플래그 적용 (최적화 후)

#### 노드 클래스에 캐시와 플래그 추가

```cpp
class GraphNode
{
public:
    GraphNode(Mesh* mesh)
    : mesh_(mesh),
      local_(Transform::origin()),
      dirty_(true)    // 처음에는 항상 dirty — 아직 계산된 적 없으므로
    {}

    void setTransform(Transform local)
    {
        local_ = local;
        dirty_ = true;  // ← 반드시 플래그를 세워야 한다!
        // 자식 노드의 dirty_는 여기서 전파하지 않는다 (render 시 처리)
    }

private:
    Transform  local_;
    Transform  world_;   // 캐시된 월드 트랜스폼
    bool       dirty_;   // 세워져 있으면 "out of date"
    Mesh*      mesh_;

    GraphNode* children_[MAX_CHILDREN];
    int        numChildren_;
};
```

#### 최적화된 렌더 순회

```cpp
// dirty 파라미터: 부모 체인 중 하나라도 dirty였으면 true
void GraphNode::render(Transform parentWorld, bool dirty)
{
    // 부모가 dirty였거나 자신이 dirty면 재계산 필요
    dirty |= dirty_;

    if (dirty)
    {
        world_ = local_.combine(parentWorld);  // 재계산
        dirty_ = false;                        // 플래그 클리어
    }
    // dirty가 false면 world_의 이전 값을 그대로 사용

    if (mesh_) renderMesh(mesh_, world_);

    for (int i = 0; i < numChildren_; i++)
    {
        // 자식에게 현재 노드의 dirty 상태를 전파
        children_[i]->render(world_, dirty);
    }
}

// 루트에서 시작 (초기 dirty = false)
graph_->render(Transform::origin(), false);
```

핵심 아이디어 시각화:

```
setTransform() 호출 시                 render() 순회 시
─────────────────────────────────────────────────────────
Ship:       dirty_ = true             dirty 전파: true
Crow's Nest: dirty_ = true            dirty 전파: true (부모가 dirty)
Pirate:     dirty_ = true             dirty 전파: true (부모가 dirty)
Parrot:     dirty_ = true             dirty 전파: true (부모가 dirty)
─────────────────────────────────────────────────────────
결과: 각 노드 world_ 계산 정확히 1회

(만약 Ship만 움직이고 나머지는 그대로라면)
Ship:       dirty_ = true             → 재계산
Crow's Nest: dirty_ = false           → dirty 전파로 재계산 (부모 dirty)
Pirate:     dirty_ = false            → dirty 전파로 재계산
Parrot:     dirty_ = false            → dirty 전파로 재계산
```

`dirty` 파라미터를 전파하는 영리한 트릭 덕분에 `setTransform()`에서 자식들의 `dirty_`를 재귀적으로 세우는 비용을 없앴다. 대신 `render()` 시 부모의 dirty 상태를 전파하는 방식으로 동일한 결과를 낸다.

> **주의**: 이 트릭은 `render()`만이 최신 월드 트랜스폼을 필요로 한다는 가정 하에 동작한다. 다른 곳에서도 월드 트랜스폼이 필요하다면 다른 전략이 필요하다.

## 설계 결정 (Design Decisions)

### 언제 더티 플래그를 청소할 것인가?

#### 옵션 A: 결과가 필요할 때 (on demand)

```
기본 데이터 변경 → dirty 설정 → ... 여러 변경 ... → 렌더링 시 계산
```

- **장점**: 결과가 전혀 사용되지 않으면 계산 자체를 건너뜀. 기본 데이터가 파생 데이터 사용보다 훨씬 자주 변경될 때 효과 극대화.
- **단점**: 계산이 오래 걸리면 그 순간 눈에 띄는 지연이 생긴다.

#### 옵션 B: 잘 정의된 체크포인트 (checkpoint)

특정 시점 — 항구 입항, 로딩 화면, 컷신 — 에 처리한다.

- **장점**: 플레이어 경험에 영향을 주지 않고 일을 처리할 수 있다. (로딩 화면 뒤에 숨기기)
- **단점**: 플레이어가 체크포인트에 도달하지 못하면 예상보다 오래 지연될 수 있다.

#### 옵션 C: 백그라운드 처리 (background)

첫 변경 시 타이머를 시작하고, 타이머가 만료되면 처리한다. (GC의 점진적 수집과 유사)

```
첫 변경 → 타이머 시작 → 변경 누적 → 타이머 만료 → 백그라운드 처리
```

- **장점**: 처리 빈도 튜닝 가능. 플레이어는 게임을 계속 진행.
- **단점**: 동시성(concurrency) 지원 필요. 기본 데이터의 동시 수정에 대한 스레드 안전성 고려 필요.

---

### 더티 추적의 세분화 수준 (Granularity)

해적선 커스터마이징 게임에서 수정된 부분만 서버에 전송하는 시나리오:

#### 세밀한 추적 (Fine-grained)

```
[갑판 1]                [갑판 2]
 [판자 1-1] dirty=false  [판자 2-1] dirty=true
 [판자 1-2] dirty=false  [판자 2-2] dirty=false
 [판자 1-3] dirty=true   ...
```

- **장점**: 실제 변경된 데이터만 정확히 전송
- **단점**: 더티 플래그 저장 메모리 증가, 처리 횟수는 많아지지만 각 처리량은 적음

#### 거친 추적 (Coarse-grained)

```
[갑판 1] dirty=false
[갑판 2] dirty=true   ← 이 갑판의 단 하나의 판자만 바뀌었어도 전체 전송
```

- **장점**: 플래그 저장 메모리 절약. 처리 고정 비용(메타데이터 등) 절감.
- **단점**: 변경되지 않은 데이터도 함께 전송/처리

## 전체 흐름 요약

```
더티 플래그 패턴 동작 흐름
──────────────────────────────────────────────────────────
기본 데이터        더티 플래그        파생 데이터
(primary data)    (dirty flag)      (derived data)
──────────────────────────────────────────────────────────
변경됨      ──→   SET (true)

           ...여러 번 변경 가능...

결과 필요?
  └─ YES, flag SET   ──→   재계산 실행   ──→   CLEAR (false)
  └─ YES, flag CLEAR ──→   캐시 값 사용  (계산 생략)
──────────────────────────────────────────────────────────
핵심: "파생 데이터가 실제로 필요할 때까지 계산을 미룬다"
──────────────────────────────────────────────────────────
```

## 다른 패턴과의 관계

- **데이터 지역성 패턴**: `if (dirty)` 분기가 브랜치 예측 실패(branch misprediction)를 일으킬 수 있다는 점을 인식해야 한다.
- **오브젝트 풀 패턴**: 물리 엔진에서 sleep/awake 오브젝트 관리에 유사한 패턴이 사용된다.
