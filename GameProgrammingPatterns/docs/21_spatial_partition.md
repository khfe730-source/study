# 20장. 공간 분할 (Spatial Partition)

> **분류**: 최적화 패턴 (Optimization Patterns)

---

## 의도 (Intent)

오브젝트를 위치 기반으로 정리된 자료구조에 저장함으로써 특정 위치의 오브젝트를 효율적으로 찾는다.

---

## 동기 (Motivation)

게임 월드에서 오브젝트들은 공간 속에 존재하며, 엔진은 끊임없이 "이 위치 근처에 무엇이 있는가?"를 묻는다. 이 질문은 물리 충돌, 오디오 거리 감쇠, 근접 채팅, 근접 전투 등 다양한 시스템에서 매 프레임 반복된다.

### 단순 무식한 접근: O(n²)

RTS 게임에서 수백 명의 유닛 간 근접 전투를 처리한다고 하자.

```cpp
// ❌ 나이브한 방법: 모든 쌍을 비교
void handleMelee(Unit* units[], int numUnits)
{
    for (int a = 0; a < numUnits - 1; a++)
    {
        for (int b = a + 1; b < numUnits; b++)
        {
            if (units[a]->position() == units[b]->position())
            {
                handleAttack(units[a], units[b]);
            }
        }
    }
}
```

이중 중첩 루프 → **O(n²)** 복잡도. 유닛이 늘어날수록 비교 횟수가 제곱으로 증가한다.

```
유닛 수    비교 횟수
  10        45
 100      4,950
1000    499,500
```

### 핵심 통찰: 공간 정렬

1D 전선(battleline)이라면 유닛을 위치순으로 정렬해두고 **이진 탐색**으로 근처 유닛을 O(log n)에 찾을 수 있다. 이 아이디어를 2D/3D 공간으로 확장한 것이 공간 분할 패턴이다.

```
1D 예시: 정렬된 배열
위치:  [1] [3] [5] [7] [9] [12] [15] [18]
         ↑
       여기서 거리 3 이내를 찾고 싶다면
       이진 탐색 → O(log n)
```

---

## 패턴 요약

각 오브젝트는 공간상의 위치를 가진다. 오브젝트를 위치별로 정리하는 공간 자료구조에 저장한다. 이 자료구조를 통해 특정 위치 근처의 오브젝트를 효율적으로 쿼리할 수 있다. 오브젝트의 위치가 바뀌면, 자료구조도 함께 갱신한다.

---

## 언제 사용하는가

- 게임 오브젝트(살아있는 유닛)와 정적 지형/아트 저장에 모두 사용
- 복잡한 게임은 콘텐츠 종류별로 여러 공간 분할을 함께 사용
- **O(n) 또는 O(n²) 위치 쿼리가 성능 병목이 될 때** 도입

---

## 주의사항 (Keep in Mind)

- **n이 충분히 클 때만 의미 있다.** 오브젝트가 적으면 자료구조 오버헤드가 더 클 수 있다.
- **위치가 자주 바뀌는 오브젝트는 관리 비용이 추가된다.** 이동할 때마다 자료구조를 갱신해야 한다.
- **메모리를 추가로 사용한다.** 공간 분할 자료구조 자체의 북키핑(bookkeeping) 메모리가 필요하다.

---

## 샘플 코드: 고정 그리드 (Fixed Grid)

가장 단순한 공간 분할 구조. 전장을 고정 크기의 셀(cell)로 나눈 격자판이다.

```
전장 그리드 (10×10)
┌───┬───┬───┬───┬───┐
│   │ ★ │   │   │   │
├───┼───┼───┼───┼───┤
│ ★ │   │ ★ │   │ ★ │
├───┼───┼───┼───┼───┤
│   │   │ ★ │   │   │
├───┼───┼───┼───┼───┤
│ ★ │   │   │ ★ │   │
└───┴───┴───┴───┴───┘
각 셀은 그 안에 있는 유닛들의 링크드 리스트를 가진다.
전투 처리는 같은 셀 안의 유닛들 사이에서만!
```

### Unit 클래스

```cpp
class Unit
{
    friend class Grid;

public:
    Unit(Grid* grid, double x, double y)
    : grid_(grid), x_(x), y_(y),
      prev_(NULL), next_(NULL)
    {
        grid_->add(this);  // 생성 시 자동으로 그리드에 등록
    }

    void move(double x, double y);

private:
    double x_, y_;
    Grid*  grid_;
    Unit*  prev_;  // 같은 셀 내 이중 연결 리스트
    Unit*  next_;
};
```

각 셀 내에서 유닛들을 **이중 연결 리스트(doubly linked list)**로 관리한다. 이유는 O(1) 삽입/삭제를 위해서다.

### Grid 클래스

```cpp
class Grid
{
public:
    Grid()
    {
        // 모든 셀을 빈 리스트로 초기화
        for (int x = 0; x < NUM_CELLS; x++)
            for (int y = 0; y < NUM_CELLS; y++)
                cells_[x][y] = NULL;
    }

    static const int NUM_CELLS = 10;
    static const int CELL_SIZE = 20;  // 월드 좌표 기준 셀 크기

private:
    Unit* cells_[NUM_CELLS][NUM_CELLS];  // 각 셀의 유닛 리스트 헤드
};
```

### 유닛 추가: Grid::add()

```cpp
void Grid::add(Unit* unit)
{
    // 월드 좌표 → 셀 인덱스 변환
    int cellX = (int)(unit->x_ / Grid::CELL_SIZE);
    int cellY = (int)(unit->y_ / Grid::CELL_SIZE);

    // 해당 셀 리스트의 맨 앞에 삽입
    unit->prev_ = NULL;
    unit->next_ = cells_[cellX][cellY];
    cells_[cellX][cellY] = unit;

    if (unit->next_ != NULL)
        unit->next_->prev_ = unit;
}
```

### 전투 처리: handleMelee()

```cpp
// 모든 셀 순회
void Grid::handleMelee()
{
    for (int x = 0; x < NUM_CELLS; x++)
        for (int y = 0; y < NUM_CELLS; y++)
            handleCell(cells_[x][y]);
}

// 셀 내부의 유닛 쌍만 비교 (O(k²), k = 셀 내 유닛 수)
void Grid::handleCell(Unit* unit)
{
    while (unit != NULL)
    {
        Unit* other = unit->next_;
        while (other != NULL)
        {
            if (unit->x_ == other->x_ &&
                unit->y_ == other->y_)
            {
                handleAttack(unit, other);
            }
            other = other->next_;
        }
        unit = unit->next_;
    }
}
```

전체 구조는 O(n²)처럼 보이지만, 각 셀의 유닛 수 k가 매우 작기 때문에 실질적 복잡도는 훨씬 낮다. n개의 유닛이 c²개의 셀에 균등 분포하면 각 셀에 n/c²개, 전체 비교는 **O(n²/c²)** 수준으로 줄어든다.

### 유닛 이동: Grid::move()

유닛이 셀 경계를 넘으면 이전 셀 리스트에서 제거하고 새 셀에 추가해야 한다.

```cpp
void Unit::move(double x, double y)
{
    grid_->move(this, x, y);
}

void Grid::move(Unit* unit, double x, double y)
{
    int oldCellX = (int)(unit->x_ / Grid::CELL_SIZE);
    int oldCellY = (int)(unit->y_ / Grid::CELL_SIZE);

    int cellX = (int)(x / Grid::CELL_SIZE);
    int cellY = (int)(y / Grid::CELL_SIZE);

    unit->x_ = x;
    unit->y_ = y;

    // 셀이 바뀌지 않았으면 위치만 갱신하고 끝
    if (oldCellX == cellX && oldCellY == cellY) return;

    // 이전 셀의 연결 리스트에서 제거
    if (unit->prev_ != NULL)
        unit->prev_->next_ = unit->next_;
    if (unit->next_ != NULL)
        unit->next_->prev_ = unit->prev_;
    if (cells_[oldCellX][oldCellY] == unit)
        cells_[oldCellX][oldCellY] = unit->next_;

    // 새 셀에 추가
    add(unit);
}
```

이중 연결 리스트를 쓰는 이유가 여기서 드러난다. 유닛의 이전/다음 포인터만 조작하면 **O(1)**로 리스트에서 제거할 수 있다.

### 공격 거리(Attack Distance) 처리

실제 게임에서는 정확히 같은 위치가 아니라 **공격 범위** 내에 있으면 전투가 성립한다.

```cpp
if (distance(unit, other) < ATTACK_DISTANCE)
{
    handleAttack(unit, other);
}
```

그런데 이렇게 하면 **서로 다른 셀에 있지만 경계 근처의 유닛**을 놓칠 수 있다.

```
┌──────────┬──────────┐
│          │          │
│    A ●───┼───● B    │ A와 B가 다른 셀에 있지만
│          │          │ 서로의 공격 범위 안에 있음
└──────────┴──────────┘
```

해결책: **인접한 절반의 셀도 함께 검사**한다 (중복 처리를 피하기 위해 4방향만).

```cpp
void Grid::handleCell(int x, int y)
{
    Unit* unit = cells_[x][y];
    while (unit != NULL)
    {
        // 같은 셀 내 유닛과 비교
        handleUnit(unit, unit->next_);

        // 인접 셀 4개도 검사 (나머지 4개는 반대편에서 처리)
        if (x > 0 && y > 0) handleUnit(unit, cells_[x-1][y-1]);
        if (x > 0)          handleUnit(unit, cells_[x-1][y]);
        if (y > 0)          handleUnit(unit, cells_[x][y-1]);
        if (x > 0 && y < NUM_CELLS - 1)
                            handleUnit(unit, cells_[x-1][y+1]);

        unit = unit->next_;
    }
}
```

```
인접 셀 검사 패턴 (U = 현재 셀, X = 검사하는 셀)
┌───┬───┬───┐
│ X │ X │   │
├───┼───┼───┤
│ X │ U │   │  ← 4방향만 검사 (왼쪽, 아래쪽 절반)
├───┼───┼───┤      나머지 4방향은 그 셀이 U가 될 때 처리됨
│ X │ X │   │
└───┴───┴───┘
```

> **주의**: 공격 거리가 셀 크기보다 크면 더 많은 인접 셀을 검사해야 한다.

## 설계 결정 (Design Decisions)

### 1. 분할이 계층적인가, 평탄한가?

| 구분 | 평탄한 분할 (Flat) | 계층적 분할 (Hierarchical) |
|------|-------------------|--------------------------|
| **예시** | Grid, 버킷 정렬 | Quadtree, BSP, k-d tree, BVH |
| **구현 복잡도** | 단순 | 복잡 |
| **메모리 사용** | 고정 (빈 공간도 할당) | 적응형 (빈 공간 효율적) |
| **오브젝트 이동** | 빠름 | 계층 여러 레이어 갱신 필요 |
| **빈 공간 처리** | 비효율 (빈 셀도 순회) | 효율적 (빈 공간 = 단일 파티션) |
| **밀집 지역 처리** | 비효율 (하나의 셀에 몰림) | 효율적 (재귀 세분화) |

```
평탄한 그리드 (모든 오브젝트가 한 구석에 몰린 경우)
┌───┬───┬───┬───┬───┐
│★★★│★★ │   │   │   │  ← 좌상단 셀들이 과부하
├───┼───┼───┼───┼───┤
│★  │   │   │   │   │     나머지 셀은 비어서 낭비
├───┼───┼───┼───┼───┤
│   │   │   │   │   │
└───┴───┴───┴───┴───┘

계층적 분할 (Quadtree, 동일 상황)
┌──────────────┬──────┬──────┐
│              │  ★★  │ ★★★  │  ← 밀집 지역만 재귀 세분화
│    (비어있음) ├──┬───┼──┬───┤
│              │★ │   │  │   │
└──────────────┴──┴───┴──┴───┘
```

### 2. 분할이 오브젝트 집합에 의존하는가?

#### 오브젝트 독립적 분할 (Object-Independent)

```
예: 고정 그리드
- 오브젝트와 무관하게 격자 경계가 미리 정해짐
- 오브젝트 추가: O(1) (해당 셀에 삽입)
- 오브젝트 이동: 빠름 (셀 경계만 확인)
- 단점: 오브젝트 분포가 불균형하면 성능 저하
```

#### 오브젝트 의존적 분할 (Object-Dependent)

```
예: BSP, k-d tree, BVH
- 오브젝트 분포를 보고 분할 경계를 결정
- 각 파티션이 균등한 수의 오브젝트를 가짐
- 장점: 균형 잡힌 성능, 일관된 프레임 타임
- 단점: 오브젝트 집합 변화 시 재구성 비용 큼
- 주로 정적 지형/아트에 적합
```

#### 절충안: Quadtree (계층은 고정, 세분화는 적응형)

```
Quadtree 세분화 과정

1단계: 전체 공간 하나의 파티션
┌──────────┐
│  ★ ★ ★  │  (MAX 오브젝트 초과)
│  ★ ★     │
└──────────┘
         ↓ 4개로 분할

2단계:
┌─────┬─────┐
│ ★ ★ │  ★  │  (좌상단 초과)
├─────┼─────┤
│ ★ ★ │     │
└─────┴─────┘
         ↓ 좌상단만 재분할

3단계:
┌──┬──┬─────┐
│★★│  │  ★  │
├──┼──┼─────┤
│  │★★│     │
└──┴──┴─────┘
```

Quadtree의 특징:
- 분할 경계는 항상 정중앙 (고정)
- 밀집 지역만 재귀 세분화 (적응형)
- 오브젝트 추가/제거/이동 모두 효율적
- 균형 보장

### 3. 오브젝트가 분할 자료구조에만 저장되는가?

#### 분할 자료구조가 유일한 저장소

```cpp
// 모든 유닛이 그리드 셀에만 존재
Grid::cells_[x][y] → Unit → Unit → Unit → NULL
```

- 장점: 메모리 절약, 동기화 불필요
- 단점: 모든 오브젝트를 순회하려면 전체 그리드를 다 뒤져야 함

#### 보조 자료구조로 활용 (별도 컬렉션 병행)

```cpp
// 전체 유닛 목록 (빠른 전체 순회용)
Unit* allUnits[MAX_UNITS];

// 그리드 (빠른 위치 쿼리용)
Grid spatialGrid;
```

- 장점: 전체 순회는 `allUnits` 배열로 O(n) 직접 접근
- 단점: 생성/소멸 시 두 자료구조 모두 갱신 필요

## 주요 공간 분할 자료구조 한눈에 보기

| 자료구조 | 선형 대응 | 특징 | 적합한 경우 |
|----------|----------|------|------------|
| **Grid** | 버킷 정렬 (Bucket Sort) | 단순, 고정 크기 | 오브젝트가 균등 분포, 이동 잦음 |
| **Quadtree** (2D) / **Octree** (3D) | 트라이(Trie) | 계층, 적응형 세분화 | 밀집도 불균일, 동적 오브젝트 |
| **BSP** (Binary Space Partition) | 이진 탐색 트리 | 임의 평면 분할 | 정적 지형, 렌더링 가시성 판단 |
| **k-d tree** | 이진 탐색 트리 | 축 정렬 분할 | 정적 데이터, 최근접 이웃 탐색 |
| **BVH** (Bounding Volume Hierarchy) | 이진 탐색 트리 | 오브젝트 기반 | 레이캐스팅, 충돌 감지 |

```
각 자료구조의 선형 대응
─────────────────────────────────────────────────
1D (선형)            다차원 (공간)
─────────────────────────────────────────────────
버킷 정렬       →    Grid
이진 탐색 트리  →    BSP / k-d tree / BVH
트라이(Trie)    →    Quadtree / Octree
─────────────────────────────────────────────────
```

## 전체 구조 요약

```
공간 분할 패턴 핵심 흐름

모든 오브젝트를 하나의 배열에 저장
→ 위치 쿼리: O(n) 또는 O(n²) — 느림

           ↓ 공간 분할 적용

오브젝트를 위치 기반 자료구조에 저장
→ 위치 쿼리: O(log n) 또는 O(k) — 빠름

트레이드오프:
  + 위치 기반 쿼리 성능 대폭 향상
  - 오브젝트 이동 시 자료구조 갱신 비용
  - 추가 메모리 사용
  - 구현 복잡성 증가

선택 기준:
  오브젝트 수 많음?        → 분할 가치 있음
  오브젝트 분포 균등?      → 평탄한 그리드 적합
  오브젝트 밀집도 불균일?  → 계층적 분할 (Quadtree 등) 적합
  오브젝트 정적?           → BSP / k-d tree 적합
  오브젝트 동적?           → Grid / Quadtree 적합
```

---

## 다른 패턴과의 관계

- **데이터 지역성 패턴**: 같은 셀의 유닛을 연속된 메모리에 배치하면 캐시 효율을 함께 높일 수 있다.
- **오브젝트 풀 패턴**: 그리드 셀의 유닛들을 오브젝트 풀로 관리하면 할당 비용도 제거할 수 있다.
- **컴포넌트 패턴**: 물리 컴포넌트만 공간 분할 자료구조에 등록하고 나머지는 분리하는 방식으로 활용할 수 있다.

---

## 요약

공간 분할 패턴의 본질은 1D에서 이미 잘 알려진 탐색 최적화를 다차원으로 확장하는 것이다. 배열을 정렬해두면 이진 탐색이 가능하듯, 오브젝트를 위치 기반 자료구조에 담아두면 근처 오브젝트를 O(n)보다 훨씬 빠르게 찾을 수 있다.

핵심 선택은 **자료구조의 종류(평탄 vs 계층)** 와 **경계의 고정 여부(오브젝트 독립 vs 의존)** 이며, 게임의 오브젝트 분포 특성과 동적 여부에 따라 최적 선택이 달라진다.
