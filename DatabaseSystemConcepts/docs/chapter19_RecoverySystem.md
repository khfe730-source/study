# Chapter 19: Recovery System

## 개요

복구 시스템은 장애 발생 시 DB를 일관된 상태로 복원한다. 장애 분류, 안정 저장 구현, WAL 기반 로그 복구(redo/undo), 체크포인트, 버퍼 관리(force/steal 정책), 원격 백업, 논리적 undo 연산, ARIES 알고리즘을 다룬다.

---

## 19.1 장애 분류

| 장애 유형 | 설명 | 복구 방법 |
|-----------|------|-----------|
| **트랜잭션 실패** | 논리 오류(divide by zero), 시스템 오류(교착 상태) | 롤백 |
| **시스템 충돌** | 하드웨어 오류, 소프트웨어 버그 → 휘발성 메모리 손실 | 로그 기반 복구 |
| **디스크 장애** | 헤드 충돌, 전송 오류 → 비휘발성 저장 손상 | 백업 + 로그 |

**페일-스톱 가정(Fail-Stop)**: 하드웨어/소프트웨어 오류는 시스템을 멈추지만, 비휘발성 저장 내용은 보존된다.

## 19.2 저장 구조

### 안정 저장 구현

```
블록 쓰기 프로토콜 (2중 복제):
1. 첫 번째 물리 블록에 쓰기
2. 첫 번째 쓰기 완료 후 두 번째 블록에 쓰기
3. 두 번째 쓰기 완료 시 출력 완성

복구 시: 두 블록 비교
  - 동일하고 오류 없음 → 정상
  - 하나에 오류 → 다른 블록으로 복원
  - 둘 다 정상이지만 다름 → 첫 번째 블록으로 두 번째 덮어씀
```

### 데이터 접근

```
디스크 (물리 블록) ←→ 메인 메모리 (버퍼 블록)
  input(B):  물리 블록 B → 메모리
  output(B): 버퍼 블록 B → 디스크

트랜잭션 Ti:
  read(X):  버퍼에 BX 없으면 input(BX) 실행 후 xi = X 값 복사
  write(X): 버퍼에 BX 없으면 input(BX) 실행 후 xi → BX의 X 갱신
  (output(BX)는 write(X)와 별개로 나중에 수행)
```

## 19.3 복구와 원자성

### 19.3.1 로그 레코드

**로그(Log)**: 안정 저장에 유지되는 순차적 DB 변경 기록.

**업데이트 로그 레코드** `<Ti, Xj, V1, V2>`:
- Ti: 트랜잭션 ID
- Xj: 변경 데이터 항목
- V1: 변경 전 값 (undo 용)
- V2: 변경 후 값 (redo 용)

**특수 로그 레코드**:
```
<Ti start>     - 트랜잭션 시작
<Ti commit>    - 트랜잭션 커밋 (이 레코드가 안정 저장에 기록되면 커밋 완료)
<Ti abort>     - 트랜잭션 중단
```

### 19.3.2 데이터베이스 변경 방식

| 방식 | 설명 | 특징 |
|------|------|------|
| **지연 변경(Deferred)** | 커밋 전까지 실제 DB 변경 안 함 | 로컬 복사본 필요 |
| **즉시 변경(Immediate)** | 트랜잭션 실행 중에도 DB 변경 | 로그에 구 값 필요 |

### 19.3.3 동시성 제어와 복구 연계

복구 알고리즘 요건: 커밋되지 않은 트랜잭션이 변경한 데이터를 다른 트랜잭션이 변경할 수 없음 → **Strict 2PL** 또는 스냅샷/검증 방식으로 충족.

### 19.3.4 트랜잭션 커밋

`<Ti commit>` 레코드가 안정 저장에 출력될 때 커밋 완료. 단, 데이터 블록은 그 시점에 디스크에 있지 않아도 됨.

### 19.3.5 로그로 Redo/Undo

```
redo(Ti): 로그에서 Ti의 모든 업데이트를 새 값으로 재적용 (순서 유지)
undo(Ti): 로그에서 Ti의 모든 업데이트를 구 값으로 복원 (역순)
           복원 시 redo-only 로그 레코드 <Ti, Xj, V1> 기록
           완료 시 <Ti abort> 기록

시스템 충돌 후 처리 기준:
  <Ti start>만 있고 commit/abort 없음 → undo(Ti)
  <Ti commit> 또는 <Ti abort> 있음    → redo(Ti)
```

**예시**:
```
Log:
  <T0 start>
  <T0, A, 1000, 950>
  <T0, B, 2000, 2050>
  <T0 commit>
  <T1 start>
  <T1, C, 700, 600>
  ← 충돌

복구:
  redo(T0): A=950, B=2050
  undo(T1): C=700  (commit 없으므로)
```

### 19.3.6 체크포인트 (Checkpoint)

```
체크포인트 절차:
1. 메모리의 모든 로그 레코드 → 안정 저장에 출력
2. 변경된 모든 버퍼 블록 → 디스크에 출력
3. <checkpoint L> 로그 레코드 기록 (L = 현재 활성 트랜잭션 목록)

복구 시: 마지막 <checkpoint L> 이후의 트랜잭션만 고려
  → 체크포인트 이전에 커밋된 트랜잭션은 redo 불필요
```

```
예시:
체크포인트 당시 활성: {T67, T69}
T68 이전 트랜잭션 → 이미 처리됨

복구 대상: T67, T69, T70, ..., T100
  커밋된 것 → redo
  미완료된 것 → undo
```

**퍼지 체크포인트(Fuzzy Checkpoint)**: 체크포인트 레코드 기록 후 버퍼 블록을 나중에 출력. 처리 중단 없음. 마지막 완료 체크포인트 위치를 별도 저장.

## 19.4 복구 알고리즘

### 19.4.1 트랜잭션 롤백

```
Ti 롤백 (정상 실행 중):
1. 로그를 역방향 스캔
2. <Ti, Xj, V1, V2> 발견 시:
   a. Xj에 V1 기록 (undo)
   b. redo-only 로그 레코드 <Ti, Xj, V1> 기록 (보상 로그)
3. <Ti start> 발견 시 중지
4. <Ti abort> 로그 레코드 기록
```

### 19.4.2 시스템 충돌 후 복구 (2-Phase)

**Phase 1: Redo 단계** (순방향 스캔, 마지막 체크포인트부터):
```
undo-list = 체크포인트의 활성 트랜잭션 목록 L
for each log record (순방향):
  - <Ti, Xj, V1, V2> 또는 <Ti, Xj, V2> → redo (Xj = V2)
  - <Ti start>         → Ti를 undo-list에 추가
  - <Ti commit>/<Ti abort> → Ti를 undo-list에서 제거
끝: undo-list = 충돌 당시 미완료 트랜잭션 목록
```

**Phase 2: Undo 단계** (역방향 스캔):
```
for each log record (역방향):
  - undo-list의 Ti에 대한 <Ti, Xj, V1, V2> → undo (Xj = V1, 보상 로그 기록)
  - <Ti start> 발견 → <Ti abort> 기록, Ti를 undo-list에서 제거
종료: undo-list가 비면 완료
```

**히스토리 반복(Repeating History)**: redo 단계에서 실패한 트랜잭션의 부분 실행까지 포함하여 재현 → 충돌 시점 상태 재구성 → 안정적 undo 가능.

### 19.4.3 그룹 커밋 (Group Commit)

여러 트랜잭션의 커밋 로그를 한 블록에 모아 한 번에 디스크 출력 → 로그 쓰기 I/O 감소.

```
HDD: 블록 쓰기 ~5~10ms → 최대 200 TPS 단독, 10개 묶으면 2,000 TPS
SSD: 블록 쓰기 ~100μs → 최대 10,000 TPS, 그룹 커밋 시 100,000 TPS
```

## 19.5 버퍼 관리

### 19.5.1 로그 레코드 버퍼링

로그 레코드를 즉시 안정 저장에 쓰지 않고 로그 버퍼에 모아서 일괄 출력.

**WAL (Write-Ahead Logging) 규칙**:
```
1. <Ti commit>을 출력하기 전에 Ti의 모든 로그 레코드가 안정 저장에 있어야 함
2. 데이터 블록을 디스크에 출력하기 전에 해당 블록의 모든 로그 레코드가 안정 저장에 있어야 함

WAL = "로그 먼저, 데이터 나중"
```

### 19.5.2 데이터베이스 버퍼링

**Force/No-Force 정책**:
| 정책 | 설명 | 선택 |
|------|------|------|
| Force | 커밋 시 변경 블록 즉시 디스크 출력 | 느림 |
| **No-Force** | 커밋 후에도 나중에 출력 가능 | 표준 |

**Steal/No-Steal 정책**:
| 정책 | 설명 | 선택 |
|------|------|------|
| **Steal** | 미커밋 트랜잭션의 블록도 디스크에 출력 가능 | 표준 |
| No-Steal | 커밋 후에만 출력 | 대량 업데이트 불가 |

**표준 조합**: No-Force + Steal → WAL 규칙만 지키면 정확성 보장.

```
WAL 기반 출력 프로토콜:
1. 블록 B1에 배타 래치 획득 (쓰기 중임 방지)
2. B1 관련 로그 레코드를 안정 저장에 모두 출력
3. B1을 디스크에 출력
4. 래치 해제
```

## 19.6 비휘발성 저장 장애 복구

```
주기적 아카이브 덤프 (하루 1회 등):
1. 로그를 안정 저장에 출력
2. 버퍼 블록을 디스크에 출력
3. DB 전체를 안정 저장에 복사
4. <dump> 로그 레코드 기록

복구 시:
  최근 덤프로 DB 복원
  → 덤프 이후의 로그를 redo
  (undo는 필요 없음: 덤프 시점의 DB는 일관됨)
```

**SQL 덤프**: DDL/DML 문으로 DB 재생성. DB 마이그레이션이나 버전 업그레이드에 활용.

## 19.7 원격 백업 시스템 (High Availability)

```
Primary Site ──[로그 레코드 전송]──► Remote Backup Site
  트랜잭션 처리             로그 적용으로 동기화
  장애 발생 시 ──────────────► Backup이 Primary 역할 인수
```

**실패 탐지**: 다중 독립 통신 링크로 네트워크 분리와 실제 장애 구별.

**전환(Failover)**: 구 Primary의 IP를 새 Primary에 할당 or 고가용성 프록시 사용.

**핫 스페어(Hot-Spare)**: Backup이 수신 로그를 즉시 적용 → 인수 즉시 가능.

**커밋 내구성 수준**:

| 수준 | 조건 | 특징 |
|------|------|------|
| One-safe | Primary에만 커밋 기록 | 빠름, 데이터 손실 가능 |
| Two-very-safe | Primary + Backup 모두 기록 | 가용성 낮음 |
| **Two-safe** | 둘 다 활성 시 두 곳 모두, Backup 다운 시 Primary만 | 균형 (권장) |

## 19.8 논리적 Undo 연산

### 배경

B⁺-Tree 같은 구조에서 조기 잠금 해제 후 물리적 undo(구 값 복원) 불가 → **논리적 undo** 필요.

```
T1이 B⁺-Tree에 (V1, R1) 삽입 → 잠금 해제
T2가 같은 노드에 (V2, R2) 삽입 → 노드 이동
T1 롤백 시: (V1, R1)을 삭제하는 보상 연산 실행
```

### 논리적 undo 로그 레코드

```
<Ti, Oj, operation-begin>   - 논리 연산 시작
... 물리 로그 레코드 ...      - 연산 중 실제 변경
<Ti, Oj, operation-end, U>  - 연산 완료 (U = undo 정보, 예: 삭제 연산)
```

**Redo**: 물리 로그 레코드만 사용 (상태 불일치 가능성 때문에 논리 redo 불가).
**Undo**: operation-end 레코드 발견 시 U 정보로 논리 undo 실행.

### 롤백 알고리즘 (논리적 undo 포함)

```
역방향 스캔 중:
  물리 로그 레코드: 일반 undo 수행
  <Ti, Oj, operation-end, U> 발견:
    → U로 논리 undo 실행 (물리 로그 기록)
    → <Ti, Oj, operation-abort> 기록
    → <Ti, Oj, operation-begin>까지 스캔 건너뜀
  <Ti, Oj, operation-abort> 발견:
    → 해당 연산의 모든 로그 건너뜀 (이미 undo됨)
  <Ti start> 발견: → <Ti abort> 기록, 롤백 완료
```

## 19.9 ARIES 알고리즘

### 주요 특징

1. **LSN (Log Sequence Number)**: 각 로그 레코드에 고유 번호. PageLSN으로 페이지에 마지막 적용된 LSN 추적.
2. **생리적 redo(Physiological Redo)**: 물리적 페이지 식별 + 논리적 내부 연산 → 로그 크기 감소.
3. **더티 페이지 테이블(DirtyPageTable)**: 메모리에서 변경됐지만 디스크에 아직 반영 안 된 페이지 목록. RecLSN(해당 페이지를 더티하게 만든 첫 LSN) 포함.
4. **퍼지 체크포인트**: DirtyPageTable과 활성 트랜잭션 목록만 기록. 변경 페이지를 즉시 디스크에 쓰지 않아도 됨.

### ARIES 데이터 구조

```
각 로그 레코드:
  LSN, PrevLSN (같은 트랜잭션의 이전 로그), TxnID, 연산 유형, 데이터 변경

보상 로그 레코드(CLR):
  UndoNextLSN (다음으로 undo할 LSN) 포함

각 페이지:
  PageLSN (마지막으로 적용된 로그의 LSN)

체크포인트 로그 레코드:
  DirtyPageTable, 활성 트랜잭션 목록, 각 트랜잭션의 LastLSN
```

### ARIES 복구 3단계

```
1. Analysis Pass (분석 패스):
   - 마지막 체크포인트부터 로그 순방향 스캔
   - DirtyPageTable 재구성
   - undo-list (미완료 트랜잭션 목록) 결정
   - RedoLSN = min(RecLSN in DirtyPageTable) 결정

2. Redo Pass (재실행 패스):
   - RedoLSN부터 순방향 스캔
   - 각 업데이트 로그 레코드에 대해:
     a. 페이지가 DirtyPageTable에 없거나 LSN < RecLSN → 건너뜀
     b. 페이지 fetch 후 PageLSN < LSN이면 redo
   - PageLSN 활용으로 불필요한 redo 최소화

3. Undo Pass (취소 패스):
   - undo-list의 트랜잭션들을 역방향으로 롤백
   - CLR 생성 (UndoNextLSN 포함)
   - <Ti start> 발견 시 <Ti abort> 기록
```

```
예시 (ARIES):
LSN 7563: <T145 begin>
LSN 7564: <T145, 4894.1, 20, 40>
LSN 7565: <T143, 7200.2, 60, 80>
LSN 7566: <T143 commit>
LSN 7567: <T145, 4894.1, 40, 60>
LSN 7568: checkpoint {T145 active}, DirtyPages={4894: RecLSN=7564, 7200: RecLSN=7565}
LSN 7569: <T146 begin>
LSN 7570: <T146, 2390.4, 50, 90>
LSN 7571: <T146 commit>
[충돌]

Analysis: RedoLSN=7564, undo-list={T145}
Redo: LSN 7564~7571 재실행 (PageLSN 비교로 건너뜀 최소화)
Undo: T145의 LSN 7567, 7564 역순 undo → CLR 기록 → <T145 abort>
```

### ARIES 추가 기능

- **중첩 최상위 연산(Nested Top Actions)**: 롤백 불필요 연산(페이지 할당 등) → CLR로 스킵
- **저장점(Savepoints)**: 트랜잭션 중간 상태 저장, 부분 롤백 지원. 교착 상태 처리에 유용.
- **세밀한 잠금**: 인덱스 튜플 수준 잠금 지원
- **복구 최적화**: DirtyPageTable로 페이지 프리패치, 비순서 redo

---

## 19.10 인메모리 DB 복구

```
최적화 포인트:
1. 인덱스: 복구 후 재구축 가능 → 인덱스 redo 로그 불필요
2. Undo 로그: 메모리에만 유지 (디스크 불필요)
3. 병렬 복구: 데이터/로그를 파티셔닝 → 여러 코어로 병렬 처리

비휘발성 RAM(NVRAM/3D-XPoint) 활용:
  - 직접 바이트 단위 접근
  - redo 로그 불필요 (충돌 후에도 데이터 보존)
  - undo 로그만 필요 (트랜잭션 중단 처리용)
```

## 19.11 핵심 요약

```
복구 알고리즘 흐름

정상 동작 중:
  트랜잭션 변경 → 로그 먼저 기록(WAL) → 버퍼에 변경 → 나중에 디스크 출력

트랜잭션 롤백:
  로그 역방향 스캔 → 구 값으로 복원 → 보상 로그 기록 → abort 레코드

시스템 충돌 후:
  Redo Phase: 마지막 체크포인트부터 순방향 재실행 (히스토리 반복)
  Undo Phase: 미완료 트랜잭션 역방향 롤백

ARIES:
  Analysis → Redo (LSN/PageLSN 최적화) → Undo (CLR/UndoNextLSN)
```

| 개념 | 핵심 포인트 |
|------|------------|
| WAL | 로그 먼저, 데이터 나중. 커밋 = commit 로그가 안정 저장에 기록된 시점 |
| No-Force + Steal | 표준 조합. 성능 최적. WAL로 정확성 보장 |
| Redo | 커밋된 트랜잭션의 변경을 새 값으로 재적용 |
| Undo | 미완료 트랜잭션의 변경을 구 값으로 복원 + 보상 로그 기록 |
| 체크포인트 | 복구 범위 축소. 이전 체크포인트 완료 후 트랜잭션은 redo 불필요 |
| 퍼지 체크포인트 | 체크포인트 중 트랜잭션 처리 계속. 성능 향상 |
| 히스토리 반복 | redo 단계에서 실패 트랜잭션 포함 모두 재실행 → 일관성 복원 |
| 논리적 undo | B⁺-Tree 등 조기 잠금 해제 후 보상 연산으로 undo |
| LSN | ARIES의 로그 레코드 식별자. PageLSN으로 불필요 redo 방지 |
| DirtyPageTable | 변경됐지만 디스크에 없는 페이지 목록. RecLSN으로 redo 시작점 결정 |
| Two-safe | 원격 백업의 커밋 내구성 균형 수준 |
