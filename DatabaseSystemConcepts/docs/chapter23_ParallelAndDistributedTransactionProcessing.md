# Chapter 23: Parallel and Distributed Transaction Processing

## 개요

분산 트랜잭션 처리는 여러 노드에 걸친 원자성·일관성·고립성·영속성을 보장해야 한다. 2단계 커밋(2PC), 영속 메시징, 분산 동시성 제어(잠금·타임스탬프·스냅샷 격리), 복제된 데이터의 동시성 제어(과반수 프로토콜·쿼럼 합의), 약한 일관성 복제, 코디네이터 선택, Paxos·Raft 합의 프로토콜을 다룬다.

---

## 23.1 분산 트랜잭션

### 시스템 구조

```
TC1         TCn        (Transaction Coordinator)
│           │
TM1         TMn        (Transaction Manager)
│           │
노드 1      노드 n
```

- **로컬 트랜잭션**: 단일 노드에서만 데이터 접근
- **글로벌 트랜잭션**: 여러 노드에 걸쳐 데이터 접근

각 노드의 TM은 로컬 ACID 보장. 글로벌 트랜잭션은 TC가 여러 TM 조율.

### 장애 유형

- 노드 장애
- 메시지 손실
- 통신 링크 장애
- **네트워크 파티션**: 연결이 없는 두 부분으로 분리 (장애와 구분 불가)

---

## 23.2 커밋 프로토콜

### 23.2.1 2단계 커밋 (2PC)

```
코디네이터 Ci와 참여 노드들:

Phase 1 (준비 단계):
  Ci → <prepare T> 로그 기록 → 모든 노드에 prepare T 메시지 전송
  각 노드 →  커밋 가능하면: <ready T> 로그 + ready T 응답
             커밋 불가하면: <no T> 로그 + abort T 응답

Phase 2 (결정 단계):
  모든 ready T 수신 시:
    Ci → <commit T> 로그 → commit T 메시지 전송
  하나라도 abort T 수신 시:
    Ci → <abort T> 로그 → abort T 메시지 전송
  
각 노드: commit/abort 수신 → <commit T>/<abort T> 로그 → 실행
```

**성공 실행 흐름**:
```
코디네이터    노드 N1          노드 N2
─────────    ───────          ───────
Force <prepare T>
Send prepare T  ────────────► Force <ready T>
                               Send ready T
                ◄────────────
                              Force <ready T>
                               Send ready T
                ◄─────────────────────────────
Force <commit T>
Send commit T   ────────────► Force <commit T>
                ─────────────────────────────►
```

**장애 처리**:

| 상황 | 처리 방법 |
|------|---------|
| 참여자 장애 (ready 전) | 코디네이터가 abort로 간주 |
| 참여자 장애 (ready 후) | 복구 시 코디네이터에 상태 질의 |
| 코디네이터 장애 후 복구 | 참여자들이 다른 노드에 status 질의 |
| 모두 ready지만 코디네이터 없음 | **블로킹**: 코디네이터 복구 전까지 대기 |

**블로킹 문제(Blocking Problem)**: ready 상태의 트랜잭션이 코디네이터 복구 전까지 대기, 잠금 보유 → 다른 트랜잭션 차단.

**복구 시 재확인**: `<ready T, L>` 로그 레코드에 잠금 정보 저장 → 복구 후 잠금 재획득 → 새 트랜잭션 처리 재개 (코디네이터 복구와 무관하게).

### 23.2.2 블로킹 방지

**분산 합의 기반 2PC**: 코디네이터 결정을 합의 프로토콜로 대체.
- 과반수 노드가 살아있으면 블로킹 없이 커밋/중단 결정 가능
- 코디네이터 장애 시 새 코디네이터 선출 → 기존 결정 확인 후 계속

**3단계 커밋(3PC)**: 네트워크 파티션 없는 환경에서 블로킹 방지. 실제로는 합의 프로토콜로 대체.

### 23.2.3 영속 메시징 (Persistent Messaging)

2PC 사용 불가(조직 경계 등) 시 대안. 은행 수표 방식과 유사.

```
구현:
  송신측:
    트랜잭션 내에서 messages_to_send 릴레이션에 메시지 기록
    별도 프로세스가 메시지 읽어 전송 (commit 후)
    ACK 받을 때까지 주기적 재전송

  수신측:
    received_messages 릴레이션에 삽입 (중복 탐지용 고유 ID)
    ACK 전송 후 메시지 처리
    이미 존재하면 중복으로 무시

특성:
  - 정확히 1회 전달 보장 (중복 전달 탐지 + ACK 기반)
  - 트랜잭션이 커밋되면 전달 보장, 중단되면 미전달 보장
  - 비동기 → 조직 경계 초월 가능
```

**워크플로우**: 복잡한 다단계 비즈니스 프로세스를 영속 메시징으로 구현. 예: 대출 심사 → 신용 조회 → 승인 → 자금 집행.

## 23.3 분산 DB의 동시성 제어

### 23.3.1 잠금 프로토콜

**단일 잠금 관리자**:
- 하나의 노드에 중앙 잠금 관리자
- 장점: 단순 구현, 단순 교착 상태 처리
- 단점: 병목, 단일 장애점

**분산 잠금 관리자**:
- 각 노드가 자신의 데이터에 대한 잠금 관리
- 글로벌 교착 탐지 필요 (대기 그래프를 여러 노드에서 병합)

### 23.3.4 타임스탬프 기반 분산 동시성 제어

전역 타임스탬프 할당 필요:
- 각 노드의 로컬 시계 + 노드 ID 조합
- 메시지에 타임스탬프 포함하여 인과성 추적(논리적 시계)

## 23.4 복제 (Replication)

### 복제본 일관성

**선형화 가능성(Linearizability)**: 읽기는 항상 가장 최근에 커밋된 쓰기 값을 반환.

```
동시성 제어 + 복제의 어려움:
  읽기-하나-쓰기-전체 (Read-one, Write-all): 단순하지만
  "쓰기 전체 가능"(Write-all-available)는 불일치 위험
  → 파티션 중 각 파티션이 독립적으로 업데이트 → 충돌
```

### 23.4.2 복제본 동시성 제어

| 프로토콜 | 설명 | 특징 |
|---------|------|------|
| **Primary Copy** | 한 복제본이 기본 → 기본 복제본 잠금 관리 | 단순, 기본 복제본 장애 시 불가 |
| **과반수 프로토콜** | n개 복제본 중 과반수(>n/2) 잠금 획득 | 분산 제어, 교착 가능 |
| **편향 프로토콜** | 공유: 임의 1개, 배타: 전체 | 읽기 비용 낮음 |
| **쿼럼 합의** | 가중치 할당, Qr+Qw>S, 2Qw>S | 유연한 읽기/쓰기 비용 조절 |

**쿼럼 합의 예시**:
```
n = 5개 복제본, 각 가중치 = 1, S = 5
읽기 쿼럼 Qr = 2, 쓰기 쿼럼 Qw = 4
  Qr + Qw = 6 > 5 ✓
  2 × Qw = 8 > 5 ✓
  → 읽기: 2개 복제본 잠금, 쓰기: 4개 잠금
```

### 23.4.3 장애 대응

**과반수 프로토콜 + 버전 번호**:
```
읽기: 과반수 복제본에서 잠금 → 가장 높은 버전 번호의 값 반환
쓰기: 과반수 복제본에서 잠금 → 버전 번호 = 최고값 + 1 → 모두에 기록

장점: 재통합 불필요 (과반수 원칙 자체로 최신 값 탐지)
```

**읽기 비용 감소**:
- 쿼럼 합의로 읽기 쿼럼 크기 조정
- 기본 복제본 기법 + 분산 합의로 기본 복제본 장애 처리

## 23.5 확장 동시성 제어

### 분산 MV2PL + 전역 일관성 타임스탬프

**Google Spanner**: TrueTime API 기반 불확실성 구간 있는 실제 시계 활용.

```
Spanner의 커밋 타임스탬프:
  TrueTime.now() = [earliest, latest] (불확실성 구간)
  커밋 타임스탬프 >= latest → 이후 트랜잭션은 더 큰 타임스탬프
  → 전역 일관성 보장

GPS + 원자 시계로 불확실성 구간을 수 밀리초 이내로 제한
```

### 분산 스냅샷 격리

각 노드에서 독립적으로 스냅샷 격리 → 글로벌 일관된 스냅샷이 필요.

## 23.6 약한 일관성 복제 (Lazy Replication)

**비동기 복제(Asynchronous/Lazy Replication)**:
- 기본 복제본에 업데이트 → 커밋 → 나중에 다른 복제본에 전파
- 장점: 지리적 복제 시에도 빠른 커밋 (네트워크 왕복 대기 없음)
- 단점: 복제본이 일시적으로 불일치, 기본 복제본 장애 시 업데이트 손실 가능

**최종 일관성(Eventual Consistency)**:
- 모든 복제본이 결국 동일 값을 가짐을 보장
- 단기적 불일치는 허용
- 많은 웹 애플리케이션에서 적합 (SNS, 쇼핑 카트 등)

---

## 23.7 코디네이터 선택 (Coordinator Selection)

장애 발생 시 새 코디네이터를 자동 선출하는 메커니즘.

### 랜덤-재시도 알고리즘 (Raft 선거와 유사)

```
각 노드에 랜덤 타임아웃 부여 (예: 150~300ms)
리더를 듣지 못하면 타임아웃 후 선거 시작:
  1. 자신의 term 번호 증가
  2. 모든 노드에 투표 요청
  3. 과반수 투표 획득 → 리더 선출
  4. 주기적 heartbeat로 리더 유지

랜덤 타임아웃으로 동시 선거 충돌 최소화
```

### 임대(Lease)

리더에게 제한된 시간 동안 읽기 서빙 권한 부여. 임대 만료 전에 갱신.
임대 미갱신 → 권한 자동 만료 → 새 리더가 안전하게 선출 가능.

## 23.8 분산 합의 (Distributed Consensus)

### 문제 정의

```
n개 노드가 하나의 값에 합의해야 함:
  - 일부 노드 장애 무관하게 동일 결정
  - 과반수 노드가 살아있으면 블로킹 없음
  - 결정 후 취소 불가

사용 사례:
  - 2PC 코디네이터 역할 대체 (블로킹 방지)
  - 복제 로그 동기화
  - 리더 선출
```

### 23.8.2 Paxos 합의 프로토콜

**참여자**:
- 제안자(Proposer): 값을 제안
- 수락자(Acceptor): 투표
- 학습자(Learner): 결정된 값 학습

**프로토콜 (단일 결정)**:

```
Phase 1a: Proposer → Acceptor: prepare(n)  (n = 제안 번호)
Phase 1b: Acceptor → Proposer: promise(n) 또는 거부
           (이전에 수락한 값 있으면 함께 전달)

Phase 2a: Proposer → Acceptor: accept(n, v)
           v = 가장 높은 번호 m의 수락값 (없으면 자체 제안값)
Phase 2b: Acceptor: n 이상의 prepare 응답 없으면 v 수락

과반수 수락 → v가 합의값

핵심: 이미 과반수가 수락한 값 v가 있으면,
      이후 어떤 제안도 v를 제안 → 일관성 보장
```

**Multi-Paxos**: 일련의 결정(로그 추가)을 위한 연속 Paxos. 코디네이터 고정으로 효율화.

### 23.8.3 Raft 합의 프로토콜

Paxos보다 이해하기 쉬운 설계 목표. **로그 기반**.

**핵심 구성**:
- **리더(Leader)**: 로그 추가 담당. 텀(Term) 단위로 선출.
- **팔로워(Follower)**: 리더의 AppendEntries 수락.

```
로그 상태 예시:
            Log Index:  1    2    3    4    5    6    7
Leader:               [1,x=2][1,z=2][1,x=3][2,x=4][3,x=1][3,y=6][3,z=4]
Follower 1:           [1,x=2][1,z=2][1,x=3][2,x=4][3,x=1]
Follower 2:           [1,x=2][1,z=2][1,x=3][2,x=4][3,x=1][3,y=6][3,z=4]

각 셀 = (term, 명령)
커밋된 항목: 과반수 노드에 복제된 항목 (index ≤ 6)
```

**AppendEntries RPC**:
```
리더 → 팔로워: {term, prevLogEntryPos, prevLogEntryTerm, logEntries, leaderCommitIndex}

팔로워 처리:
  1. term < currentTerm → false 반환 (구 리더 메시지)
  2. prevLog 불일치 → false (로그 불일치)
  3. 불일치 항목 삭제 + 새 항목 추가
  4. commitIndex 업데이트
  5. true 반환
```

**리더 선출**:
- 랜덤 타임아웃으로 팔로워가 선거 시작
- 로그가 최신인 후보만 투표 받음 → 새 리더는 모든 커밋된 항목 보유 보장

**로그 복구**: 새 리더가 팔로워의 구 로그를 리더 로그로 덮어씀.

### 23.8.4 합의 기반 내결함성 복제

```
상태 머신(State Machine) 복제:
  여러 노드가 동일 상태 머신의 복사본 유지
  같은 순서로 같은 명령 실행 → 동일 상태 보장

구현:
  1. 클라이언트가 명령을 Raft/Paxos 로그에 추가 요청
  2. 과반수 노드에 복제 후 커밋
  3. 모든 노드가 커밋된 명령을 순서대로 상태 머신에 적용
  4. 노드 장애 시 새 노드가 로그 재생으로 상태 복원

활용: 분산 KV 스토어(etcd, ZooKeeper), 분산 DB 메타데이터, 복제된 서비스
```

## 23.9 핵심 요약

```
분산 트랜잭션 처리 계층:

원자성 보장:
  단일 데이터센터 → 2PC
  장애 내성 필요  → 합의(Paxos/Raft) 기반 2PC

일관성:
  강한 일관성 → 2PC + 과반수 프로토콜
  최종 일관성 → 비동기 복제 + 영속 메시징

가용성:
  노드 장애 → 복제 + 과반수 프로토콜
  코디네이터 장애 → 합의 기반 비블로킹 2PC
  데이터센터 장애 → 지리적 복제 + 비동기 복제

CAP 트레이드오프:
  C(일관성) + A(가용성) vs P(파티션 허용)
  네트워크 파티션 중 C와 A를 동시에 보장 불가
```

| 개념 | 핵심 포인트 |
|------|------------|
| 2PC | 준비+결정 2단계, 모두 ready면 커밋, 블로킹 위험 |
| 블로킹 문제 | 코디네이터 장애 시 참여자가 결정 대기 |
| 영속 메시징 | 정확히 1회 전달, 비동기, 조직 경계 초월 |
| 과반수 프로토콜 | n/2+1 복제본 잠금, 장애 내성 |
