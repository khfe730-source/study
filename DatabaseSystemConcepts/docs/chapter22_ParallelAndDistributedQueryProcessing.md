# Chapter 22: Parallel and Distributed Query Processing

## 개요

병렬 쿼리 처리는 단일 쿼리를 여러 노드에서 동시에 실행하여 응답 시간을 단축한다. 인트라연산 병렬성(병렬 정렬·조인·집계), 인터연산 병렬성(독립 연산·파이프라인), 교환 연산자 모델(Exchange Operator), 공유 메모리 최적화, 분산 쿼리 처리를 다룬다.

---

## 22.1 개요

```
병렬 처리 유형:

인터쿼리 병렬성(Inter-query): 여러 쿼리를 노드들에 분산 실행
  → 트랜잭션 처리 시스템의 처리량 향상 (Ch.23)

인트라쿼리 병렬성(Intra-query): 단일 쿼리를 여러 노드에서 동시 처리
  └─ 인트라연산(Intraoperation): 한 연산을 여러 노드에서 실행
  └─ 인터연산(Interoperation): 여러 연산을 동시에 실행
```

인트라연산 병렬성이 가장 중요. 릴레이션 내 튜플 수만큼 확장 가능.

## 22.2 병렬 정렬

### 범위 파티셔닝 정렬

```
Step 1: 릴레이션을 정렬 속성 기준 범위 파티셔닝
        → 모든 노드에서 병렬로 tuples 재분배

Step 2: 각 노드에서 자신의 파티션을 독립적으로 정렬

결과: 노드 N1 < 노드 N2 < ... < Nm 의 순서로 이미 정렬됨
      → 단순 연결(concatenation)으로 전체 정렬 완성
```

### 병렬 외부 정렬-병합

```
Step 1: 각 노드에서 로컬 정렬 (정렬된 런 생성)

Step 2: 범위 파티셔닝으로 재분배 + 병합(Ordered Merge)

장점: 초기 로컬 정렬 활용으로 총 작업량 감소
```

## 22.3 병렬 조인

### 파티셔닝 조인 (Partitioned Join)

동일 조인 속성으로 두 릴레이션을 같은 방식으로 파티셔닝:

```
r ⋈ s (조인 속성 A)

1. r을 A 기준 해시/범위 파티셔닝 → r1, r2, ..., rn
2. s를 동일 함수로 파티셔닝    → s1, s2, ..., sn
3. 각 노드 Ni에서: ri ⋈ si (로컬 조인)

장점: 각 노드가 독립적 → 완전 병렬
조건: r과 s가 동일 파티셔닝 함수 사용 (이미 파티셔닝된 경우 재분배 불필요)
```

### 비대칭 단편-복제 조인 (Asymmetric Fragment and Replicate)

한쪽 릴레이션이 너무 작을 때:

```
작은 릴레이션 s → 모든 노드에 브로드캐스팅(복제)
큰 릴레이션 r   → 파티셔닝 (또는 이미 파티셔닝됨)

각 노드: ri ⋈ s_복사본

장점: 큰 릴레이션의 파티셔닝을 유지한 채 작은 릴레이션만 전송
```

### 대칭 단편-복제 조인

두 릴레이션 모두 파티셔닝 불일치 시:

```
m × n 그리드: m = 그룹 수, n = 노드 수/그룹

r의 각 파티션 → 같은 행의 모든 노드에 복제
s의 각 파티션 → 같은 열의 모든 노드에 복제
각 노드 (i, j): ri ⋈ sj

비용: r과 s 모두 √n 배 복제
```

## 22.4 기타 연산

### 집계 (Aggregation)

```
최적화: 부분 집계(Partial Aggregation)

Step 1: 각 노드에서 로컬 부분 집계
        (예: 각 노드의 dept_name별 sum(salary))

Step 2: GROUP BY 속성으로 해시 파티셔닝 + 재집계

장점: 네트워크 전송량 대폭 감소
      sum/count/min/max: 결합 법칙 성립 → 부분 집계 후 최종 집계
      avg: sum과 count를 별도로 부분 집계 후 최종 avg 계산
```

### MapReduce

```
Map 함수:   각 입력 튜플 → (key, value) 쌍들 출력
Shuffle:    같은 key를 가진 쌍들을 같은 Reducer로 전송
Reduce 함수: (key, [value, value, ...]) → 결과

관계 대수 표현:
  Map = 선택 + 투영 + 부분 집계
  Shuffle = 교환 연산자 (해시 파티셔닝)
  Reduce = 집계 완성
```

## 22.5 병렬 쿼리 계획 평가

### 22.5.1 인터연산 병렬성

**파이프라인 병렬성**: 한 연산의 출력이 다른 연산의 입력으로 즉시 전달.

```
σ → ⋈ → Π
      ↑       ↑
 노드 N1   노드 N2
  (동시 실행 가능)
```

**독립 병렬성**: 서로 의존하지 않는 연산들을 동시 실행.
```
t1 = r1 ⋈ r2   (N1~N3에서)
t2 = r3 ⋈ r4   (N4~N6에서)   ← 동시에!
t  = t1 ⋈ t2   (N7에서)      ← 이전 결과 필요 후
```

### 22.5.2 교환 연산자 모델 (Exchange Operator Model)

**핵심 아이디어**: 데이터 재분배는 교환 연산자(Exchange)가 전담, 다른 연산자는 로컬 실행만.

```
교환 연산자 파티셔닝 방식:
  1. 해시 파티셔닝
  2. 범위 파티셔닝
  3. 브로드캐스팅 (모든 노드에 복제)
  4. 단일 노드 집중

교환 연산자 = 소스 노드의 파티셔닝 + 목적 노드의 병합

┌──────────────────────────────────────────────────────┐
│  r1 ──[Partition]──────────────[Merge]── r1'         │
│  r2 ──[Partition]──────────────[Merge]── r2'         │
│  ...                                                  │
│       각 소스→목적 노드로 튜플 전송                   │
└──────────────────────────────────────────────────────┘
```

**교환 연산자로 표현한 병렬 알고리즘들**:

| 알고리즘 | 교환 연산자 역할 |
|---------|--------------|
| 범위 파티셔닝 정렬 | 범위 파티셔닝 → 로컬 정렬 |
| 파티셔닝 조인 | 해시 파티셔닝 후 로컬 조인 |
| 집계 | 해시 파티셔닝 후 로컬 집계 |
| 비대칭 조인 | 소형 릴레이션 브로드캐스팅 |

**장점**: 기존 단일 노드 DB 엔진 재사용 가능 (교환 연산자만 추가).

**Pull(Demand-Driven)** vs **Push(Producer-Driven)**:
- Pull: 수요 기반 이터레이터 (단일 노드 표준)
- Push: 생산자 기반 → 병렬 시스템에서 더 효율적 (함수 호출 오버헤드 감소)

### 22.5.3 전체 병렬 쿼리 계획 예시

```
쿼리: r.C,s.D γsum(s.E) (r ⋈r.A=s.B s)

병렬 실행 계획:
  r ──[E1: r.A 해시파티셔닝]──┐
                              ├──[HJ: 해시 조인]──[E3: C,D 파티셔닝]──[HA: 집계]──[E4: 단일노드]── 결과
  s ──[E2: s.B 해시파티셔닝]──┘

파이프라인 단계:
  단계 1: E1 파티셔닝 (r), E2 파티셔닝 (s)  ← 독립 실행
  단계 2: HJ 빌드+프로브 (E1, E2 완료 후)
  단계 3: E3 파티셔닝 + HA 집계 (파이프라인)
  단계 4: E4로 최종 결과 집중
```

## 22.6 공유 메모리 아키텍처 쿼리 처리

```
공유 메모리에서의 최적화:
  - 공유 해시 테이블: 병렬 해시 조인 시 모든 코어가 단일 테이블 공유
  - 공유 버퍼: 결과를 각 코어에 복사하지 않고 공유 메모리에서 직접 접근
  - 래치-프리 데이터 구조: CAS로 잠금 없이 병렬 처리
  - 코어 친화성: 데이터를 로컬 NUMA 메모리에 배치

파이프라인:
  - 여러 코어가 파이프라인 단계를 분담
  - 작업 큐(Work Queue)로 튜플 분배
  - 캐시 인식 파티셔닝: 파티션이 L3 캐시에 맞도록 크기 조정
```

## 22.7 병렬 실행 최적화

### 22.7.1 병렬 조인 순서 선택

분산 설정에서 조인 순서 선택:
- 파티셔닝된 조인: 두 릴레이션이 동일 속성으로 파티셔닝된 경우 재분배 불필요
- 브로드캐스트: 한쪽 릴레이션이 매우 작으면 브로드캐스팅이 효율적

**흥미로운 파티셔닝(Interesting Partitioning)**: 중간 결과가 이후 연산에 유용한 파티셔닝.

### 비용 모델

```
교환 비용 = 전송 튜플 수 × 네트워크 전송 비용/튜플
           + 시작 비용 × 노드 수

총 비용 = Σ 로컬 연산 비용 + Σ 교환 연산 비용
```

소스 ──[브로커/Kafka]──► 연산자1 ──► 연산자2 ──► 싱크

## 22.9 분산 쿼리 처리

```
분산 쿼리 처리 최적화:
1. 사이트 선택: 쿼리를 데이터 근처에서 실행
2. 반세미조인(Semi-join): 불필요한 전송 최소화

반세미조인:
  r ⋉(r.A=s.A) s
  = ΠR(r ⋈ s)     (관계 대수 표현)
  
  처리:
  Step 1: s에서 조인 속성 A 투영 → s.A (크기 감소)
  Step 2: s.A를 r 사이트로 전송
  Step 3: r 사이트에서 r과 s.A를 조인 → r에서 매칭되는 튜플만 선택
  Step 4: 선택된 r 튜플을 s 사이트로 전송
  Step 5: 실제 조인 실행

비용: s의 A 컬럼만 전송 (전체 s보다 훨씬 작음)
```

## 22.10 핵심 요약

```
병렬 쿼리 처리 결정 트리:

릴레이션이 이미 파티셔닝됨?
  예 → 동일 속성으로 파티셔닝? → 파티셔닝 조인
       다른 속성으로 파티셔닝? → 재파티셔닝 후 조인
  아니요 → 한쪽이 매우 작음? → 브로드캐스팅 조인
          둘 다 큼?           → 해시/범위 파티셔닝 후 조인

집계 최적화:
  결합 가능(sum/count/min/max)? → 부분 집계 후 재집계
  결합 불가(avg)?               → sum+count 부분 집계 후 나눗셈
```

| 개념 | 핵심 포인트 |
|------|------------|
| 파티셔닝 조인 | 동일 파티셔닝 함수 → 로컬 조인, 완전 병렬 |
| 브로드캐스팅 | 작은 릴레이션 전체 노드에 복제 후 로컬 조인 |
| 교환 연산자 | 재분배 전담, 기존 엔진 재사용 가능 |
| 부분 집계 | 네트워크 전송량 대폭 감소 |
| 파이프라인 병렬성 | 연산간 결과 즉시 전달, 중간 결과 저장 없음 |
| Push vs Pull | Push가 병렬 시스템에서 효율적 |
| 반세미조인 | 분산 조인 시 불필요 데이터 전송 최소화 |
| 교환 비용 | 네트워크 전송 + 시작 비용이 전체 계획 비용에 포함 |
