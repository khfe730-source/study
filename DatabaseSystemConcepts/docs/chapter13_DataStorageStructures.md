# Chapter 13: Data Storage Structures

## 개요

레코드를 블록에 어떻게 저장하고, 파일을 어떻게 조직하며, 데이터베이스 버퍼를 어떻게 관리하는지를 다룬다. 고정/가변 길이 레코드, 슬롯페이지 구조, 파일 조직(힙/순차/멀티테이블 클러스터링), 데이터 사전, 버퍼 관리 전략, 컬럼 지향 저장, 인메모리 데이터베이스 최적화를 설명한다.

---

## 13.1 데이터베이스 저장 아키텍처

```
비휘발성 저장 (HDD/SSD)
  ├── 블록(Block) 단위 I/O (4~8KB)
  └── 레코드(Record)들을 블록에 매핑

주 메모리 (Buffer)
  └── 접근 대상 블록만 로드하여 처리

레코드 < 블록 크기 (일반적) → 하나의 블록에 여러 레코드 저장
레코드 > 블록 크기 (LOB) → 별도 저장 후 포인터 참조
```

---

## 13.2 파일 조직 (File Organization)

### 13.2.1 고정 길이 레코드 (Fixed-Length Records)

```
레코드 크기 = 53 bytes (ID:5, name:20, dept_name:20, salary:8)

블록 크기 = 4096 bytes
블록 당 레코드 수 = ⌊4096 / 53⌋ = 77개 (나머지 공간 미사용)
```

**삭제 처리**: 삭제된 레코드를 이동하는 대신 **자유 리스트(Free List)** 유지.

```
헤더 → [삭제된 레코드 1] → [삭제된 레코드 2] → null

삽입 시: 헤더가 가리키는 첫 번째 빈 슬롯 사용 (또는 파일 끝에 추가)
```

```
파일 구조:
┌────────────────────────────────────────────┐
│ Header: 첫 번째 빈 레코드 포인터            │
├─────────────────────────────────────────────┤
│ Record 0: 10101 | Srinivasan | Comp.Sci | 65000 │
│ Record 1: [→ Record 4] (삭제됨)            │
│ Record 2: 15151 | Mozart     | Music   | 40000 │
│ Record 3: 22222 | Einstein   | Physics | 95000 │
│ Record 4: [→ Record 6] (삭제됨)            │
│ ...                                         │
└─────────────────────────────────────────────┘
```

### 13.2.2 가변 길이 레코드 (Variable-Length Records)

**레코드 내부 구조**: 고정 크기 헤더 + 가변 데이터

```
Null Bitmap (1byte) | (offset, length) × N | 고정 길이 속성 | 가변 길이 데이터

예: instructor (ID varchar(5), name varchar(20), dept_name varchar(20), salary numeric(8))
┌─────┬──────────┬──────────┬──────────┬──────────┬─────────────────────────────┐
│0000 │ (21, 5)  │ (26, 10) │ (36, 10) │  65000   │ 10101 Srinivasan Comp. Sci. │
│NullB│ ID 오프셋│name 오프셋│dept 오프셋│ salary  │← 가변 데이터 영역           │
└─────┴──────────┴──────────┴──────────┴──────────┴─────────────────────────────┘
0     1          5          9          13         21         26        36         46
```

- **Null Bitmap**: i번째 비트가 1이면 i번째 속성이 NULL → 오프셋/길이/데이터 저장 불필요

### 13.2.3 슬롯페이지 구조 (Slotted-Page Structure)

가변 길이 레코드를 블록 안에서 관리하는 표준 방식.

```
┌─────────────────────────────────────────────────────────┐
│            블록 헤더 (Block Header)                      │
│  엔트리 수 | 빈 공간 끝 오프셋 | [크기1, 위치1] [크기2, 위치2] ... │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ← 빈 공간(Free Space) →                               │
│                                                         │
│                  [Record N] ... [Record 2] [Record 1]  │
│                          ← 레코드는 뒤에서부터 채워짐  │
└─────────────────────────────────────────────────────────┘
```

**동작 원리**:
- 삽입: 뒤쪽에서 앞으로 레코드 배치, 헤더에 엔트리 추가
- 삭제: 헤더 엔트리 제거, 인접 레코드 이동하여 단편화 해소 (free space는 연속 유지)
- 레코드 이동 가능 → 외부 포인터는 헤더 엔트리를 가리켜야 함 (레코드 직접 포인터 금지)

### 13.2.4 대형 객체 (LOB: Large Objects)

- **BLOB/CLOB**: 이진/문자 대형 객체
- 레코드 내부 저장 불가 시: 별도 위치에 저장 + 레코드에 포인터 보관
- 선택지:
  1. DB 관리 파일 시스템에 저장
  2. DB 내부에 **B⁺-Tree 파일 조직**으로 저장 (임의 위치 바이트 범위 접근 가능)
  3. 파일 시스템에 경로 문자열로 저장 (단, FK 무결성 검증 불가)

## 13.3 파일 내 레코드 조직 방식

### 13.3.1 힙 파일 조직 (Heap File Organization)

공간이 있는 곳 어디에나 레코드 삽입. 순서 없음.

**자유 공간 맵(Free-Space Map)**:
```
블록 수 = 16인 경우 (3비트 엔트리):
[4][2][1][4][7][3][6][5][1][2][0][1][1][0][5][6]

값 7 → 7/8 이상의 공간이 비어있음
값 0 → 공간 없음

2단계 자유 공간 맵 (100개 엔트리마다 최대값):
[4][7][2][6]  ← 빠른 검색 가능
```

삽입 시 2단계 맵부터 스캔 → 적합한 블록 찾기 → 실제 블록에 삽입.

### 13.3.2 순차 파일 조직 (Sequential File Organization)

레코드를 검색 키(search key) 순서로 정렬하여 저장.

- **삽입 처리**: 동일 블록에 빈 공간 없으면 → **오버플로우 블록(Overflow Block)** 사용
- 시간이 지나면 물리적 순서와 논리적 순서 불일치 → 주기적 **재조직(Reorganization)** 필요

```
10101 → 12121 → 15151 → 22222 → 32222(오버플로우) → 32343 → ...
(포인터 체인으로 연결)
```

### 13.3.3 다중 테이블 클러스터링 파일 조직 (Multitable Clustering)

서로 관련된 여러 릴레이션의 레코드를 같은 블록에 저장.

```
블록 내 구조 (department + instructor, dept_name 기준 클러스터링):
┌──────────────────────────────────────────────────────────┐
│ [dept: Comp.Sci. | Taylor | 100000]                      │
│   [inst: 10101 | Srinivasan | Comp.Sci. | 65000]         │
│   [inst: 45565 | Katz       | Comp.Sci. | 75000]         │
│   [inst: 83821 | Brandt     | Comp.Sci. | 92000]         │
│ [dept: Physics | Watson | 70000]                         │
│   [inst: 33456 | Gold | Physics | 87000]                 │
└──────────────────────────────────────────────────────────┘
```

**장점**: `department ⋈ instructor` 조인 시 블록 접근 횟수 감소.
**단점**: `SELECT * FROM department` 단일 테이블 조회 시 블록당 부서 레코드 수 감소 → 더 많은 블록 접근.

### 13.3.4 테이블 파티셔닝 (Table Partitioning)

```sql
-- 논리적으로 하나의 테이블이지만
SELECT * FROM transaction WHERE year = 2019
-- 내부적으로는 transaction_2019만 접근 (파티션 가지치기)
```

- 파티션별 다른 저장 장치 사용 가능 (최신 데이터는 SSD, 구 데이터는 HDD)

## 13.4 데이터 사전 (Data Dictionary / System Catalog)

데이터베이스의 메타데이터(schema, 통계, 권한 등)를 저장.

```
Relation_metadata (relation_name, num_attrs, storage_org, location)
Attribute_metadata (relation_name, attr_name, domain, position, length)
Index_metadata (index_name, relation_name, index_type, index_attrs)
View_metadata (view_name, definition)
User_metadata (user_name, encrypted_password, group)
```

- 메타데이터 자체도 관계형 테이블로 저장 (self-describing)
- `Relation_metadata`의 위치는 DB 코드에 하드코딩 (부트스트랩 문제)
- DB 시작 시 메모리 내 해시 테이블/트리로 로드 → 모든 쿼리가 사전 조회 필요

## 13.5 데이터베이스 버퍼 (Database Buffer)

### 버퍼 관리자 역할

```
프로그램 요청 → [버퍼 관리자] → 블록이 버퍼에 있으면 주소 반환
                              → 없으면 디스크에서 읽어 버퍼에 로드 후 반환
                                (필요 시 기존 블록 evict → 수정됐으면 디스크 쓰기)
```

### 핀(Pin) / 언핀(Unpin)

```
pin(block)    : 블록을 버퍼에서 제거 불가 상태로 표시 (pin_count++)
unpin(block)  : 제거 가능 상태로 변경 (pin_count--)
pin_count > 0 : evict 불가
```

### 공유/배타 잠금 (버퍼 레벨)

```
Shared Lock   : 읽기용, 여러 프로세스 동시 허용
Exclusive Lock: 쓰기용, 하나만 허용, 다른 공유/배타 없을 때만 획득 가능

순서: pin → lock → 접근 → unlock → unpin
```

### 13.5.1 버퍼 교체 전략

| 전략 | 설명 | 적합한 상황 |
|------|------|-------------|
| **LRU** (Least Recently Used) | 가장 오래 전에 접근된 블록 교체 | 일반 OS, 반복 접근 패턴 |
| **Toss-Immediate** | 더 이상 필요 없는 블록 즉시 해제 | 한 번만 읽히는 블록 (Nested Loop Join의 외부 관계) |
| **MRU** (Most Recently Used) | 가장 최근에 접근된 블록 교체 | Nested Loop Join의 내부 관계 (매 외부 튜플마다 완전 스캔) |

**MRU가 유리한 이유** (중첩 루프 조인 예시):
```python
for each tuple i in instructor:          # 외부: Toss-Immediate
    for each tuple d in department:      # 내부: 매번 전체 스캔
        if i.dept_name == d.dept_name:   # department의 현재 블록 = 곧 다시 필요 없음
            output join result           # → MRU로 즉시 교체 대상
```

### 13.5.2 강제 출력 (Forced Output)

복구 시스템의 요구로 특정 블록을 디스크에 강제로 쓰는 연산. 트랜잭션 커밋 전 로그 레코드 선행 기록(WAL: Write-Ahead Logging)과 연계.

### 13.5.3 로그 디스크와 저널링

```
쓰기 순서 문제:
  포인터 업데이트 → 노드 데이터 쓰기 (정상 순서)
  중간에 크래시 → 불일치 발생

해결: 로그 디스크 (Log Disk)
  1. 모든 쓰기를 순서대로 로그 디스크에 기록 (순차 I/O → 빠름)
  2. 실제 디스크에는 나중에 최적 순서로 쓰기
  3. 크래시 복구 시 로그 재생 → 미완료 쓰기 재실행
```

**저널링 파일 시스템(Journaling FS)**: 로그 디스크 기반 파일 시스템. 크래시 후 빠른 재시작(fsck 불필요).

## 13.6 컬럼 지향 저장 (Column-Oriented Storage)

### 행 지향 vs 열 지향

```
행 지향 (Row Store):           열 지향 (Column Store):
┌──────────────────────┐       ID 파일:   [10101][12121][15151]...
│ID  | name  | salary  │       name 파일: [Srinivasan][Wu][Mozart]...
│10101 Srini  65000    │       salary 파일: [65000][90000][40000]...
│12121 Wu     90000    │
│15151 Mozart 40000    │
└──────────────────────┘
```

### 열 지향 저장의 장점

1. **I/O 감소**: 필요한 열만 읽음. 예) `SELECT AVG(salary)` → salary 파일만 읽기.
2. **캐시 효율**: 같은 열의 연속 값 → 캐시 라인 낭비 없음.
3. **압축 효율**: 동일 타입 값 연속 → RLE·딕셔너리 인코딩 효과 극대화.
4. **벡터 처리(Vector Processing)**: CPU SIMD로 배열 단위 병렬 비교·집계 가능.

### 열 지향 저장의 단점

1. **단일 튜플 조회 비용**: 여러 열 파일을 I/O하여 재조합 필요.
2. **업데이트/삭제 비용**: 압축된 전체 시퀀스 재기록 필요.
3. **압축 해제 비용**: 특정 레코드 접근 시 앞 부분부터 압축 해제 필요.

### ORC/Parquet 포맷

```
ORC 파일 구조:
┌──────────────────────────────────────────────────────┐
│                    Stripe 1 (~250MB)                  │
│ ┌─────────────────────────────────────────────────┐  │
│ │ Index Data: 10,000 값마다 오프셋 저장 (빠른 탐색) │  │
│ ├─────────────────────────────────────────────────┤  │
│ │ Row Data:                                        │  │
│ │   [Col1 압축 데이터][Col2 압축 데이터]...        │  │
│ ├─────────────────────────────────────────────────┤  │
│ │ Stripe Footer                                   │  │
│ └─────────────────────────────────────────────────┘  │
│                    Stripe 2                           │
│                    ...                                │
│ File Footer                                          │
└──────────────────────────────────────────────────────┘
```

**스트라이프 인덱스**: 10,000개 값마다 파일 내 오프셋 저장 → 원하는 튜플 그룹으로 직접 점프 가능. 선택 조건 불만족 그룹 건너뜀.

### 하이브리드 행/열 저장

- **SAP HANA**: 트랜잭션용 행 저장 + 분석용 열 저장 동시 지원.
- 새 튜플은 행 저장에 삽입, 이후 열 저장으로 마이그레이션.

## 13.7 인메모리 데이터베이스 저장 구조

### 차이점: 버퍼 기반 DB vs 인메모리 DB

```
버퍼 기반 DB:
  record ptr = (block_id, offset)
  접근 시: 버퍼 해시 조회 → 버퍼 적재 → 간접 접근
  비용: 다수의 CPU 사이클

인메모리 DB:
  record ptr = 직접 메모리 포인터
  접근 시: 포인터 역참조 (단일 CPU 명령)
  조건: 레코드가 이동하지 않아야 함
```

### 인메모리 컬럼 저장

```
논리 배열 = 여러 물리 배열로 분할 (새 레코드 삽입 시 재할당 회피)

간접 테이블(Indirection Table):
  [ptr1 → 물리배열1][ptr2 → 물리배열2]...

i번째 요소 접근: 간접 테이블 → 해당 물리배열 → 오프셋 계산
```

## 13.8 핵심 요약

```
저장 구조 의사결정 트리

데이터 접근 패턴?
├── 단일 레코드 점조회 → 해시/B⁺-Tree 인덱스 + 힙
├── 범위 조회 + 정렬 → 순차 파일 or B⁺-Tree 파일 조직
├── 빈번한 조인 (JOIN) → 멀티테이블 클러스터링
└── 분석 집계 쿼리 → 컬럼 지향 저장 (ORC/Parquet)
```

| 개념 | 핵심 포인트 |
|------|------------|
| 슬롯페이지 | 블록 내 가변 레코드 관리, 헤더 간접 포인터로 이동 허용 |
| 자유 리스트 | 삭제 레코드를 링크드 리스트로 추적, 삽입 시 재활용 |
| 힙 파일 | 순서 없음, FSM으로 빈 공간 추적, 가장 단순 |
| 순차 파일 | 검색 키 정렬, 오버플로우 블록 → 주기적 재조직 필요 |
| 멀티테이블 클러스터링 | 조인 성능↑, 단일 테이블 스캔 성능↓ |
| 데이터 사전 | 메타데이터를 DB 테이블로 저장, 시작 시 메모리 로드 |
| LRU/MRU | 쿼리 접근 패턴에 따른 버퍼 교체 전략 |
| 컬럼 지향 저장 | 분석 쿼리 최적화 (I/O↓, 압축↑, SIMD↑), OLAP에 적합 |
| 인메모리 DB | 직접 포인터 사용, 버퍼 관리자 제거, 간접 테이블로 배열 확장 |
