# Chapter 15: Query Processing

## 개요

쿼리 처리는 SQL을 내부 표현(관계 대수)으로 변환하고 실제 데이터를 추출하는 모든 과정을 포함한다. 비용 모델(I/O 블록 전송 + 랜덤 접근), 셀렉션 알고리즘, 외부 정렬, 조인 알고리즘(중첩 루프/블록 중첩 루프/인덱스 중첩 루프/병합 조인/해시 조인), 집합·중복 제거·집계 연산, 파이프라이닝 vs 구체화 평가를 다룬다.

---

## 15.1 개요

```
SQL 쿼리
    │
    ▼ 파싱·변환 (parser & translator)
관계 대수 식 (relational-algebra expression)
    │
    ▼ 최적화 (optimizer)  ← 통계 정보(statistics) 사용
실행 계획 (query-evaluation plan)
    │ 평가 원시(evaluation primitive) = 알고리즘이 명시된 연산
    ▼
쿼리 실행 엔진 (query-execution engine)
    │
    ▼
결과 출력
```

**평가 계획(Evaluation Plan)**: 각 연산에 대해 구체적인 알고리즘과 사용할 인덱스를 명시한 실행 계획.

```
예: Πsalary(σsalary<75000(instructor))
    vs Πsalary(σsalary<75000(instructor)) - 인덱스 1 사용
```

## 15.2 쿼리 비용 측정

### 비용 모델 파라미터

```
tT = 블록 하나 전송 시간 (HDD: ~0.1ms/4KB, SSD SATA: ~10μs/4KB)
tS = 랜덤 I/O 시작 시간 / 탐색 시간 (HDD: ~4ms, SSD SATA: ~90μs)

비용 = b × tT + S × tS
        블록 전송 수    랜덤 접근 수
```

| 저장 장치 | tS | tT (4KB) |
|----------|----|---------|
| HDD | ~4 ms | ~0.1 ms |
| SSD (SATA) | ~90 μs | ~10 μs |
| SSD (NVMe PCIe) | ~20~60 μs | ~2 μs |
| 메인 메모리 | <100 ns | <1 μs |

**단순화 가정**: 비용 모델에는 최종 결과 출력 비용 미포함. 메모리 M 블록 크기 파라미터 사용.

## 15.3 셀렉션 연산 (Selection)

### 알고리즘별 비용 요약

| 알고리즘 | 비용 | 조건 |
|---------|------|------|
| **A1** 선형 탐색 | tS + br×tT | 항상 사용 가능 |
| **A1** 키 동등 조건 | tS + (br/2)×tT | 평균 |
| **A2** 클러스터링 B⁺-Tree, 키 동등 | (hi+1)×(tT+tS) | 클러스터링 인덱스 |
| **A3** 클러스터링 B⁺-Tree, 비키 동등 | hi×(tT+tS) + tS + b×tT | b = 결과 블록 수 |
| **A4** 보조 B⁺-Tree, 키 동등 | (hi+1)×(tT+tS) | 보조 인덱스 |
| **A4** 보조 B⁺-Tree, 비키 동등 | (hi+n)×(tT+tS) | n = 레코드 수 (최악) |
| **A5** 클러스터링, 비교 조건 | A3와 동일 | ≤, ≥ |
| **A6** 보조, 비교 조건 | A4와 동일 | 비용 매우 높을 수 있음 |

- hi = B⁺-Tree 높이. 실무에서 내부 노드는 버퍼에 상주 → hi ≈ 1 가정.
- 보조 인덱스 + 비키: n이 크면 선형 탐색보다 느릴 수 있음.

### 비트맵 인덱스 스캔 (PostgreSQL)

보조 인덱스가 있으나 결과 레코드 수 불확실 시 사용하는 하이브리드:

```
1. 비트맵 초기화 (릴레이션 블록 수만큼 비트, 모두 0)
2. 보조 인덱스 스캔 → 매칭 레코드의 블록 번호 → 해당 비트 1로 설정
3. 비트맵을 스캔하여 비트=1인 블록만 순차 읽기 → 매칭 튜플 필터링

최선: 인덱스 스캔과 동일  최악: 선형 탐색보다 약간 비쌈
```

### 복합 셀렉션

- **A7 (합집합 조건, 하나의 인덱스)**: 가장 선택적인 단순 조건으로 인덱스 탐색 후 메모리에서 나머지 조건 테스트.
- **A8 (합집합 조건, 복합 인덱스)**: 복합 검색 키가 있으면 직접 사용.
- **A9 (합집합 조건, 포인터 교집합)**: 각 조건에 대해 레코드 포인터 목록 획득 후 교집합. 정렬 후 I/O 최소화.
- **A10 (합집합 조건, 포인터 합집합)**: 각 조건 포인터 합집합. 하나라도 인덱스 없으면 선형 탐색 불가피.

## 15.4 정렬 (Sorting)

대용량 릴레이션을 위한 **외부 정렬-병합(External Sort-Merge)**.

### 알고리즘

```
1단계: 런(run) 생성
  i = 0
  repeat:
    메모리 M 블록 읽기 → 인메모리 정렬 → 런 파일 Ri에 쓰기
    i++
  until 릴레이션 끝

2단계: N-way 병합 (N = 초기 런 수, N < M 가정)
  각 Ri에서 1 블록씩 버퍼에 로드
  repeat:
    모든 버퍼 중 최솟값 선택 → 출력
    해당 버퍼 소진 시 다음 블록 읽기
  until 모든 버퍼 소진
```

```
초기 런 수: ⌈br / M⌉
병합 패스 수: ⌈log⌊M/bb⌋-1(br/M)⌉   (bb = 런당 버퍼 블록 수)
전체 블록 전송: br × (2 × 패스 수 + 1)
디스크 탐색: 2⌈br/M⌉ + ⌈br/bb⌉ × (2×패스수 − 1)
```

**예시** (M=3, bb=1, br=12): 60 블록 전송 + 44 탐색.

## 15.5 조인 연산 (Join)

### 기본 예시 릴레이션

```
student:  nstudent = 5,000,  bstudent = 100 블록
takes:    ntakes = 10,000,   btakes  = 400 블록
```

### 15.5.1 중첩 루프 조인 (Nested-Loop Join)

```python
for each tuple tr in r:        # 외부 릴레이션
    for each tuple ts in s:    # 내부 릴레이션
        if (tr, ts) satisfy θ: output tr·ts
```

| 경우 | 블록 전송 | 탐색 |
|------|---------|------|
| 최악 (1 블록 버퍼) | nr × bs + br | nr + br |
| 최선 (전부 메모리) | br + bs | 2 |

**예**: student(외부) ⋈ takes(내부), 최악 = 5,000×400+100 = **2,000,100** 전송.

### 15.5.2 블록 중첩 루프 조인 (Block Nested-Loop Join)

```python
for each block Br of r:
    for each block Bs of s:
        모든 (tr, ts) 쌍 테스트
```

| 경우 | 블록 전송 | 탐색 |
|------|---------|------|
| 최악 | br × bs + br | 2 × br |
| 최선 | br + bs | 2 |

**예**: 최악 = 100×400+100 = **40,100** 전송 + 200 탐색. 기본 중첩 루프 대비 50배 개선.

**최적화**:
- 작은 릴레이션을 외부로 사용
- 메모리 M 블록 중 M-2 블록을 외부로 한 번에 읽기 → 내부 스캔 횟수 ⌈br/(M-2)⌉로 감소
- 내부 루프 방향 교대 (Forward/Backward) → 버퍼 재사용

### 15.5.3 인덱스 중첩 루프 조인 (Indexed Nested-Loop Join)

내부 릴레이션 조인 속성에 인덱스가 있을 때:

```
비용 = br × (tT + tS) + nr × c
c = 내부 릴레이션에서 단일 셀렉션 비용 (인덱스 탐색)
```

**예**: student ⋈ takes, takes.ID에 B⁺-Tree 클러스터링 인덱스 (높이 4, 리프 1회 I/O):
→ 100 + 5,000×5 = **25,100** 접근 (블록 전송 + 탐색 합산).

블록 중첩 루프(40,100 전송 + 200 탐색) 대비 블록 전송은 줄었지만 탐색 비용이 높음.

### 15.5.4 병합 조인 (Merge Join / Sort-Merge Join)

두 릴레이션이 **조인 속성 기준으로 정렬**되어 있을 때 단일 패스 처리.

```
pr = 첫 r 튜플 포인터;  ps = 첫 s 튜플 포인터
while (ps ≠ null and pr ≠ null):
    s에서 같은 JoinAttrs 값을 가진 집합 Ss 수집
    r에서 Ss의 조인 키보다 작은 튜플 건너뜀
    r에서 Ss와 같은 조인 키 튜플 모두 → Ss의 각 튜플과 출력
```

**비용** (정렬된 경우):

```
블록 전송: br + bs
탐색:      ⌈br/bb⌉ + ⌈bs/bb⌉  (bb = 릴레이션당 버퍼 블록 수)
```

정렬이 안 된 경우: 외부 정렬 비용 추가.

**하이브리드 병합 조인**: 한 릴레이션은 정렬, 다른 쪽은 보조 B⁺-Tree 인덱스 보유 시 사용. 인덱스 리프를 병합한 후 물리 주소로 정렬하여 I/O 최적화.

### 15.5.5 해시 조인 (Hash Join)

해시 함수 h로 두 릴레이션을 파티셔닝하여 파티션 단위로 조인.

```
1. 파티셔닝:
   for each ts in s: Hsi = Hsi ∪ {ts}  (i = h(ts[JoinAttrs]))
   for each tr in r: Hri = Hri ∪ {tr}  (i = h(tr[JoinAttrs]))

2. 파티션별 조인:
   for i = 0 to nh:
     Hsi를 인메모리 해시 인덱스로 구축 (빌드 단계)
     Hri로 해시 인덱스 탐색 (프로브 단계)
     매칭 튜플 출력
```

**nh 최솟값**: ⌈bs/M⌉ (빌드 릴레이션 s의 각 파티션이 메모리에 맞아야 함).

**비용** (재귀 파티셔닝 없을 때):

```
블록 전송: 3(br + bs) + 4nh   (≈ 3(br+bs))
탐색:      2(⌈br/bb⌉ + ⌈bs/bb⌉) + 2nh
```

**예**: M=20, student ⋈ takes, nh=5:
- 블록 전송: 3(100+400) = **1,500** + 오버헤드
- 탐색: 2(⌈100/3⌉+⌈400/3⌉) ≈ **336**

**재귀 파티셔닝**: M < nh 시 파티셔닝을 반복. M ≥ √bs 이면 1패스로 충분.

**하이브리드 해시 조인**: 메모리 여유분으로 첫 번째 파티션(s0)을 메모리에 유지 → 해당 파티션 디스크 쓰기/읽기 절약.

### 조인 알고리즘 비교

| 알고리즘 | 블록 전송 (최악) | 탐색 | 특징 |
|---------|---------------|------|------|
| 중첩 루프 | nr×bs + br | nr+br | 인덱스·정렬 불필요 |
| 블록 중첩 루프 | br×bs + br | 2br | 작은 릴레이션을 외부로 |
| 인덱스 중첩 루프 | br + nr×c | br + nr | 내부에 인덱스 필요 |
| 병합 조인 | br + bs | ⌈br/bb⌉+⌈bs/bb⌉ | 정렬 필요, 범위 쿼리 적합 |
| 해시 조인 | 3(br+bs) | ~336 | 등가 조인, 균등 분포 전제 |

## 15.6 기타 연산

### 중복 제거 (Duplicate Elimination)

**정렬 기반**: 외부 정렬 후 인접 중복 제거. 런 생성 중 조기 제거 가능.
**해시 기반**: 전체 튜플로 파티셔닝 후 각 파티션에서 인메모리 해시 인덱스로 중복 제거.

### 집합 연산 (Set Operations)

두 릴레이션 정렬 후 단일 스캔으로 처리 (비용 br + bs):

| 연산 | 처리 방법 |
|------|---------|
| r ∪ s | 동시 스캔, 동일 튜플은 하나만 출력 |
| r ∩ s | 양쪽 모두에 존재하는 튜플 출력 |
| r − s | r에만 존재하는 튜플 출력 |

해시 기반: 같은 해시 함수로 파티셔닝 후 각 파티션별 인메모리 처리.

### 외부 조인 (Outer Join)

**방법 1**: Inner join 계산 → r − ΠR(q1)의 NULL-패딩 튜플 추가.
**방법 2**: 중첩 루프 조인 확장 → 매칭되지 않는 외부 튜플에 NULL 추가.
**병합/해시 조인 확장**: 매칭되지 않는 튜플을 NULL-패딩하여 결과에 추가.

### 집계 (Aggregation)

GROUP BY 속성 기준으로 정렬 또는 해시 → 각 그룹에 집계 적용.

**온더플라이(On-the-fly)**: sum/min/max/count/avg를 그룹 구성 중 점진적으로 계산:
- sum/count: 누적 합산
- avg: sum + count 별도 유지 → 마지막에 나눔
- min/max: 현재 최솟값/최댓값과 비교

전체 결과가 메모리에 맞으면 br 블록 전송만으로 완료 (기록·읽기 없음).

## 15.7 표현식 평가

### 15.7.1 구체화 평가 (Materialized Evaluation)

각 연산의 결과를 임시 릴레이션에 저장하여 다음 연산에 사용.

```
πname(σbuilding="Watson"(department) ⋈ instructor)

단계:
1. σbuilding="Watson"(department) → 임시 r1 (디스크 쓰기)
2. r1 ⋈ instructor → 임시 r2 (디스크 쓰기)
3. πname(r2) → 최종 결과
```

**비용**: 각 연산 비용 합 + 중간 결과 디스크 쓰기 비용.

### 15.7.2 파이프라인 평가 (Pipelined Evaluation)

한 연산의 출력 튜플이 생성되는 즉시 다음 연산에 전달 → 임시 파일 저장 없음.

**장점**:
1. 중간 결과 디스크 읽기/쓰기 비용 제거
2. 첫 결과를 빠르게 생성 (응답 시간↓)

#### 수요 기반 파이프라인 (Demand-Driven, Lazy/Pull)

이터레이터 인터페이스: `open()`, `next()`, `close()`.

```python
# 선형 탐색 셀렉션 이터레이터
def open():   file_scan 시작
def next():   선택 조건 만족하는 다음 튜플 반환
def close():  파일 스캔 종료
```

#### 생산자 기반 파이프라인 (Producer-Driven, Eager/Push)

각 연산이 별도 프로세스/스레드로 실행. 출력 버퍼가 차면 대기, 소비자가 읽으면 재개.
현대 CPU에서 함수 호출 오버헤드 감소로 수요 기반보다 효율적.

#### 파이프라인 단계 (Pipeline Stage)

```
파이프라인 엣지 (pipelined edge): 두 연산 동시 실행, 즉시 전달
블로킹 엣지 (blocking/materialized edge): 완전 계산 후 다음 연산

예: 해시 조인 서브연산
  파티셔닝r ← [파이프라인] ← 릴레이션 r
  파티셔닝s ← [파이프라인] ← 릴레이션 s
  빌드-프로브 ← [블로킹] ← 파티셔닝r, 파티셔닝s
  집계 ← [파이프라인] ← 빌드-프로브
```

**블로킹 연산**: 정렬, 해시 조인 등 → 모든 입력을 봐야 결과 생성 가능.

### 15.7.3 연속 스트림 파이프라이닝

스트리밍 데이터의 연속 쿼리에는 생산자 기반 파이프라인 사용. 텀블링 윈도우: 타임스탬프가 더 높은 튜플 도착 시 윈도우 완료 신호.

## 15.8 인메모리 쿼리 처리

### 15.8.1 캐시 인식 알고리즘

```
CPU 캐시 계층:
  L1: ~64KB, ~1ns
  L2: ~256KB, ~5ns
  L3: ~10MB, ~10~15ns
  메인 메모리: ~50~100ns
```

**캐시 친화적 정렬**:
- 런 크기를 L3 캐시 크기(수 MB)에 맞춤 → 런 내 정렬 시 캐시 미스 최소화
- 병합 단계: 순차 접근 → 프리패치 효과

**캐시 친화적 해시 조인**:
- 빌드 릴레이션을 더 작은 파티션으로 나눠 각 파티션 + 인덱스가 캐시에 맞도록 처리

### 15.8.2 쿼리 컴파일 (Query Compilation)

전통적 쿼리 프로세서: 쿼리 계획을 인터프리터로 실행 → 메타데이터 반복 조회, 함수 호출 오버헤드.

**컴파일드 쿼리**: 쿼리 계획을 머신 코드/바이트코드로 컴파일 → 속성 오프셋 컴파일 타임 상수화, 함수 호출 최소화 → 최대 10배 속도 향상.

### 15.8.3 컬럼 지향 저장에서의 쿼리 처리

분석 쿼리에서 일부 속성만 접근 → 해당 컬럼 파일만 읽기. 벡터 처리 명령어(SIMD)로 다수 속성 값에 병렬 연산 적용.

## 15.9 핵심 요약

```
조인 선택 가이드

인덱스 없음, 메모리 부족 → 블록 중첩 루프 조인
내부 릴레이션에 인덱스 존재 → 인덱스 중첩 루프 조인
두 릴레이션 정렬됨 → 병합 조인 (비용 최소)
두 릴레이션 미정렬, 대규모 → 해시 조인 (병렬성 좋음)
```

| 개념 | 핵심 포인트 |
|------|------------|
| 비용 모델 | b×tT + S×tS, HDD tS≫tT, SSD는 tS 대폭 감소 |
| A4 보조 인덱스 비키 | 레코드당 랜덤 I/O → n이 크면 선형 탐색보다 느림 |
| 외부 정렬-병합 | 런 생성 + N-way 병합, 비용 br×(2×패스수+1) |
| 블록 중첩 루프 | br×bs + br, 작은 릴레이션 외부로 |
| 병합 조인 | 정렬 전제, br+bs 전송, 순차 I/O |
| 해시 조인 | 3(br+bs), 파티셔닝 + 빌드/프로브 |
| 파이프라이닝 | 중간 결과 디스크 기록 제거, 빠른 첫 응답 |
| 수요 기반 vs 생산자 기반 | open/next/close vs 버퍼+스레드 |
| 컴파일드 쿼리 | 머신 코드로 컴파일, 인터프리터 오버헤드 제거 |
