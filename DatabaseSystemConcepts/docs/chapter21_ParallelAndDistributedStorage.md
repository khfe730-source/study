# Chapter 21: Parallel and Distributed Storage

## 개요

병렬·분산 스토리지의 핵심은 데이터 파티셔닝이다. 라운드 로빈·해시·범위 파티셔닝 전략, 스큐 처리(가상 노드/동적 재파티셔닝), 복제(위치 결정·일관성·업데이트 프로토콜), 병렬 인덱싱(로컬/글로벌 인덱스), 분산 파일 시스템, 병렬 Key-Value 스토어를 다룬다.

---

## 21.2 데이터 파티셔닝

### 수평 파티셔닝 전략

n개 노드 N1, N2, ..., Nn에 튜플을 분산.

| 전략 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **라운드 로빈** | i번째 튜플 → 노드 N((i-1) mod n)+1 | 균등 분산 보장 | 포인트/범위 쿼리에 비효율 |
| **해시 파티셔닝** | h(t[A]) → 노드 번호 | 포인트 쿼리 효율적, 균등 분산 | 범위 쿼리 비효율 |
| **범위 파티셔닝** | 분할 벡터 [v1..vn-1]로 범위 할당 | 포인트+범위 쿼리 효율 | 스큐 발생 가능 |

```
범위 파티셔닝 예시:
파티셔닝 벡터: [15, 40, 75]

값 < 15    → Node 1
15 ≤ 값 < 40 → Node 2
40 ≤ 값 < 75 → Node 3
값 ≥ 75    → Node 4
```

### 파티셔닝 전략별 쿼리 지원

| 쿼리 유형 | 라운드 로빈 | 해시 | 범위 |
|---------|-----------|------|------|
| 전체 스캔 | ✅ (최적) | ✅ | ✅ |
| 포인트 쿼리 | ❌ (전체 스캔) | ✅ (단일 노드) | ✅ (단일 노드) |
| 범위 쿼리 | ❌ | ❌ | ✅ (연속 노드) |

## 21.3 파티셔닝 스큐 처리

**데이터 분포 스큐**: 일부 파티션에 튜플이 집중.
- **속성 값 스큐**: 특정 파티셔닝 속성 값이 많은 튜플에 등장
- **파티션 스큐**: 파티셔닝 벡터가 불균등하게 선택됨

**스큐의 영향**: 가장 큰 파티션의 처리 시간이 전체 지연 결정 → 병렬화 효과 저하.

### 21.3.1 균형 범위 파티셔닝 벡터

```
방법: 히스토그램(분포 정보) 사용
  1. 속성 값 빈도 히스토그램 사전 계산
  2. 히스토그램 기반으로 균등 분할 벡터 결정
  3. 랜덤 샘플링으로 빠른 히스토그램 근사 가능

단점: 정적(static) → 데이터 변화에 미적응, 주기적 재계산 필요
```

### 21.3.2 가상 노드 파티셔닝 (Virtual Node)

실제 노드보다 훨씬 많은 **가상 노드**를 생성 후 가상 노드를 실제 노드에 매핑.

```
실제 노드 수: n = 4
가상 노드 수: m = 40 (10배)

매핑 테이블: virtual_to_real_map[40]
  VN 0 → N1, VN 1 → N2, VN 2 → N3, ...
  (라운드 로빈 또는 부하 기반 매핑)

장점:
  - 한 범위에 많은 튜플 → 여러 가상 노드로 분산 → 여러 실제 노드에 분포
  - 탄력성: 새 노드 추가 시 일부 가상 노드만 이동 → 무중단 확장
```

**일관성 해싱(Consistent Hashing)**: 키와 노드 ID 모두 동일 해시 공간에 매핑. 노드 추가/제거 시 최소한의 데이터 이동. P2P 방식으로 마스터 없이 라우팅 가능.

### 21.3.3 동적 재파티셔닝 (Tablets)

```
테이블 → 여러 태블릿(tablet, 가상 노드) 분할

파티션 테이블 (Partition Table):
  키 범위      │ 태블릿 ID  │ 노드 ID
  [─∞, 2012)  │ Tablet0   │ Node0
  [2012, 2013) │ Tablet1   │ Node1
  [2013, 2014) │ Tablet2   │ Node2
  ...

태블릿 분할(Split): 태블릿이 너무 크면 키 범위를 반으로 분할
  [2017, ∞) → [2017, 2018) + [2018, ∞)

태블릿 이동(Move): 과부하 노드의 태블릿을 다른 노드로 이동

파티션 테이블 관리: 마스터 노드 또는 분산(복제 후 모든 노드 캐시)
```

## 21.4 복제 (Replication)

### 복제의 필요성

n개 노드 시스템에서 단일 노드 장애 확률이 높음.
```
단일 노드 MTTF = 5년
100개 노드 시스템 → 평균 18일마다 장애!
→ 복제 없이는 데이터 손실 불가피
```

### 복제 위치 전략

```
랙 내 복제: 같은 랙의 다른 노드 (단일 노드 장애 대응)
랙 간 복제: 다른 랙의 노드 (랙 장애 대응)
데이터센터 간 복제: 지리적 분리 (재해 대응)
```

**부하 분산**: 한 노드의 모든 파티션 복제본을 단일 노드에 두지 않음.
→ 장애 시 n-1개 노드가 추가 부담을 균등 분담.

### 21.4.2 복제 업데이트 프로토콜

| 프로토콜 | 설명 | 장점 | 단점 |
|---------|------|------|------|
| **2PC (2단계 커밋)** | 모든 복제본에 원자적 업데이트 | 강한 일관성 | 블로킹 위험 |
| **영속 메시징** | 마스터에 먼저 업데이트, 나중에 전파 | 빠른 커밋 | 일시적 불일치(최종 일관성) |
| **합의 프로토콜(Consensus)** | 다수결 기반 장애 허용 | 블로킹 없음, 마스터 불필요 | 복잡성 |

## 21.5 병렬 인덱싱

### 로컬 인덱스 vs 글로벌 인덱스

| 종류 | 위치 | 사용 시나리오 |
|------|------|-------------|
| **로컬 인덱스** | 각 노드의 로컬 데이터에 구축 | 파티셔닝 속성 기반 쿼리 |
| **글로벌 기본 인덱스** | 파티셔닝 속성의 로컬 인덱스들 | 파티셔닝 키로 단일 노드 조회 |
| **글로벌 보조 인덱스** | 비파티셔닝 속성의 별도 파티셔닝 | 임의 속성 효율적 조회 |

```
예시: student 릴레이션, 파티셔닝 속성 = ID
       글로벌 보조 인덱스 on name

구현:
  1. (name, ID) 쌍으로 구성된 index_name 릴레이션 생성
  2. index_name을 name으로 파티셔닝
  3. name에 로컬 인덱스 구축

쿼리 "name = 'Zhang'" 처리:
  1. name 파티션 함수로 index_name 태블릿 찾기
  2. 로컬 인덱스로 해당 ID 조회
  3. ID 기반 파티션에서 실제 튜플 조회
```

## 21.6 분산 파일 시스템 (Distributed File Systems)

**HDFS (Hadoop Distributed File System)**:
```
NameNode (마스터): 파일→블록 매핑 메타데이터 관리
DataNode (슬레이브): 실제 블록 저장

파일 쓰기: NameNode에서 DataNode 목록 획득 → 파이프라인 복제
블록 크기: 기본 128MB
복제 인수: 기본 3 (단일 노드 장애, 랙 장애 허용)
```

## 21.7 병렬 Key-Value 스토어

NoSQL 스토리지 시스템. SQL/트랜잭션 미지원 대신 극한 확장성 추구.

### HBase / Google Bigtable 아키텍처

```
행 키(Row Key) 기반 정렬된 테이블
열 패밀리(Column Family)로 속성 그룹화
타임스탬프로 다중 버전 관리

구조: 행키 + 열패밀리 + 컬럼 + 타임스탬프 → 값
```

**태블릿 서버(Tablet Server)**: 특정 키 범위의 태블릿 서빙.
**마스터 서버**: 태블릿 할당 및 부하 분산.

### Apache Cassandra / Amazon DynamoDB

```
분산 해시 테이블 기반
일관성 해싱으로 노드에 데이터 분배
구성 가능한 복제 인수 및 일관성 수준:
  R + W > N → 강한 일관성 (R=읽기 쿼럼, W=쓰기 쿼럼, N=복제 수)
```

## 21.8 핵심 요약

```
파티셔닝 전략 선택 가이드:

전체 스캔 위주  → 라운드 로빈
포인트 쿼리 위주 → 해시 파티셔닝
범위 쿼리 포함  → 범위 파티셔닝 (균형 벡터 필요)
스큐 처리 필요  → 가상 노드 + 동적 태블릿 분할
```

| 개념 | 핵심 포인트 |
|------|------------|
| 라운드 로빈 | 균등 분산, 포인트/범위 쿼리 비효율 |
| 해시 파티셔닝 | 포인트 쿼리 O(1), 범위 쿼리 전체 스캔 필요 |
| 범위 파티셔닝 | 범위 쿼리 효율적, 스큐 주의 |
| 가상 노드 | 스큐 완화, 탄력적 확장 지원 |
| 태블릿 분할 | 동적 재파티셔닝, B⁺-Tree 노드 분할과 유사 |
| 일관성 해싱 | P2P 방식, 노드 추가/제거 시 최소 데이터 이동 |
| 복제 위치 | 랙 내+랙 간+데이터센터 간 계층적 복제 |
| 글로벌 보조 인덱스 | 비파티셔닝 속성의 별도 파티셔닝된 인덱스 |
