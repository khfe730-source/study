# Chapter 18: Concurrency Control

## 개요

동시성 제어는 트랜잭션 고립성을 보장하는 메커니즘이다. 잠금 기반 프로토콜(2PL, 트리 프로토콜), 교착 상태 처리, 다중 단위 잠금, 팬텀 처리, 타임스탬프 순서화, 낙관적 검증, 다중 버전(MVCC), 스냅샷 격리(SI/SSI), 인덱스 동시성 제어를 다룬다.

---

## 18.1 잠금 기반 프로토콜

### 잠금 모드

| 모드 | 심볼 | 가능한 연산 |
|------|------|-----------|
| Shared | S | 읽기만 |
| Exclusive | X | 읽기 + 쓰기 |

**잠금 호환성 행렬**:

```
     S    X
  S  ✅   ❌
  X  ❌   ❌

comp(A, B) = true → A 모드가 B 모드와 호환 → 동시 허용
```

- `lock-S(Q)`, `lock-X(Q)`, `unlock(Q)` 명령어
- 같은 데이터 항목에 여러 S-Lock 동시 가능
- X-Lock은 어떤 잠금도 없을 때만 획득 가능

### 18.1.2 굶주림(Starvation) 방지

잠금 요청 시 두 조건 모두 만족해야 허용:
1. 현재 호환되지 않는 잠금이 없음
2. 먼저 요청한 대기 중인 트랜잭션이 없음

### 18.1.3 2단계 잠금 프로토콜 (Two-Phase Locking, 2PL)

```
성장 단계(Growing Phase): 잠금 획득만
수축 단계(Shrinking Phase): 잠금 해제만

잠금 포인트(Lock Point): 성장 단계의 마지막 잠금 획득 시점
→ 잠금 포인트 순서로 트랜잭션 직렬화 가능
```

2PL은 **충돌 직렬가능성**을 보장하지만, **교착 상태는 방지하지 않음**.

**엄격 2PL(Strict 2PL)**: 배타 잠금을 트랜잭션 커밋까지 보유 → 연쇄 롤백 방지.

**강성 2PL(Rigorous 2PL)**: 모든 잠금을 커밋까지 보유 → 커밋 순서로 직렬화.

**잠금 변환(Lock Conversion)**:
- Upgrade(S→X): 성장 단계에서만
- Downgrade(X→S): 수축 단계에서만

### 18.1.4 잠금 관리자 구현

```
잠금 테이블(Lock Table): 해시 테이블 (데이터 항목 → 잠금 요청 연결 리스트)
각 항목: (트랜잭션ID, 잠금 모드, 승인 여부)

┌──────────────────────────────────────────────────────┐
│ I4 → [T1: X, 승인] → [T23: S, 대기]                │
│ I7 → [T23: S, 승인] → [T8: X, 대기]                │
│ I912 → [T23: S, 승인] → [T1: S, 승인]              │
└──────────────────────────────────────────────────────┘
```

### 18.1.5 그래프 기반 프로토콜 (Tree Protocol)

데이터 항목에 부분 순서(→)가 정의되어 있을 때 사용.

**트리 프로토콜 규칙**:
1. 첫 잠금은 어느 항목이든 가능
2. 이후 Q를 잠그려면 Q의 부모가 현재 잠겨 있어야 함
3. 잠금 해제는 언제든 가능
4. 한 번 잠금 해제한 항목은 다시 잠글 수 없음

**장점**: 교착 상태 없음, 조기 잠금 해제 가능 → 높은 동시성.
**단점**: 경로 상의 불필요한 항목도 잠궈야 할 수 있음.

## 18.2 교착 상태 처리

### 교착 상태 정의

```
{T0, T1, ..., Tn}: T0는 T1이 가진 항목을 기다리고,
                   T1는 T2가 가진 항목을 기다리고, ...,
                   Tn는 T0가 가진 항목을 기다림 → 순환 대기
```

### 18.2.1 교착 상태 방지

**방법 1: 전체 선순위 잠금** — 모든 잠금을 실행 전에 획득. 단점: 실용성↓, 자원 활용률↓.

**방법 2: 타임스탬프 기반 선점**:

| 기법 | 규칙 | 결과 |
|------|------|------|
| **Wait-Die** (비선점) | Ti가 더 오래됨(작은 TS) → 대기; Ti가 더 젊음 → 롤백(dies) | 오래된 트랜잭션이 대기 |
| **Wound-Wait** (선점) | Ti가 더 오래됨 → Tj 선점(roll back Tj); Ti가 더 젊음 → 대기 | 오래된 트랜잭션이 선점 |

두 기법 모두 불필요한 롤백 발생 가능. 단, **기아(Starvation) 없음** (재시작 시 기존 타임스탬프 유지).

**방법 3: 잠금 타임아웃** — 지정 시간 초과 시 트랜잭션 롤백. 간단하나 기아 가능, 적절한 타임아웃 설정 어려움.

### 18.2.2 교착 상태 탐지 및 복구

**대기 그래프(Wait-for Graph)**:
- 노드: 트랜잭션
- 엣지 Ti → Tj: Ti가 Tj가 가진 항목을 기다림
- **사이클 존재 ↔ 교착 상태**

```
T18 → T20 → T19 → T18  (사이클 → 교착!)
→ T18, T19, T20 중 하나를 롤백
```

**피해자(Victim) 선택 기준**: 최소 비용 (실행 시간, 사용 자원, 관련 트랜잭션 수).
**부분 롤백**: 교착 해소에 필요한 최소한까지만 롤백.
**기아 방지**: 롤백 횟수를 비용 요소에 포함.

## 18.3 다중 단위 잠금 (Multiple Granularity)

데이터 항목의 크기를 다단계로 정의하여 유연한 잠금 수행.

```
계층 구조 (루트 → 리프 순):
DB → 영역(Area) → 파일(File) → 레코드(Record)

상위 노드 잠금 → 하위 모든 노드 암묵적 잠금
```

**의도 잠금(Intention Locks)**:

| 모드 | 의미 |
|------|------|
| IS (Intention Shared) | 하위에서 S-Lock 수행 예정 |
| IX (Intention Exclusive) | 하위에서 X-Lock 수행 예정 |
| SIX (Shared + Intention Exclusive) | 현재 노드 S-Lock + 하위에서 X-Lock 예정 |

**호환성 행렬**:

```
     IS   IX   S    SIX  X
IS   ✅   ✅   ✅   ✅   ❌
IX   ✅   ✅   ❌   ❌   ❌
S    ✅   ❌   ✅   ❌   ❌
SIX  ✅   ❌   ❌   ❌   ❌
X    ❌   ❌   ❌   ❌   ❌
```

**프로토콜 규칙**:
1. 잠금은 루트→리프 순으로 획득
2. 해제는 리프→루트 순으로
3. S 또는 IS: 부모가 IX 또는 IS
4. X, SIX, 또는 IX: 부모가 IX 또는 SIX
5. 한 번 잠금 해제 후 새 잠금 금지 (2PL)
6. 자식이 잠겨있으면 부모 해제 불가

**잠금 에스컬레이션(Lock Escalation)**: 튜플 잠금이 너무 많으면 테이블 수준 잠금으로 교체.

## 18.4 삽입/삭제 연산과 팬텀 현상

### 팬텀 현상

```sql
-- T30: 물리학과 교수 수 조회
SELECT count(*) FROM instructor WHERE dept_name = 'Physics';

-- T31: 새 물리학과 교수 삽입
INSERT INTO instructor VALUES (11111, 'Feynman', 'Physics', 94000);
```

T30과 T31은 공통 튜플이 없지만 논리적으로 충돌(팬텀 튜플). 튜플 수준 잠금만으로는 탐지 불가.

**인덱스 잠금 프로토콜(Index-Locking Protocol)**:
- 모든 릴레이션에 반드시 인덱스 존재
- 튜플 조회 시 접근한 **인덱스 리프 노드**에도 S-Lock
- 삽입/삭제/업데이트 시 영향받는 **인덱스 리프 노드**에 X-Lock
- 2PL 규칙 준수

## 18.5 타임스탬프 기반 프로토콜

### 타임스탬프 할당

트랜잭션 Ti 시작 시 고유 타임스탬프 TS(Ti) 부여. `TS(Ti) < TS(Tj)` → Ti가 Tj보다 먼저 시작.

각 데이터 항목 Q에:
- **W-timestamp(Q)**: 가장 최근에 Q를 write한 트랜잭션의 TS
- **R-timestamp(Q)**: 가장 최근에 Q를 read한 트랜잭션의 TS

### 타임스탬프 순서화 프로토콜

```
Ti가 read(Q) 요청 시:
  TS(Ti) < W-timestamp(Q) → 거부(Ti 롤백) : Ti는 이미 쓰인 이전 값을 읽으려 함
  TS(Ti) ≥ W-timestamp(Q) → 허용, R-timestamp(Q) = max(R-timestamp(Q), TS(Ti))

Ti가 write(Q) 요청 시:
  TS(Ti) < R-timestamp(Q) → 거부(Ti 롤백) : Ti의 쓰기가 이미 읽힌 뒤에 도착
  TS(Ti) < W-timestamp(Q) → 거부(Ti 롤백) : 더 새로운 값이 이미 기록됨
  그 외                  → 허용, W-timestamp(Q) = TS(Ti)
```

**특성**: 교착 상태 없음. 충돌 직렬가능성 보장. 긴 트랜잭션 기아 가능.

### Thomas의 쓰기 규칙 (Thomas' Write Rule)

`TS(Ti) < W-timestamp(Q)` 이고 `TS(Ti) ≥ R-timestamp(Q)` → 쓰기를 **무시**(롤백 없음).

이미 더 최신 값이 기록되었으므로 Ti의 쓰기는 어차피 필요 없음 → 충돌 직렬가능이 아닌 **뷰 직렬가능** 허용.

## 18.6 검증 기반 프로토콜 (낙관적 동시성 제어)

읽기 전용 트랜잭션이 많을 때 적합. 충돌을 가정하지 않고 실행 후 검증.

**3단계 실행**:
1. **읽기 단계**: 로컬 변수로 읽기. DB는 실제로 쓰지 않음.
2. **검증 단계**: 직렬가능성 위반 여부 확인.
3. **쓰기 단계**: 검증 통과 시 실제 DB에 쓰기. 읽기 전용 트랜잭션은 생략.

각 트랜잭션에 세 타임스탬프:
- StartTS(Ti), ValidationTS(Ti), FinishTS(Ti)

**검증 조건**: 모든 Tk (TS(Tk) < TS(Ti))에 대해 다음 중 하나 만족:
1. FinishTS(Tk) < StartTS(Ti) (순서 명확)
2. Tk의 쓰기 집합 ∩ Ti의 읽기 집합 = ∅, 이고 FinishTS(Tk) < ValidationTS(Ti)

## 18.7 다중 버전 동시성 제어 (MVCC)

데이터 항목의 여러 버전을 유지하여 읽기 연산이 절대 대기하지 않도록.

### 다중 버전 타임스탬프 순서화

각 데이터 항목 Q에 버전 시퀀스 `<Q1, Q2, ..., Qm>`.

각 버전 Qk에:
- Content: 실제 값
- W-timestamp(Qk): 이 버전을 만든 트랜잭션의 TS
- R-timestamp(Qk): 이 버전을 읽은 가장 최신 트랜잭션의 TS

**읽기**: Ti가 read(Q) → W-timestamp ≤ TS(Ti)인 최신 버전 반환 (항상 성공).
**쓰기**: TS(Ti) < R-timestamp(Qk) → Ti 롤백. 아니면 새 버전 생성.

### 다중 버전 2PL

- 읽기 전용 트랜잭션: 타임스탬프 기반 → 절대 대기 없음
- 업데이트 트랜잭션: 강성 2PL → 커밋 순서로 직렬화
- ts-counter를 커밋 시 증가 → 버전 타임스탬프로 활용

## 18.8 스냅샷 격리 (Snapshot Isolation)

### 작동 원리

```
트랜잭션 Ti 시작 시 DB 스냅샷 제공
  → StartTS(Ti) 이전에 커밋된 모든 변경 사항 포함
  → StartTS(Ti) 이후의 변경 사항은 보이지 않음

업데이트는 private workspace에서 수행
커밋 시 DB에 반영 (원자적으로)
```

### 업데이트 검증

**First Committer Wins**: 동시 트랜잭션이 같은 항목을 쓰면 먼저 커밋한 쪽만 허용.

**First Updater Wins**: 쓰기 잠금 기반. 선점한 쪽이 커밋 허용. 나중 쪽은 대기 후 확인.

### Write Skew 이상 현상

```
Ti: read(A), read(B), write(A)  (A = B 대입)
Tj: read(A), read(B), write(B)  (B = A 대입)

두 트랜잭션이 각자의 스냅샷에서 읽고 서로 다른 항목을 씀
→ 둘 다 커밋 허용 → A, B의 값이 바뀜 (비직렬화 가능!)
```

**이유**: 읽기-쓰기 충돌(Read-Write Conflict)이 추적되지 않음.

### 직렬화 가능 스냅샷 격리 (SSI, PostgreSQL 9.1+)

**원리**: 읽기-쓰기 충돌 엣지 추적. 트랜잭션이 들어오는(incoming) + 나가는(outgoing) 읽기-쓰기 충돌 엣지를 둘 다 가지면 롤백 → 비직렬화 가능 스케줄 방지.

```sql
-- 특정 쿼리에 쓰기 충돌로 처리 강제 (for update 절)
SELECT * FROM instructor WHERE ID = 22222 FOR UPDATE;
```

## 18.9 실제 약한 일관성 수준

### Degree-Two Consistency

S-Lock은 읽기 직후 해제 가능(비2PL). X-Lock은 커밋까지 보유. 직렬가능 미보장. **Read Committed** 격리 수준의 한 구현.

### 커서 안정성 (Cursor Stability)

현재 처리 중인 튜플에만 S-Lock, 처리 후 즉시 해제. 업데이트된 튜플은 X-Lock 유지. 고빈도 접근 릴레이션의 동시성 향상.

### 사용자 상호작용을 포함한 트랜잭션

긴 대화형 트랜잭션에서 전체를 단일 DB 트랜잭션으로 처리하면 잠금 시간이 너무 길어짐.

**버전 번호 기반 낙관적 동시성 제어**:
```
1. 튜플 읽기 시 버전 번호 기억
2. 로컬에서 업데이트 수행
3. 커밋 시 현재 버전 번호 = 읽을 때 버전 번호인지 확인
   → 일치하면 커밋 + 버전 번호 증가
   → 불일치하면 중단 (다른 트랜잭션이 중간에 변경)
```

## 18.10 고급 동시성 제어 주제

### 온라인 인덱스 생성

1. 릴레이션 스냅샷으로 인덱스 생성
2. 생성 중 발생한 업데이트를 로그로 추적
3. 공유 잠금으로 릴레이션 잠근 후 나머지 업데이트 적용
4. 인덱스 완성 후 잠금 해제

### B⁺-Tree 동시성 제어

**크래빙 프로토콜(Crabbing Protocol)**:
- 탐색: 루트→리프로 내려가면서 자식 S-Lock → 부모 S-Lock 해제
- 삽입/삭제: 탐색과 동일하다가 리프에서 X-Lock → 분할/합병 시 부모 X-Lock 상향
- 잠금을 짧은 시간만 보유 → **래치(Latch)**라 부름

**B-Link Tree**: 오른쪽 형제 포인터 추가 → 한 번에 하나의 내부 노드만 잠금 가능. 교착 상태 없음.

**다음 키 잠금(Next-Key Locking)**: 범위 조회와 삽입 충돌을 인덱스 리프에서 탐지. 팬텀 방지.

### 인메모리 DB 동시성 제어

데이터가 메모리에 있어 인덱스 연산 시간이 매우 짧음 → 코스 단위(전체 인덱스) 래치도 OK.

**래치-프리(Latch-free) 자료구조**: CAS(Compare-And-Swap) 명령어로 잠금 없이 구현.

```c
// CAS를 이용한 래치-프리 삽입
insert_latchfree(head, value) {
    node = new node(value)
    repeat:
        oldhead = head
        node.next = oldhead
        result = CAS(head, oldhead, node)  // 원자적: head == oldhead면 head = node
    until result == success
}
```

**ABA 문제**: 포인터가 A → B → A로 변경될 때 CAS가 잘못 성공할 수 있음. 포인터+카운터 쌍으로 CAS 적용하여 해결.

## 18.11 핵심 요약

```
동시성 제어 메커니즘 비교

메커니즘        교착상태   직렬가능성   읽기차단   특징
-----------   --------  ---------  -------  ----
2PL           발생       충돌 직렬   발생      가장 널리 사용
Strict 2PL    발생       충돌 직렬   발생      연쇄 롤백 없음
Tree Protocol 없음        충돌 직렬   발생      순서 필요
Timestamp     없음        충돌 직렬   없음      롤백 없음(로직상)
Validation    없음        충돌 직렬   없음      읽기가 많을 때
Snapshot(SI)  없음        뷰 직렬    없음      Write Skew 가능
SSI           없음        충돌 직렬   없음      SI + 추가 추적
```

| 개념 | 핵심 포인트 |
|------|------------|
| 2PL | 성장→수축 두 단계, 충돌 직렬가능 보장 |
| Strict 2PL | X-Lock을 커밋까지 → 연쇄 롤백 없음 |
| Wait-Die | 오래된 트랜잭션 대기, 젊은 것 롤백 |
| Wound-Wait | 오래된 트랜잭션이 젊은 것 선점 |
| 다중 단위 잠금 | IS/IX/SIX 의도 잠금, 루트→리프 획득 |
| 인덱스 잠금 | 팬텀 방지, 리프 노드에 잠금 |
| 타임스탬프 | W/R-timestamp 비교로 순서 강제 |
| Thomas 규칙 | 오래된 write 무시 → 뷰 직렬가능 |
| 낙관적 검증 | 읽기 단계→검증→쓰기, 충돌 없을 때 효율적 |
| 스냅샷 격리 | 읽기 블로킹 없음, Write Skew 주의 |
| SSI | 읽기-쓰기 충돌 추적 → 완전 직렬가능 |
| Crabbing | B⁺-Tree 래치, 내려가며 크래빙 |
| Next-Key Lock | 팬텀 방지, 다음 키 값도 잠금 |
