# Chapter 14: Indexing

## 개요

인덱스는 쿼리가 릴레이션 전체 스캔 없이 필요한 레코드를 빠르게 찾을 수 있게 해주는 보조 자료구조다. 순서 인덱스(밀집/희소, 클러스터링/비클러스터링), B⁺-Tree(구조·탐색·삽입·삭제), 해시 인덱스, 복합 검색 키, 쓰기 최적화 구조(LSM Tree, Buffer Tree), 비트맵 인덱스, 공간·시간 인덱스를 다룬다.

---

## 14.1 기본 개념

- **검색 키(Search Key)**: 레코드 조회 기준 속성(집합). Primary Key와 무관하며, 여러 검색 키 가능.
- **인덱스 항목(Index Entry)**: `(search-key value, record pointer(s))`
- 인덱스 평가 기준: 접근 타입, 접근 시간, 삽입 시간, 삭제 시간, 공간 오버헤드

---

## 14.2 순서 인덱스 (Ordered Indices)

### 밀집 인덱스 vs 희소 인덱스

| 구분 | 밀집(Dense) | 희소(Sparse) |
|------|------------|--------------|
| 항목 수 | **모든** 검색 키 값 | **일부** 검색 키 값만 |
| 사용 조건 | 클러스터링/비클러스터링 모두 가능 | **클러스터링 인덱스만** 가능 |
| 공간 | 더 큼 | 더 작음 |
| 접근 속도 | 빠름 (직접 조회) | 비교적 느림 (인근 항목에서 순차 스캔) |

```
밀집 인덱스 (Dense Index):
10101 → [10101, Srinivasan, ...]
12121 → [12121, Wu, ...]
15151 → [15151, Mozart, ...]
...

희소 인덱스 (Sparse Index, 블록당 1 항목):
10101 → [블록 0]
32343 → [블록 1]
76766 → [블록 2]

희소 인덱스 탐색: "22222" 검색
  → 22222 < 32343 이므로 10101 항목의 블록 0부터 순차 스캔
```

**실무 권장**: 블록당 하나의 항목을 가진 희소 인덱스 → 공간 최소화 + I/O 최소화.

### 클러스터링 vs 비클러스터링

- **클러스터링(Clustering) = Primary Index**: 인덱스 검색 키 순서 = 파일 물리적 저장 순서
- **비클러스터링(Non-clustering) = Secondary Index**: 검색 키 순서 ≠ 저장 순서

비클러스터링 인덱스는 반드시 **밀집**이어야 하며, 비유일 검색 키의 경우 버킷(bucket)을 통한 간접 포인터 사용.

```
비클러스터링 인덱스 (dept_name):
Biology   → [버킷: 76766]
Comp.Sci. → [버킷: 10101, 45565, 83821]
Finance   → [버킷: 12121, 76543]
...
```

### 다단계 인덱스 (Multilevel Index)

인덱스 자체가 너무 크면, 인덱스에 대한 희소 인덱스 생성.

```
100만 레코드, 블록당 100 인덱스 항목 → 내부 인덱스: 10,000 블록
이진 탐색: ⌈log₂(10,000)⌉ = 14 블록 접근

2단계 인덱스:
  외부 인덱스: 10,000/100 = 100 블록 (메모리 상주 가능)
  내부 인덱스 1개 블록만 추가 읽기
  → 총 1~2 블록 접근!
```

## 14.3 B⁺-Tree 인덱스

### 14.3.1 B⁺-Tree 구조

**균형 트리**: 루트에서 모든 리프까지의 경로 길이 동일.

```
비리프 노드 (Internal Node):
┌────┬────┬────┬────┬────┬────┬────┐
│ P1 │ K1 │ P2 │ K2 │ P3 │ K3 │ P4 │
└────┴────┴────┴────┴────┴────┴────┘
n개의 포인터, n-1개의 검색 키 값
├── P1: K1 미만의 레코드가 있는 서브트리
├── P2: K1 이상 K2 미만
└── P4: K3 이상

리프 노드 (Leaf Node):
┌────┬─────┬────┬─────┬────┬─────┬──────────────┐
│ P1 │ K1  │ P2 │ K2  │ P3 │ K3  │ Pn(다음 리프) │
└────┴─────┴────┴─────┴────┴─────┴──────────────┘
Pi: 검색 키 Ki를 가진 레코드 포인터
Pn: 다음 리프 노드 포인터 (범위 쿼리용 체인)
```

**차수 n 규칙**:
- 리프 노드: ⌈(n-1)/2⌉ ~ n-1 개의 값
- 비리프 노드(루트 제외): ⌈n/2⌉ ~ n 개의 포인터
- 루트: 2 ~ n 개의 포인터 (리프이기도 하면 0 ~ n-1 값)

**B⁺-Tree 예시 (n=4)**:
```
              [Mozart]
        /                \
   [Caliﬁeri|Einstein]   [Srinivasan]
  /    |       |     \        |      \
[Br|Ca] [Cr|El] [Go|Ka|Ki] [Mo|Si|Sr] [Wu]

리프 체인: Brandt → Caliﬁeri → Crick → El Said → Gold → ... → Wu
```

### 14.3.2 B⁺-Tree 탐색

```python
def find(v):
    C = root
    while C is not leaf:
        i = smallest i such that v <= C.K[i]
        if not found:   C = C.P[last]    # v > 최대 키
        elif v == C.K[i]: C = C.P[i+1]
        else:             C = C.P[i]      # v < C.K[i]
    # C는 리프 노드
    if K[i] == v: return P[i]
    else: return null
```

**복잡도**: 루트에서 리프까지 ⌈log₍n/2₎(N)⌉ 노드 접근. n=100, N=1,000,000 → ⌈log₅₀(1,000,000)⌉ = 4 블록 읽기.

**범위 쿼리**: `findRange(lb, ub)` — lb에 해당하는 리프까지 탐색 후 Pn 체인을 따라 ub까지 순차 스캔.

### 14.3.3 B⁺-Tree 삽입

```
삽입 흐름:
1. 리프 노드 L 탐색
2. L에 공간 있으면 삽입 (완료)
3. L이 가득 찼으면 분할(split):
   - 새 리프 L' 생성
   - n개 값을 앞 ⌈n/2⌉, 뒤 ⌊n/2⌋ 로 분배
   - L' 의 최소 키 값 K'를 부모에 삽입 → insert_in_parent(L, K', L')
4. 부모도 가득 찼으면 재귀적으로 비리프 노드도 분할
5. 루트까지 분할되면 새 루트 생성 (트리 높이 1 증가)
```

**비리프 노드 분할 차이**:
- 리프 분할: 경계 키 K'를 양쪽 모두에 유지
- 비리프 분할: 중간 키 K''를 **부모로만 올리고** 어느 자식에도 두지 않음

```
삽입 "Adams" 예시 (n=4):
[Brandt|Caliﬁeri|Crick] → 가득참
분할 → [Adams|Brandt] + [Caliﬁeri|Crick]
부모에 "Caliﬁeri" 삽입
```

### 14.3.4 B⁺-Tree 삭제

```
삭제 흐름:
1. 리프 노드에서 항목 제거
2. 노드가 충분히 차있으면 완료 (리프: ≥ ⌈(n-1)/2⌉, 비리프: ≥ ⌈n/2⌉)
3. 너무 적으면:
   a. 형제 노드와 합병(coalesce): 합산이 n 이하면 합병 → 부모에서 항목 삭제 → 재귀
   b. 형제 노드와 재분배(redistribute): 형제가 충분히 차있으면 빌림
      - 부모의 구분 키가 새 경계 키로 교체됨
4. 루트가 자식 하나만 남으면 루트 삭제 → 자식이 새 루트 (트리 높이 1 감소)
```

**삭제 후 비리프의 키 유지**: 삭제된 키가 비리프에 남아있을 수 있음 (라우팅 목적).

### 14.3.5 비유일 검색 키

실용적 해결책: 검색 키 ai에 기본키 Ap를 추가한 **복합 유일 키** `(ai, Ap)` 사용.

```sql
-- name으로 인덱스 생성 시 내부적으로:
CREATE INDEX ON instructor ((name, ID));  -- ID는 uniquifier

-- name = 'Gold' 검색:
findRange(('Gold', -∞), ('Gold', +∞))
```

## 14.4 B⁺-Tree 확장

### 14.4.1 B⁺-Tree 파일 조직

리프 노드에 레코드 포인터 대신 **실제 레코드** 저장.

```
B⁺-Tree File Organization:
              [I]
           /       \
        [C|F]      [M]
       /  |  \    /   \
 [A,4] [B,8] [C,1|D,9] [I,4|J,8] [M,4|N,8]
(키, 레코드값)
```

- 인덱스-순차 파일 조직의 성능 저하 문제 해결
- B⁺-Tree로 레코드 자체를 조직 → 삽입/삭제 시 자동 재균형
- 형제 노드 참여 재분배: 각 노드가 ≥ ⌊2n/3⌋ 채워지도록 유지 (3-way 분할)

### 14.4.2 보조 인덱스와 레코드 재배치

B⁺-Tree 파일 조직에서 리프 노드 분할 시 레코드가 이동 → 보조 인덱스의 포인터가 무효화됨.

**해결책**: 보조 인덱스의 포인터 대신 **기본 인덱스 검색 키 값** 저장.

```
보조 인덱스 (dept_name):
  Comp.Sci. → [ID: 10101, 45565, 83821]  (레코드 포인터 대신 기본키 저장)

검색 과정: 보조 인덱스 → 기본키 목록 → 기본 인덱스로 실제 레코드 조회 (2단계)
```

### 14.4.3 접두사 압축 (Prefix Compression)

비리프 노드에 전체 문자열 대신 구분에 충분한 접두사만 저장 → fanout 증가.

```
"Silberschatz"와 "Silver"를 구분하는 비리프 키: "Silbert" 또는 "Silbi"
```

### 14.4.4 벌크 로딩 (Bulk Loading)

대량 레코드를 한 번에 인덱스에 삽입하는 효율적 방법.

```
일반 삽입 방식 문제:
  - 각 삽입마다 랜덤 리프 접근 → 디스크: 각 I/O 10ms
  - 1억 레코드 → 최소 1,000,000초!

벌크 로딩 방식:
  1. 임시 파일에 (search_key, record_ptr) 쌍 생성
  2. 검색 키 순으로 정렬 (효율적 외부 정렬 사용)
  3. 정렬된 항목으로 B⁺-Tree 바텀업(bottom-up) 구성:
     - 리프 노드를 연속 블록에 채워 생성
     - 상위 레벨은 각 노드의 최소 키로 구성
     - 이전보다 ~1000배 빠름!
```

### 14.4.5 B-Tree vs B⁺-Tree

| 특성 | B⁺-Tree | B-Tree |
|------|---------|--------|
| 비리프의 검색 키 | 중복 저장 (리프에도) | 한 번만 저장 |
| 팬아웃 | 높음 | 낮음 (추가 데이터 포인터) |
| 삭제 복잡도 | 낮음 | 높음 (비리프에서도 삭제 가능) |
| 범위 쿼리 | 리프 체인 → 효율적 | 트리 내 랜덤 접근 |
| 실제 사용 | **거의 모든 DB** | 거의 미사용 |

> 실무에서 "B-Tree 인덱스"라고 부르는 것은 대부분 **B⁺-Tree**를 의미한다.

### 14.4.6 플래시 저장과 인덱싱

SSD 특성:
- 랜덤 읽기: 20~100μs (HDD의 ~100배 빠름)
- 쓰기: in-place 불가, 소거(2~5ms) 필요
- 최적 노드 크기: 플래시 페이지 크기에 맞게 소형화

**벌크 로딩의 SSD 장점**: 페이지 쓰기 횟수 감소 → 소거 연산 감소 → 수명 연장.

**쓰기 최적화** (14.8절 참고): LSM Tree, Buffer Tree 등.

## 14.5 해시 인덱스

### 인메모리 해시 인덱스

```
h(search_key) → bucket_address

버킷 = 링크드 리스트 (overflow chaining = closed addressing)

삽입: h(Ki) 계산 → 버킷 i의 리스트에 추가
검색: h(Ki) 계산 → 버킷 i의 리스트에서 Ki 값 비교
삭제: h(Ki) 계산 → 버킷 i에서 항목 제거
```

**해시 vs B⁺-Tree**:

| 연산 | 해시 인덱스 | B⁺-Tree |
|------|------------|---------|
| 동등 조회 | O(1) 평균 | O(log N) |
| 범위 쿼리 | **지원 안 함** | 지원 |
| 정렬 스캔 | **지원 안 함** | 지원 |

### 디스크 기반 해시 인덱스

```
버킷 오버플로우: h(Ki) 버킷이 가득 찼을 때 → 오버플로우 버킷 체인
오버플로우 체인: bucket_0 → overflow_0_1 → overflow_0_2 → null

버킷 수 = (nr / fr) × (1 + d)   (d ≈ 0.2 여유 공간)
```

**정적 해싱(Static Hashing)**: 버킷 수 고정. 레코드 수 급증 시 재구축 필요.
**동적 해싱**: 버킷 수를 점진적으로 늘림. 선형 해싱(Linear Hashing), 확장 가능 해싱(Extendable Hashing) 등.

## 14.6 복합 키 접근

### 복합 검색 키 인덱스

```sql
-- (dept_name, salary) 복합 인덱스
-- 효율적인 쿼리:
WHERE dept_name = 'Finance' AND salary = 80000     -- 동등 조건
WHERE dept_name = 'Finance' AND salary < 80000     -- 범위 조건 (두 번째만)
WHERE dept_name = 'Finance'                        -- 첫 번째 속성만

-- 비효율적:
WHERE dept_name < 'Finance' AND salary < 80000     -- 두 속성 모두 비교
```

복합 인덱스의 검색 키 순서는 **사전식(Lexicographic)** 정렬.

### 커버링 인덱스 (Covering Index)

검색 키 외에 추가 속성 값을 인덱스에 저장 → 인덱스만으로 쿼리 응답 가능.

```sql
-- ID에 대한 비클러스터링 인덱스에 salary 추가
-- SELECT salary FROM instructor WHERE ID = '10101'
-- → 실제 레코드 접근 없이 인덱스만으로 응답!
```

## 14.7 인덱스 생성 및 사용

```sql
-- 인덱스 생성
CREATE INDEX dept_index ON instructor (dept_name);

-- 유일 인덱스 (후보 키)
CREATE UNIQUE INDEX dept_index ON instructor (dept_name);

-- 삭제
DROP INDEX dept_index;
```

**자동 인덱스**: 대부분의 DB는 기본 키에 자동으로 인덱스 생성. 외래 키도 인덱스 권장.

**인덱스 비용**: 조회 성능↑, 업데이트 시 유지 비용↑. 과도한 인덱스는 DML 성능 저하.

## 14.8 쓰기 최적화 인덱스 구조

### 14.8.1 LSM Tree (Log-Structured Merge Tree)

랜덤 쓰기가 많은 워크로드에서 B⁺-Tree보다 훨씬 효율적.

```
구조:
┌────────────────────────────────────────────────────────────┐
│   L0 (인메모리 B⁺-Tree)  ← 모든 새 삽입 여기에            │
│   가득 차면 L1과 합병                                       │
├────────────────────────────────────────────────────────────┤
│   L1 (디스크 B⁺-Tree, 최대 크기 S)                        │
│   가득 차면 L2와 합병                                       │
├────────────────────────────────────────────────────────────┤
│   L2 (디스크 B⁺-Tree, 최대 크기 S×k)                      │
├────────────────────────────────────────────────────────────┤
│   L3 (최대 크기 S×k²)                                      │
└────────────────────────────────────────────────────────────┘
```

**합병(Merge) 과정**:
1. L0 리프를 검색 키 순으로 스캔
2. L1 리프와 병합 정렬 방식으로 합침
3. 새 L1을 바텀업 방식으로 구성 → **순차 I/O만 사용!**

**삭제/업데이트**: 삭제 항목(deletion entry) 삽입. 합병 시 원본 항목과 매칭 후 둘 다 폐기.

**단계적 합병(Stepped-Merge)**: 각 레벨에 최대 b개 트리 허용 → b개가 되면 상위 레벨로 합병. 쓰기 비용↓, 읽기 비용↑ (Bloom Filter로 보상).

**사용 시스템**: Google BigTable, HBase, Cassandra, MongoDB, LevelDB, SQLite4, MySQL(MyRocks).

### 14.8.2 버퍼 트리 (Buffer Tree)

B⁺-Tree의 각 내부 노드에 버퍼 추가.

```
내부 노드:
┌────────────────────────────────────┐
│ P1 K1 P2 K2 P3 K3 P4 K4 P5 K5 P6  │ ← 트리 포인터/키
├────────────────────────────────────┤
│          [버퍼 영역]               │ ← 미처리 삽입 항목
└────────────────────────────────────┘

삽입: 루트 버퍼에 기록 → 버퍼 가득 차면 자식으로 한꺼번에 밀어냄
     → 여러 항목이 한 노드에 일괄 처리 (I/O 상각)
```

**LSM vs Buffer Tree**:

| 특성 | LSM Tree | Buffer Tree |
|------|----------|-------------|
| 쓰기 I/O | 순차 (낮은 탐색 비용) | 랜덤 |
| 읽기 비용 | 높음 (여러 트리 탐색) | 낮음 |
| HDD 환경 | LSM 우수 | 불리 |
| SSD 환경 | 비슷 | Buffer Tree 우수 가능 |

## 14.9 비트맵 인덱스 (Bitmap Index)

값의 종류가 적은 속성(카디날리티가 낮음)에 특화된 인덱스.

```
instructor_info (ID, gender, income_level):
  gender: {m, f}
  income_level: {L1, L2, L3, L4, L5}

레코드 번호:    0     1     2     3     4
gender:       m     f     f     m     f
income_level: L1    L2    L3    L4    L1

비트맵 (gender):
  m: 10010
  f: 01101

비트맵 (income_level):
  L1: 10001
  L2: 01000
  L3: 00100
  L4: 00010
  L5: 00000
```

**다중 조건 교집합**: 비트 AND 연산만으로 처리 (CPU 한 사이클에 64비트 처리).

```
gender='f' AND income_level='L2':
  f 비트맵:  01101
  L2 비트맵: 01000
  AND 결과:  01000  → 레코드 1만 해당
```

**장점**: 여러 조건의 조합을 매우 빠르게 처리. OLAP/데이터 웨어하우스에 적합.

## 14.10 공간·시간 인덱스

### 14.10.1 공간 데이터 인덱싱

B⁺-Tree는 1차원 정렬에 기반 → 2D 이상 공간 쿼리에 비효율.

**k-d Tree**: 각 레벨에서 번갈아 가며 한 차원씩 분할.

```
2차원 k-d Tree:
  레벨 1: x 축으로 분할 (x < x0은 왼쪽, x >= x0은 오른쪽)
  레벨 2: y 축으로 분할
  레벨 3: x 축으로 분할
  ...

범위 탐색: 각 노드에서 쿼리 범위와 분할 경계 비교 → 관련 서브트리만 탐색
```

**R-Tree**: 영역(선분, 다각형 등)을 인덱싱. 각 노드가 **최소 경계 박스(Bounding Box)**로 표현.

```
R-Tree 구조:
  리프 노드: 실제 객체 저장
  비리프 노드: 자식 노드들의 경계 박스 + 포인터

범위 쿼리: 쿼리 박스와 현재 노드의 경계 박스가 겹치는 경우만 탐색 (가지 치기)
```

### 14.10.2 시간 데이터 인덱싱

시간 구간 `[start_time, end_time]`이 있는 데이터.

```
시간 인덱스 전략:
  현재 튜플 (end_time = ∞): B⁺-Tree(attr, start_time) 로 별도 저장
  과거 튜플: R-Tree (2차원: attr 값 × 시간 축)

쿼리: attr=v, 시점=t1
  → 현재 인덱스: attr=v, start_time ≤ t1 범위 탐색
  → 과거 인덱스: R-Tree 범위 쿼리 (v × t1 포함 구간)
```

## 14.11 핵심 요약

```
인덱스 선택 가이드

쿼리 타입?
├── 동등 조회만 → 해시 인덱스
├── 범위 조회 또는 정렬 → B⁺-Tree
├── 저 카디날리티 + 복합 조건 → 비트맵
├── 공간 데이터 → R-Tree / k-d Tree
└── 시간 구간 → R-Tree + B⁺-Tree 조합

쓰기 패턴?
├── 랜덤 쓰기 집약 → LSM Tree (HDD) / Buffer Tree (SSD)
└── 대량 일괄 삽입 → 벌크 로딩 (정렬 후 바텀업 구성)
```

| 개념 | 핵심 포인트 |
|------|------------|
| 밀집/희소 인덱스 | 밀집=모든 키, 희소=일부 키 (클러스터링만 가능) |
| 클러스터링 인덱스 | 파일 정렬 순서와 일치, 하나의 릴레이션에 하나만 |
| B⁺-Tree 높이 | ⌈log₍n/2₎(N)⌉, n=100이면 4단계로 100만 레코드 탐색 |
| 삽입 분할 | 리프→비리프 재귀적 분할, 루트 분할 시 높이 증가 |
| 삭제 합병/재분배 | 노드가 반 이하면 형제와 합병 or 재분배 |
| 비유일 검색 키 | 복합 유일 키 (ai, 기본키) 사용 권장 |
| B⁺-Tree 파일 조직 | 레코드 자체를 리프에 저장, 자동 재균형 |
| 벌크 로딩 | 정렬 후 바텀업 구성 → 수백 배 속도 향상 |
| LSM Tree | 인메모리 L0 + 디스크 L1..Lk, 순차 I/O로 합병 |
| 비트맵 인덱스 | 비트 AND/OR로 다중 조건 초고속 처리 |
| R-Tree | 경계 박스 기반 공간 인덱스, 범위/최근접 쿼리 |
