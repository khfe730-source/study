# Chapter 20: Database-System Architectures

## 개요

데이터베이스 시스템 아키텍처는 하드웨어, 네트워킹, 병렬·분산 처리 요구사항에 의해 결정된다. 중앙집중식 시스템에서 시작하여 서버 아키텍처, 병렬 시스템(공유 메모리/디스크/무공유/계층형), 분산 시스템, 클라우드 서비스 모델을 다룬다.

---

## 20.1 개요

```
진화 경로:
단일 머신 DBMS (1960s~)
    ↓ 멀티 코어 + 대용량 메모리
공유 메모리 병렬 DB (1990s~)
    ↓ 웹 스케일 데이터 폭증
공유 없음(Shared-Nothing) 병렬 DB / 분산 DB (2000s~)
    ↓ 인프라 관리 외주화
클라우드 기반 DB 서비스 (2010s~)
```

---

## 20.2 중앙집중식 데이터베이스 시스템

| 유형 | 특징 |
|------|------|
| 단일 사용자 시스템 | 스마트폰, PC. 간단한 동시성, 경량 복구 |
| 멀티 사용자 시스템 | 서버. 완전한 ACID 트랜잭션 지원 |
| 임베디드 DB | 단일 앱에 연결, SQL 미지원 가능 (예: SQLite) |

**조립식 병렬성**: 소수 강력한 프로세서, 공유 메모리 → 거친 입자 병렬성(Coarse-grained parallelism).
**세밀 입자 병렬성(Fine-grained)**: 수천 개 소형 프로세서 사용.

## 20.3 서버 시스템 아키텍처

### 트랜잭션 서버 (Transaction Server)

- 클라이언트 → SQL 전송 → 서버 실행 → 결과 반환
- 다수의 프로세스/스레드가 **공유 메모리**에 접근

```
프로세스 구성:
  서버 프로세스     → SQL 쿼리 처리
  잠금 관리자      → 잠금 허가·해제·교착 탐지
  DB 쓰기 프로세스  → 변경 버퍼 블록 → 디스크
  로그 쓰기 프로세스 → 로그 버퍼 → 안정 저장
  체크포인트 프로세스 → 주기적 체크포인트
  프로세스 모니터   → 프로세스 장애 감지 및 복구

공유 메모리:
  버퍼 풀, 잠금 테이블, 로그 버퍼, 쿼리 계획 캐시
```

**원자적 명령어**:
- `test-and-set(M)`: M을 읽고 1로 설정 (원자적) → 배타 잠금 구현
- `compare-and-swap(M, Vo, Vn)`: M == Vo이면 M = Vn (원자적) → 래치 구현

공유 메모리 내 잠금 테이블 직접 갱신 → 메시지 오버헤드 없이 빠른 잠금 처리.

### 데이터 서버 (Data Server)

- 클라이언트가 데이터 항목(파일, 페이지, 객체)을 직접 요청
- CAD, 병렬 스토리지 시스템에 활용

**클라이언트 최적화 전략**:
- **프리패칭(Prefetching)**: 곧 필요할 항목을 미리 가져옴
- **데이터 캐싱**: 클라이언트에 데이터 캐시. 유효성 검사 필요.
- **잠금 캐싱**: 잠금을 클라이언트에 캐시 → 서버 통신 없이 접근. 서버가 콜백(callback)으로 잠금 회수 가능.
- **적응형 잠금 단위**: 멀티 데이터 항목 요청 시 페이지 잠금(더 큰 단위)으로 우선 처리, 충돌 시 세밀 잠금으로 에스컬레이션 해제.

## 20.4 병렬 시스템

### 20.4.2 성능 지표

**스피드업(Speedup)**: 동일 작업을 더 빠르게 처리.
```
Speedup = TS / TL
선형 스피드업: N배 자원 → N배 빠름
아mdahl의 법칙: 직렬 부분 p가 있으면 최대 스피드업 = 1/(1-p)
```

**스케일업(Scaleup)**: 더 큰 작업을 동일 시간에 처리.
```
Scaleup = TS / TL (N배 자원, N배 큰 작업)
배치 스케일업: DB 크기 증가에 비례
트랜잭션 스케일업: 트랜잭션 수 증가에 비례 (병렬화에 더 유리)
```

**스피드업/스케일업 저해 요소**:
1. 순차 처리(Sequential computation) → Amdahl 법칙
2. 시작 비용(Start-up costs) → 수천 프로세스 초기화 오버헤드
3. 간섭(Interference) → 공유 자원 경합
4. 편향(Skew) → 가장 느린 서브태스크가 전체 시간 결정

### 20.4.3 상호연결 네트워크

| 토폴로지 | 특징 | 현재 사용 |
|---------|------|---------|
| 버스(Bus) | 단순, 병목 발생 | 프로세서 내부 |
| 링(Ring) | 각 링크 독립 전송 | 드물게 사용 |
| 메시(Mesh) | 격자형, 인접 연결 | 프로세서 내 코어간 |
| 하이퍼큐브 | log(n) 홉 | 과거 병렬 컴퓨터 |
| **트리형(Tree-like/Fat-tree)** | 에지→집계→코어 스위치 | **데이터센터 표준** |

**데이터센터 트리형 토폴로지**:
```
서버들 ─── 탑오브랙(ToR) 스위치 ─── 집계 스위치 ─── 코어 스위치
  (서버 48개/랙)           (다수 ToR 연결)    (인터넷 연결)

장점: 멀티 경로로 대역폭 증가, 스위치 장애 시 우회 경로 존재
핵심 기술: Ethernet, InfiniBand(저지연), RDMA(직접 메모리 접근)
```

### 20.4.4 병렬 데이터베이스 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│   P1  P2  P3 ... Pn      P = 프로세서                   │
│   └──────────────┘        M = 메모리                    │
│         M                 Disk = 디스크                 │
│  (a) 공유 메모리: 모든 P가 M 공유                       │
├─────────────────────────────────────────────────────────┤
│  [P1,M1] [P2,M2] ...     공유 디스크에 연결             │
│    └────── Disk ──────┘                                 │
│  (b) 공유 디스크: 각 P는 자체 메모리, Disk는 공유       │
├─────────────────────────────────────────────────────────┤
│  [P1,M1,D1] [P2,M2,D2] ... 공유 없음                   │
│  (c) 공유 없음: 각 노드가 독립적 (P,M,D 모두 분리)     │
├─────────────────────────────────────────────────────────┤
│  [공유 메모리 노드] 여러 개를 공유 없음으로 연결         │
│  (d) 계층형: 현대 병렬 DB의 표준                        │
└─────────────────────────────────────────────────────────┘
```

| 아키텍처 | 확장성 | 통신 비용 | 장애 허용 | 용도 |
|---------|-------|---------|---------|------|
| 공유 메모리 | 제한적 (수백 코어) | 매우 낮음 | 낮음 | 단일 서버 고성능 |
| 공유 디스크 | 중간 | 중간 | 높음 (RAID) | 고가용성 클러스터 |
| 공유 없음 | 높음 (수천 노드) | 높음 | 중간 | 웹 스케일 데이터 |
| 계층형 | 매우 높음 | 혼합 | 높음 | **현대 병렬 DB** |

### 20.4.5 공유 메모리 상세

**NUMA (Non-Uniform Memory Architecture)**:
- 각 프로세서에 로컬 메모리 연결 → 로컬 접근 빠름
- 다른 프로세서의 메모리 접근 시 인터커넥트 경유 → 느림

**캐시 일관성(Cache Coherency)**:
```
코어1이 메모리 위치 A를 수정 → 코어2의 캐시에 구 값 존재 → 불일치!

해결: MESI 프로토콜 (Modified/Exclusive/Shared/Invalid)
  쓰기 요청 → 다른 캐시의 해당 캐시 라인 무효화(Invalidate)
  메모리 장벽(Memory Barrier):
    sfence: 모든 쓰기가 캐시 밖으로 나갈 때까지 대기
    lfence: 모든 무효화 메시지 처리 후 읽기 진행
    mfence: 양쪽 모두
```

## 20.5 분산 시스템

```
병렬 DB (단일 데이터센터) vs 분산 DB (지리적 분산)
  │  낮은 지연, 높은 대역폭    │  높은 지연, 낮은 대역폭
  │  드문 네트워크 파티션       │  빈번한 링크 장애 가능성
  │  중앙 관리                  │  부분 자율성 가능
  │  동일 용량 노드              │  다양한 규모/기능 노드
```

**분산 DB 장점**:
- **데이터 공유**: 다른 사이트의 데이터에 접근 가능
- **자율성**: 각 사이트가 로컬 데이터에 대한 통제권 유지

**유형**:
- **동종 분산 DB**: 공통 글로벌 스키마, 동일 소프트웨어, 적극적 협력
- **연합 DB / 이종 분산 DB**: 기존 DB들의 통합, 각 사이트 독립적 운영

## 20.6 트랜잭션 처리 (분산)

**2단계 커밋(2PC)**: 분산 원자성 보장의 핵심 프로토콜.
- Phase 1: 코디네이터 → prepare 메시지 → 각 노드 ready/abort 응답
- Phase 2: 모두 ready면 commit, 하나라도 abort면 abort 결정 전파

**분산 동시성 제어**: 각 노드에서 로컬 잠금 + 글로벌 교착 상태 탐지.

## 20.7 클라우드 기반 서비스

### 서비스 모델 계층

```
사용자 ─────────────────────────────────────
                  SaaS (소프트웨어-서비스)
              PaaS (플랫폼-서비스)
          IaaS (인프라-서비스)
물리 하드웨어 (클라우드 공급자 관리)
```

| 모델 | 클라이언트 책임 | 공급자 제공 | 예시 |
|------|--------------|-----------|------|
| **IaaS** | OS, DB, 앱 설치 | 가상 머신/컨테이너 | AWS EC2 |
| **PaaS** | 앱 설치/유지 | 데이터 저장소, DB, 앱서버 | AWS RDS, Google Cloud SQL |
| **SaaS** | 사용만 | 앱 포함 모든 것 | Salesforce |

**주요 개념**:
- **탄력성(Elasticity)**: 수요에 따라 자원 즉시 확장/축소
- **가상 머신(VM)**: 하나의 물리 머신에서 여러 독립 OS 실행 (하이퍼바이저 관리)
- **컨테이너(Container)**: OS 커널 공유, VM보다 가벼움. Docker, Kubernetes
- **마이크로서비스 아키텍처**: 앱을 독립적인 소형 서비스들의 집합으로 구성

**클라우드 사용의 법적·보안 고려사항**: 데이터 거주 국가의 법률 적용, 보안 침해 시 책임 등.

## 20.8 핵심 요약

| 개념 | 핵심 포인트 |
|------|------------|
| 트랜잭션 서버 | 공유 메모리 + 다중 프로세스, CAS/test-and-set으로 상호 배제 |
| 스피드업/스케일업 | Amdahl 법칙: 직렬 부분이 병렬화 한계 결정 |
| 트리형 토폴로지 | 현대 데이터센터 표준, 멀티 경로로 대역폭 증가 |
| 공유 없음 | 가장 높은 확장성 (수천 노드), 현대 병렬 DB의 기본 |
| 계층형 | 공유메모리 노드 + 공유없음 상위 → 현대 병렬 DB |
| NUMA | 로컬/원격 메모리 속도 차이, 캐시 일관성 관리 필요 |
| SAN | 고속 로컬 네트워크로 다수 스토리지 연결 |
| 클라우드 IaaS/PaaS/SaaS | 책임 범위와 제공 범위의 트레이드오프 |
