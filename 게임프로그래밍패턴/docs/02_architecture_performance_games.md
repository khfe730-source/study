# Chapter 1: 아키텍처, 성능, 그리고 게임 (Architecture, Performance, and Games)

> **Game Programming Patterns** — Robert Nystrom 저  
> Part I: Introduction / Chapter 1

---

## 개요

이 챕터는 패턴으로 바로 들어가기 전에, 저자가 **소프트웨어 아키텍처**와 **게임**에 대해 어떻게 생각하는지를 정리한 글이다. 아키텍처, 성능, 개발 속도 사이의 트레이드오프(trade-off)를 이해하는 것이 이 책 전체를 읽는 데 핵심적인 맥락이 된다.

---

## 소프트웨어 아키텍처란 무엇인가? (What is Software Architecture?)

이 책은 그래픽의 선형대수, 물리의 미적분, AI의 알파-베타 가지치기를 가르치지 않는다.  
**코드와 코드 사이의 공간**을 다룬다 — 코드를 작성하는 것이 아니라 **구성하는 것**에 관한 책이다.

모든 프로그램은 어떤 형태로든 구조를 가진다. 설령 `main()` 하나에 모든 것을 밀어 넣은 구조일지라도. 그렇다면 **좋은 구조란 무엇인가?**

### 좋은 아키텍처란?

> 변경을 가했을 때, 마치 그 변경을 미리 예상하고 만들어진 것처럼 느껴지는 코드.

핵심은 **아키텍처는 변경(change)에 관한 것**이라는 점이다.
- 아무도 코드를 바꾸지 않는다면, 아키텍처는 무의미하다.
- 아키텍처의 척도는 **얼마나 쉽게 변경을 수용하는가**이다.

---

## 변경은 어떻게 이루어지는가? (How do you make a change?)

코드를 수정하기 위한 과정은 다음과 같다:

```
[코드 이해] → [해결책 고민] → [코드 작성] → [정리(cleanup)] → [반복]
```

이 흐름에서 **가장 시간이 많이 걸리는 단계는 "코드 이해"** 다.  
코드를 뇌에 로드하는 과정은 느리고 비용이 크다.

> 디스크에서 RAM으로 데이터를 올리는 것도 느리지만,  
> 그것을 인간의 뇌에 올리는 것은 훨씬 더 느리다.

---

## 디커플링이 어떻게 도움이 되는가? (How can decoupling help?)

**소프트웨어 아키텍처의 핵심 목표:**

> 코드를 변경하기 위해 머릿속에 넣어야 하는 **지식의 양을 최소화**하는 것.

두 코드가 **커플링(coupling)** 되어 있다는 것은, 하나를 이해하려면 다른 하나도 이해해야 한다는 뜻이다.  
**디커플링(decoupling)** 은 각 부분을 독립적으로 이해하고 변경할 수 있게 만든다.

- **이해 측면**: 관련 없는 코드를 머리에 올릴 필요가 없어진다.
- **변경 측면**: 한 부분의 변경이 다른 부분으로 파급되지 않는다.

---

## 비용은 얼마인가? (At What Cost?)

디커플링과 좋은 아키텍처는 **공짜가 아니다**.

- 코드를 작성할 때마다 전체 구조와 조화롭게 통합하는 **추가적인 노력**이 필요하다.
- **추상화(abstraction)** 와 **확장 포인트**를 어디에 도입할지 신중히 판단해야 한다.

### 과도한 아키텍처의 함정

미래를 예측하며 모든 곳에 유연성을 추가하다 보면:

- 인터페이스와 추상화가 도처에 넘쳐난다.
- 플러그인 시스템, 추상 기반 클래스, 가상 메서드의 홍수.
- 실제로 동작하는 코드를 찾으려면 수십 겹의 스캐폴딩을 헤쳐야 한다.
- 이론적으로는 이해할 코드가 줄어야 하지만, **추상화 레이어 자체가 정신적 공간을 잠식**한다.

> **"YAGNI" (You Aren't Gonna Need It)**  
> 나중에 필요할 것이라는 추측으로 코드를 복잡하게 만들지 마라.

---

## 성능과 속도 (Performance and Speed)

아키텍처 비판자들은 자주 이렇게 말한다: **"추상화는 성능을 해친다"**.

### 유연성 vs 성능의 스펙트럼

| 방식 | 결정 시점 | 유연성 | 런타임 비용 |
|---|---|---|---|
| 구체 메서드 직접 호출 | 작성 시 (author time) | 낮음 | 없음 |
| 가상 메서드 / 인터페이스 | 런타임 (runtime) | 높음 | 있음 |
| C++ 템플릿 메타프로그래밍 | 컴파일 시 (compile time) | 중간 | 없음 |

**성능 최적화는 가정(assumption)에 의존한다.**

- 적이 256마리를 넘지 않는다고 가정 → ID를 1바이트에 패킹 가능
- 특정 타입에만 메서드를 호출한다 → 정적 디스패치(static dispatch) 또는 인라인(inline) 가능
- 모든 엔티티가 같은 클래스 → 연속 배열(contiguous array)로 캐시 효율 극대화

유연성은 가정을 줄이고, 최적화는 가정을 늘린다. **이 둘은 근본적으로 긴장 관계에 있다.**

### 그러나 유연성도 중요하다

> **재미있는 게임을 빠르게 만드는 것이, 빠른 게임을 재미있게 만드는 것보다 쉽다.**

게임 디자인은 반복(iteration)과 실험을 요구한다. 아이디어를 빠르게 시도해볼 수 있어야 좋은 게임이 나온다. 디자인이 안정되기 전까지는 유연성을 유지하고, **이후 성능이 필요한 부분의 추상화를 제거하는 전략**이 현실적이다.

## 나쁜 코드의 가치 (The Good in Bad Code)

잘 설계된 코드만이 답이 아니다. **상황에 따라 다른 코딩 스타일이 필요하다.**

### 프로토타이핑(Prototyping)의 정당성

초기 아이디어 검증 단계에서는 아름다운 아키텍처보다 **빠른 프로토타입**이 우선이다.

> 게임플레이 아이디어가 실제로 재미있는지 확인하기도 전에  
> 아키텍처에 시간을 쏟는 것은 낭비다.

**단, 프로토타입 코드는 반드시 버릴 수 있어야 한다.**

흔한 함정:

```
Boss: "그냥 프로토타입이니까 빠르게 만들어 봐."
Dev:  "몇 가지 코너를 자르면 며칠 안에 대충 동작하는 걸 만들 수 있어요."
Boss: "좋아!"
(며칠 후)
Boss: "좋은데, 그거 조금만 정리해서 실제 버전으로 쓰면 안 될까?"
```

프로토타입이 실제 코드가 되는 것을 방지하는 트릭: **프로토타입을 게임과 다른 언어로 작성한다.** 그러면 실제 게임에 쓰려면 반드시 다시 작성해야 한다.

---

## 균형 잡기 (Striking a Balance)

세 가지 힘이 동시에 작용한다:

| 목표 | 설명 |
|---|---|
| **좋은 아키텍처** | 프로젝트 수명 전반에 걸쳐 코드 이해를 쉽게 한다 |
| **빠른 런타임 성능** | 게임이 목표 플랫폼에서 원활하게 동작한다 |
| **빠른 기능 구현** | 오늘의 피처를 오늘 완성한다 |

이 세 가지는 **부분적으로 서로 충돌**한다.

- 좋은 아키텍처는 장기 생산성을 높이지만, 유지에 지속적인 노력이 필요하다.
- 가장 빠르게 작성한 코드가 가장 빠르게 실행되는 경우는 드물다.
- 빠른 기능 구현을 위해 쌓인 해킹(hack)은 미래 생산성을 갉아먹는다.

> 쉬운 답은 없다. 다만 서로 다른 맛의 트레이드오프가 있을 뿐이다.

이 불확실성이 저자에게는 오히려 **흥미로운 이유**다. 체스가 마스터할 수 없기에 평생 탐구할 수 있듯, 소프트웨어 설계도 완벽한 해답 없이 끊임없이 고민하는 분야이기 때문이다.

## 단순함 (Simplicity)

저자가 가장 중요하게 생각하는 원칙:

> **가장 깨끗하고 직접적인 해결책을 찾아라.**  
> 코드를 읽고 나서 다른 방법이 존재할 수 없다고 느껴지는 그 코드.

단순한 코드의 특성:

- **코드 양이 적다** → 머릿속에 올려야 할 양도 적다.
- **오버헤드가 적다** → 자연스럽게 빠른 경향이 있다.
- 단, **단순한 코드가 작성에 더 적은 시간이 걸린다는 뜻은 아니다.** 좋은 해결책은 코드의 축적이 아닌 **증류(distillation)** 다.

> *"더 짧은 편지를 쓰려 했지만, 시간이 부족했습니다."*  
> — 블레즈 파스칼 (Blaise Pascal)

> *"완벽함이란 더 이상 추가할 것이 없을 때가 아니라,  
> 더 이상 뺄 것이 없을 때 달성된다."*  
> — 앙투안 드 생텍쥐페리 (Antoine de Saint-Exupery)

---

## 저자의 핵심 조언

1. **추상화와 디커플링은 빠른 발전을 가능하게 한다.** 단, 그 코드에 정말 유연성이 필요하다고 확신하기 전까지는 시간을 쏟지 마라.

2. **성능은 개발 사이클 전반에 걸쳐 고민해야 한다.** 하지만 코드에 가정을 굳혀버리는 저수준(low-level) 최적화는 최대한 늦추어라.

3. **빠르게 게임 디자인 공간을 탐색하되**, 뒤에 엉망을 남기지 마라. 그 코드와 함께 살아가야 하는 것은 결국 자신이다.

4. **버릴 코드라면 예쁘게 만드는 데 시간 낭비하지 마라.** 록스타가 내일 체크아웃할 호텔 방을 엉망으로 두는 것처럼.

5. **무엇보다, 재미있는 것을 만들고 싶다면 만드는 과정 자체를 즐겨라.**

## 핵심 개념 정리

| 개념 | 설명 |
|---|---|
| **커플링 (Coupling)** | 두 코드 조각이 서로 없이는 이해할 수 없는 상태 |
| **디커플링 (Decoupling)** | 각 부분을 독립적으로 이해하고 변경할 수 있는 상태 |
| **추상화 (Abstraction)** | 구현 세부사항을 숨기고 인터페이스를 통해 소통하는 설계 |
| **프로토타이핑 (Prototyping)** | 디자인 질문에 답하기 위해 기능만 겨우 동작하는 코드를 빠르게 작성하는 것 |
| **YAGNI** | "You Aren't Gonna Need It" — 지금 당장 필요하지 않은 유연성을 미리 추가하지 마라 |
| **정적 디스패치 (Static Dispatch)** | 컴파일 시점에 어떤 메서드를 호출할지 결정되는 방식 |
| **가상 디스패치 (Virtual Dispatch)** | 런타임에 실제 타입에 따라 호출할 메서드가 결정되는 방식 |
