# Chapter 9: 게임 루프 (Game Loop)

> **Game Programming Patterns** — Robert Nystrom 저  
> Part III: Sequencing Patterns / Chapter 9

---

## 의도 (Intent)

> 게임 시간의 진행을 사용자 입력 및 프로세서 속도로부터 디커플링한다.

게임 루프는 이 책에서 빼놓을 수 없는 가장 중요한 패턴이다. 거의 모든 게임에서 사용되며, 게임 프로그래밍을 다른 프로그래밍과 구분 짓는 핵심 개념이다.

---

## 배치 모드 → 이벤트 루프 → 게임 루프

### 배치 모드 프로그램 (Batch Mode)

초기 컴퓨터 프로그램은 **한 번 실행하고 결과를 내고 끝나는** 방식이었다.

```
[코드 입력] → [실행] → [결과 출력] → 종료
```

### 이벤트 루프 (Event Loop)

인터랙티브 프로그램의 등장. 텍스트 어드벤처 게임이 초기 예시다.

```cpp
while (true)
{
    char* command = readCommand(); // 입력을 기다린다 (블로킹)
    handleCommand(command);
}
```

현대 GUI 프로그램도 동일한 구조:

```cpp
while (true)
{
    Event* event = waitForEvent(); // 사용자 이벤트를 기다린다 (블로킹)
    dispatchEvent(event);
}
```

**문제**: 입력이 없으면 프로그램이 멈춘다. 게임은 입력 없이도 계속 실행되어야 한다.

### 게임 루프 (Game Loop)

```cpp
while (true)
{
    processInput(); // 입력을 기다리지 않는다 (논블로킹)
    update();       // 게임 시뮬레이션 1스텝 진행
    render();       // 현재 상태 렌더링
}
```

핵심 차이: **입력을 기다리지(block) 않는다.**

---

## 게임 시간 문제 (The Time Problem)

루프가 가능한 한 빠르게 돌면 두 가지 문제가 생긴다:

| 문제 | 현상 |
|---|---|
| **빠른 머신** | 게임이 너무 빠르게 진행돼 플레이 불가 |
| **느린 머신** | 게임이 느리게 진행돼 품질 저하 |

> 예전 PC에 "터보 버튼"이 있었던 이유: 새 PC가 너무 빨라 옛 게임이 정상 속도로 안 돌아갔기 때문.

게임 루프의 두 번째 핵심 역할: **다양한 하드웨어에서 일관된 속도로 게임이 실행되도록 한다.**

---

## 구현 방식의 진화

### 방식 1: 그냥 최대한 빠르게 (Uncapped)

```cpp
while (true)
{
    processInput();
    update();
    render();
}
```

- 단순하지만 하드웨어 속도에 그대로 종속된다.

---

### 방식 2: 고정 FPS + Sleep

목표 FPS(예: 60)를 정해두고, 프레임이 일찍 끝나면 남은 시간을 sleep한다.

```cpp
while (true)
{
    double start = getCurrentTime();
    processInput();
    update();
    render();
    sleep(start + MS_PER_FRAME - getCurrentTime());
}
```

```
한 프레임 = 16ms (60FPS 기준)
처리 시간이 10ms → 6ms sleep
처리 시간이 20ms → sleep 없음 (프레임 드롭)
```

- **장점**: 너무 빠른 실행 방지, 배터리 절약
- **단점**: 너무 느린 실행은 해결 못 함. 처리가 16ms를 넘으면 게임이 슬로우다운.

---

### 방식 3: 가변 타임스텝 (Variable Time Step)

지난 프레임에 걸린 실제 시간만큼 게임을 진행한다.

```cpp
double lastTime = getCurrentTime();
while (true)
{
    double current = getCurrentTime();
    double elapsed = current - lastTime; // 실제 경과 시간
    processInput();
    update(elapsed); // 경과 시간을 넘겨 게임 시간 조정
    render();
    lastTime = current;
}
```

```
빠른 머신: elapsed 작음 → 작은 스텝 × 많은 횟수
느린 머신: elapsed 큼  → 큰 스텝 × 적은 횟수
→ 실제 시간 기준으로 동일하게 진행
```

- **장점**: 다양한 하드웨어에서 일관된 실시간 진행, 빠른 머신은 더 부드러운 경험
- **단점**: **비결정적(non-deterministic)이고 불안정하다**

**가변 타임스텝의 치명적 문제:**

부동소수점 연산에서 스텝 횟수에 비례해 오차가 누적된다.

```
Fred의 고사양 머신: 50프레임에 걸쳐 총알 이동 계산
George의 저사양 머신: 5프레임에 걸쳐 총알 이동 계산

→ 같은 총알이 다른 위치에!
```

멀티플레이어 네트워크 동기화, 물리 시뮬레이션 안정성 모두 파괴된다.

> 물리 엔진의 **댐핑(damping)** 은 특정 타임스텝에 맞게 세밀하게 튜닝된다.  
> 타임스텝이 가변적이면 댐핑이 무너지고 물체가 공중으로 날아가는 "폭발" 현상이 발생한다.

---

### 방식 4: 고정 업데이트 + 가변 렌더링 (권장) ★

**핵심 아이디어**: 물리/AI 업데이트는 고정 타임스텝으로, 렌더링은 유연하게.

```cpp
double previous = getCurrentTime();
double lag = 0.0;

while (true)
{
    double current = getCurrentTime();
    double elapsed = current - previous;
    previous = current;
    lag += elapsed; // 실제 시간 누적

    processInput();

    // 고정 스텝으로 따라잡기
    while (lag >= MS_PER_UPDATE)
    {
        update();           // 고정 타임스텝 업데이트
        lag -= MS_PER_UPDATE;
    }

    render(); // 렌더링은 lag에 무관하게 실행
}
```

```
실제 시간:  |----16ms----|----16ms----|----20ms----|
업데이트:   |update|     |update|     |update|upd|
렌더링:         R            R              R
lag:        16→0         16→0         20→4→0
```

| 상황 | 동작 |
|---|---|
| 빠른 머신 | 업데이트 후 lag가 0에 가까워 렌더링 자주 실행 → 부드러운 화면 |
| 느린 머신 | lag가 쌓이면 inner loop에서 여러 번 update → 렌더링은 건너뜀 |
| 매우 느린 머신 | inner loop 최대 반복 횟수를 제한해 완전 락업(lock-up) 방지 |

---

## 잔류 lag와 보간 렌더링

고정 업데이트 + 가변 렌더링 방식의 남은 문제:

```
업데이트 타임라인:  |---U---|---U---|---U---|
렌더링 타임라인:        R           R       R
                         ↑
               업데이트 사이의 임의 시점에 렌더링
```

총알이 왼쪽 → 오른쪽으로 이동할 때, 두 업데이트 사이 시점에 렌더링하면 아직 왼쪽에 있는 총알이 그려진다 → 끊김 현상.

**해결: 보간(Interpolation)**

```cpp
render(lag / MS_PER_UPDATE); // 0.0 ~ 1.0 사이의 정규화된 값 전달
```

```
lag = 0    → 이전 프레임 위치 그대로
lag = 0.5  → 이전 프레임과 다음 프레임 중간 위치로 보간
lag = 0.99 → 다음 프레임 위치에 거의 근접
```

렌더러는 현재 위치와 속도를 알고 있으므로, `lag` 비율만큼 위치를 보간해 부드러운 움직임을 표현한다.

> 보간은 미래를 예측하는 것(외삽, extrapolation)이므로 틀릴 수 있다.  
> 그러나 보간 없이 끊기는 것보다 훨씬 자연스럽게 보인다.

## 설계 결정 (Design Decisions)

### 1. 게임 루프를 직접 소유하는가?

| 선택 | 특징 |
|---|---|
| **플랫폼 이벤트 루프 사용** | 단순, 플랫폼 친화적. 타이밍 제어 불가. 웹 브라우저(`requestAnimationFrame`) 등 |
| **게임 엔진 루프 사용** | 작성 불필요. 하지만 커스터마이징 제한 |
| **직접 작성** | 완전한 제어권. 플랫폼 이벤트 처리를 직접 관리해야 함 (`PeekMessage()` 등) |

> **라이브러리 vs 엔진의 차이**:  
> 라이브러리는 네가 메인 루프를 소유하고 라이브러리를 호출한다.  
> 엔진은 엔진이 루프를 소유하고 네 코드를 호출한다.

---

### 2. 전력 소비를 어떻게 관리하는가?

모바일/노트북 시대에는 배터리 소모도 중요한 설계 고려사항이다.

| 전략 | 설명 | 적합한 플랫폼 |
|---|---|---|
| **최대 속도로 실행** | 남는 사이클을 FPS/그래픽 품질 향상에 사용 | PC (유선 전원) |
| **프레임 레이트 상한 제한** | 목표 FPS(30/60) 달성 후 남는 시간은 sleep | 모바일, 노트북 |

---

### 3. 게임플레이 속도를 어떻게 제어하는가?

| 방식 | 장점 | 단점 |
|---|---|---|
| **고정 스텝, 동기화 없음** | 가장 단순 | 하드웨어 속도에 종속 |
| **고정 스텝 + sleep 동기화** | 단순, 배터리 절약 | 느린 머신에서 슬로우다운 |
| **가변 타임스텝** | 빠름/느림 모두 대응 | 비결정적, 물리 불안정 |
| **고정 업데이트 + 가변 렌더링** | 안정적, 적응형 | 구현 복잡도 높음 |

## 전체 구조 다이어그램

```
게임 루프 1회전 (1 tick / 1 frame)
┌─────────────────────────────────────────────────────┐
│                                                     │
│  1. processInput()                                  │
│     └── 이전 프레임 이후 입력 수집 (논블로킹)          │
│                                                     │
│  2. update() × N  ← 고정 타임스텝 내부 루프           │
│     ├── AI 처리                                     │
│     ├── 물리 시뮬레이션                              │
│     └── 게임 상태 갱신                               │
│                                                     │
│  3. render(lag / MS_PER_UPDATE)                     │
│     └── 현재 상태 + lag 보간으로 화면 출력            │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 핵심 정리

| 개념 | 설명 |
|---|---|
| **게임 루프** | 입력 처리 → 업데이트 → 렌더링을 반복하는 메인 루프 |
| **FPS (Frames Per Second)** | 초당 루프 실행 횟수. 높을수록 부드러운 화면 |
| **고정 타임스텝 (Fixed Time Step)** | 매 업데이트마다 동일한 시간 단위를 사용. 결정론적이고 안정적 |
| **가변 타임스텝 (Variable Time Step)** | 실제 경과 시간을 기반으로 업데이트. 편리하지만 비결정적 |
| **결정론 (Determinism)** | 같은 입력 → 항상 같은 출력. 멀티플레이어·버그 재현에 필수 |
| **lag** | 게임 시계가 실제 시계보다 뒤처진 양. 따라잡기 루프의 기준 |
| **보간 렌더링 (Interpolated Rendering)** | lag 비율로 이전/다음 프레임 사이 위치를 보간해 부드러운 화면 표현 |
| **댐핑 (Damping)** | 물리 시뮬레이션 안정화를 위한 에너지 감쇠. 고정 타임스텝에 맞게 튜닝 필요 |
