## 개요

현대 컴퓨터는 **이진(binary)** 신호로 정보를 저장하고 처리한다. 비트를 어떻게 묶고 해석하느냐에 따라 부호 없는 정수(unsigned), 부호 있는 정수(two's complement), 부동소수점(floating point) 등 다양한 표현이 가능하다.

세 가지 핵심 수 표현:
- **Unsigned**: 0 이상의 정수 (전통적 이진 표기)
- **Two's-complement**: 음수 포함 정수 (가장 일반적인 부호 있는 정수 표현)
- **IEEE 754 Floating-point**: 실수 근사 표현

## 2.1 정보 저장 (Information Storage)

### 바이트와 가상 메모리

컴퓨터는 개별 비트 대신 **8비트(1바이트)** 를 최소 주소 단위로 사용한다. 기계 수준 프로그램은 메모리를 거대한 바이트 배열(**가상 메모리**, virtual memory)로 본다. 각 바이트는 고유한 **주소(address)** 를 가지며, 가능한 모든 주소의 집합이 **가상 주소 공간(virtual address space)** 이다.

### 2.1.1 16진수 표기 (Hexadecimal Notation)

1바이트 = 8비트 → 이진수로 00000000₂ ~ 11111111₂, 십진수로 0 ~ 255.

이진수는 장황하고 십진수는 변환이 불편하다. **16진수(hexadecimal)** 로 표기하면 편리하다:
- 0~9, A~F(또는 a~f) 사용
- C에서는 `0x` 또는 `0X` 접두사

```
십진수   16진수   이진수
  0       0x0     0000
  9       0x9     1001
 10       0xA     1010
 15       0xF     1111
255       0xFF    11111111
```

**16진수 ↔ 이진수 변환**: 4비트씩 묶어서 1:1 대응

```
0x3A  →  0011 1010
0xFA1D37B  →  0000 1111 1010 0001 1101 0011 0111 1011
```

**16진수 ↔ 십진수 변환**: $x = 2^n$ 형태면 16진수로 바로 변환 가능.  
예) $2^{11} = 2048_{10} = 0x800$ (n = 11 = 4×2 + 3 → 8뒤에 0이 2개)

### 2.1.2 데이터 크기 (Data Sizes)

모든 컴퓨터는 **워드 크기(word size)** 가 있으며, 이것이 포인터 데이터의 명목상 크기를 결정한다. 64비트 워드 = 가상 주소 공간 최대 $2^{64}$ 바이트.

C 데이터 타입의 크기 (바이트):

| C 선언 | 32비트 프로그램 | 64비트 프로그램 |
|--------|:----------:|:----------:|
| `char` | 1 | 1 |
| `short` | 2 | 2 |
| `int` | 4 | 4 |
| `long` | 4 | **8** |
| `int32_t` | 4 | 4 |
| `int64_t` | 8 | 8 |
| `char *` (포인터) | 4 | **8** |
| `float` | 4 | 4 |
| `double` | 8 | 8 |

> **함정**: 역사적으로 `int`에 포인터를 저장하는 코드가 많았다. 32비트에서는 동작하지만 64비트에서는 버그가 된다. `int32_t`, `int64_t` 같은 고정 크기 타입(`<stdint.h>`) 사용을 권장.

### 2.1.3 주소 지정과 바이트 순서 (Addressing and Byte Ordering)

멀티바이트 객체는 연속된 메모리에 저장되며, **가장 작은 주소가 객체의 주소**가 된다.

예) `int x`가 주소 `0x100`에 저장될 때, 4바이트는 `0x100~0x103`에 위치.

**바이트 순서(byte ordering)**:

```
값: 0x01234567 (4바이트)

빅 엔디언(Big Endian):    가장 중요한 바이트(MSB)가 낮은 주소
  주소:  0x100  0x101  0x102  0x103
  값:    0x01   0x23   0x45   0x67

리틀 엔디언(Little Endian): 가장 덜 중요한 바이트(LSB)가 낮은 주소
  주소:  0x100  0x101  0x102  0x103
  값:    0x67   0x45   0x23   0x01
```

- **빅 엔디언**: SPARC, z/Series (IBM 메인프레임)
- **리틀 엔디언**: x86-64 (Intel/AMD), ARM (주로)
- **바이 엔디언(bi-endian)**: 설정 가능 (ARM, PowerPC)

**엔디언이 중요한 상황**:
1. 네트워크로 이진 데이터 전송 시 (네트워크 바이트 순서 = 빅 엔디언)
2. 정수 데이터를 담은 바이너리 파일을 다른 머신에서 읽을 때
3. 디버거로 메모리를 직접 들여다볼 때

```c
// 바이트 순서 확인 코드
typedef unsigned char *byte_pointer;

void show_bytes(byte_pointer start, size_t len) {
    for (size_t i = 0; i < len; i++)
        printf(" %.2x", start[i]);
    printf("\n");
}

int x = 0x01234567;
show_bytes((byte_pointer) &x, sizeof(int));
// 리틀 엔디언: 67 45 23 01
// 빅 엔디언:   01 23 45 67
```

### 2.1.4~2.1.5 문자열 및 코드 표현

- **문자열**: ASCII로 인코딩된 문자들의 배열. null 문자(`\0`, 0x00)로 종료. 텍스트 데이터는 이진 데이터와 달리 엔디언 무관하게 플랫폼 호환.
- **코드**: 동일한 C 소스라도 컴파일 후 기계어는 머신/OS마다 달라 **이진 호환성(binary compatibility) 없음**.

### 2.1.6 불리언 대수 (Boolean Algebra)

비트에 대한 4가지 기본 연산:

| 연산 | 기호 | 규칙 |
|------|------|------|
| NOT | `~` | 비트 반전 |
| AND | `&` | 둘 다 1일 때만 1 |
| OR  | `\|` | 하나라도 1이면 1 |
| XOR | `^` | 다를 때만 1 |

비트 벡터로 확장하면 집합 연산과 동일:
- AND(`&`) = 교집합, OR(`|`) = 합집합, XOR(`^`) = 대칭 차집합, NOT(`~`) = 여집합

### 2.1.7 C에서의 비트 연산

C의 비트 연산자(`|`, `&`, `~`, `^`)는 **모든 정수형 타입**에 적용 가능. 16진수 → 이진수 → 연산 → 16진수 순서로 계산:

```c
~0x41         // ~[01000001] = [10111110] = 0xBE
0x69 & 0x55   // [01101001] & [01010101] = [01000001] = 0x41
0x69 | 0x55   // [01101001] | [01010101] = [01111101] = 0x7D
```

**XOR을 이용한 임시 변수 없는 스왑**:

```c
// x ^ x = 0, x ^ 0 = x 성질 활용
void inplace_swap(int *x, int *y) {
    *y = *x ^ *y;  // Step 1
    *x = *x ^ *y;  // Step 2
    *y = *x ^ *y;  // Step 3
}
// 단, x == y (동일 메모리)이면 0이 됨 — 주의!
```

**비트 마스킹(masking)** 패턴:
```c
x & 0xFF         // 최하위 바이트만 추출
x | 0xFF         // 최하위 바이트를 모두 1로
x ^ 0xFF         // 최하위 바이트 비트 반전
```

### 2.1.8 논리 연산 vs 비트 연산

| 구분 | 연산자 | 의미 | 결과 타입 |
|------|--------|------|---------|
| 비트 연산 | `&`, `\|`, `~`, `^` | 비트별 연산 | 정수 |
| 논리 연산 | `&&`, `\|\|`, `!` | 전체를 true/false | 0 또는 1 |

```c
!0x41   // 0x00 (0이 아니므로 false → 0)
!0x00   // 0x01 (0이므로 true → 1)
0x69 && 0x55  // 0x01 (둘 다 non-zero → true)
0x69 || 0x55  // 0x01
```

`&&`와 `||`은 **단락 평가(short-circuit evaluation)**: 첫 번째 피연산자로 결과가 확정되면 두 번째 평가하지 않음.

```c
// p가 NULL이면 *p 역참조하지 않음
if (p && *p == 'a') { ... }
```

### 2.1.9 C에서의 시프트 연산

| 연산 | 구문 | 설명 |
|------|------|------|
| 왼쪽 시프트 (Left shift) | `x << k` | 왼쪽으로 k비트 이동, 오른쪽 k비트는 0으로 채움 |
| 논리 오른쪽 시프트 (Logical right shift) | `x >> k` (unsigned) | 왼쪽 k비트를 0으로 채움 |
| 산술 오른쪽 시프트 (Arithmetic right shift) | `x >> k` (signed) | 왼쪽 k비트를 MSB(부호 비트)로 채움 |

```c
// w = 8비트 예시
x        = [01100010]  →  0x62
x << 4   = [00100000]  →  0x20  (상위 4비트 소실)
x >> 4   = [00000110]  →  0x06  (논리, unsigned)
x >> 4   = [00000110]  →  0x06  (산술, 양수면 동일)

x        = [10010101]  →  MSB=1
x >> 4   = [00001001]  →  0x09  (논리, unsigned)
x >> 4   = [11111001]  →  0xF9  (산술, signed, MSB=1이므로 1로 채움)
```

> **C 표준 주의**: signed 타입에 `>>`를 사용할 경우 산술/논리 시프트 여부는 구현 정의(implementation-defined). 단, 대부분의 머신에서 산술 시프트 수행. shift 양이 워드 크기 이상이면 **undefined behavior**.

## 2.2 정수 표현 (Integer Representations)

### 2.2.1 C 정수형 범위

**32비트 프로그램 기준:**

| 타입 | 최솟값 | 최댓값 |
|------|--------|--------|
| `char` | -128 | 127 |
| `short` | -32,768 | 32,767 |
| `int` | -2,147,483,648 | 2,147,483,647 |
| `long` | -2,147,483,648 | 2,147,483,647 |
| `unsigned` | 0 | 4,294,967,295 |
| `int64_t` | -9,223,372,036,854,775,808 | 9,223,372,036,854,775,807 |

> 범위가 **비대칭**임에 주목: 음수 범위가 양수보다 1 크다. Two's complement 특성.

### 2.2.2 비부호 인코딩 (Unsigned Encodings)

w비트 벡터 $\vec{x} = [x_{w-1}, x_{w-2}, \ldots, x_0]$ 에 대해:

$$B2U_w(\vec{x}) = \sum_{i=0}^{w-1} x_i \cdot 2^i$$

범위: $0 \leq B2U_w(\vec{x}) \leq UMax_w = 2^w - 1$

**$B2U_w$는 전단사 함수(bijection)**: w비트 벡터와 $[0, 2^w-1]$ 정수 사이의 1:1 대응.

### 2.2.3 2의 보수 인코딩 (Two's-Complement Encodings)

MSB(최상위 비트)에 **음의 가중치**를 부여:

$$B2T_w(\vec{x}) = -x_{w-1} \cdot 2^{w-1} + \sum_{i=0}^{w-2} x_i \cdot 2^i$$

- **MSB = 1** → 음수 (부호 비트, sign bit)
- **MSB = 0** → 음이 아닌 수

범위: $TMin_w = -2^{w-1}$ ~ $TMax_w = 2^{w-1} - 1$

예) 4비트:
```
[1011] → -1·2³ + 0·2² + 1·2¹ + 1·2⁰ = -8 + 0 + 2 + 1 = -5
[0101] → 0·2³ + 1·2² + 0·2¹ + 1·2⁰ =  0 + 4 + 0 + 1 =  5
```

**중요 상수** (w=32 기준):

| 상수 | 값 | 16진수 |
|------|-----|--------|
| `UMax` | 4,294,967,295 | `0xFFFFFFFF` |
| `TMax` | 2,147,483,647 | `0x7FFFFFFF` |
| `TMin` | -2,147,483,648 | `0x80000000` |
| `-1`   | -1             | `0xFFFFFFFF` |

`TMin`은 절댓값이 `TMax + 1`이므로 `-TMin = TMin` (오버플로). C에서 `INT_MIN`을 쓸 때 `-2147483647 - 1`로 표기하는 이유.

### 2.2.4 부호/비부호 변환 (Conversions)

C에서 signed ↔ unsigned 캐스팅은 **비트 패턴 보존**, 해석만 변경:

$$T2U_w(x) = \begin{cases} x + 2^w & x < 0 \\ x & x \geq 0 \end{cases}$$

$$U2T_w(u) = \begin{cases} u - 2^w & u > TMax_w \\ u & u \leq TMax_w \end{cases}$$

```c
short x = -12345;   // 비트: [1100 1111 1100 0111]
unsigned short u = (unsigned short) x;
// u = 53191  (같은 비트 패턴, 다른 해석)
// 53191 = 65536 - 12345 = 2^16 + (-12345)
```

### 2.2.5 C에서 부호/비부호 혼용의 함정

C의 묵시적 캐스팅 규칙: **signed + unsigned 연산 시 signed를 unsigned로 변환**

```c
// 직관에 반하는 결과들
-1 < 0U          // → 0 (false!) : -1이 unsigned로 변환되어 UMax가 됨
2147483647U > -2147483647-1  // → 0 (false!) : 오른쪽이 unsigned로 변환
(unsigned)-1 > -2   // → 1 (true) : 둘 다 unsigned 변환
```

```c
// 보안 취약점 예시: 길이 비교 버그
// size_t는 unsigned 타입
int strlonger(char *s, char *t) {
    return strlen(s) - strlen(t) > 0;  // WRONG!
    // strlen 반환이 unsigned라서 음수 결과가 없음
    // 올바른 코드: return strlen(s) > strlen(t);
}
```

### 2.2.6 비트 표현 확장 (Bit Expansion)

작은 타입 → 큰 타입 변환 시:

- **Unsigned → 큰 unsigned**: **제로 확장(zero extension)** — 상위 비트를 0으로 채움
- **Signed → 큰 signed**: **부호 확장(sign extension)** — 상위 비트를 MSB로 채움

```c
short sx = -12345;  // 비트: [1100 1111 1100 0111]
int   x  = sx;      // 부호 확장: [1111 1111 1111 1111 1100 1111 1100 0111]
                    // 값 보존: 여전히 -12345

unsigned short usx = 53191;  // 비트: [1100 1111 1100 0111]
unsigned int  ux   = usx;    // 제로 확장: [0000 0000 0000 0000 1100 1111 1100 0111]
                             // 값 보존: 여전히 53191
```

**주의**: C에서 `short` → `unsigned int` 변환은 먼저 부호 확장 후 unsigned로 해석!

```c
short sx = -1;           // [1111 1111 1111 1111]
unsigned int ux = sx;    // 먼저 int로 부호확장 → -1 → 그 다음 unsigned → 4294967295
```

### 2.2.7 수 잘라내기 (Truncation)

큰 타입 → 작은 타입 변환 시 상위 비트를 버린다. 이는 **오버플로**의 한 형태:

```c
int x = 53191;      // 비트: [0000 0000 0000 0000 1100 1111 1100 0111]
short sx = (short)x; // 하위 16비트만: [1100 1111 1100 0111] → -12345
int y = sx;          // 부호 확장: -12345
```

- Unsigned 자르기: $x' = x \bmod 2^k$ (k = 목표 비트 수)
- Signed 자르기: 먼저 unsigned처럼 자른 후 two's complement로 재해석

### 2.2.8 부호/비부호 사용 지침

**Unsigned 사용이 적합한 경우**:
- 비트 플래그(bit flags) — 수치 해석 없이 비트 모음으로 사용할 때
- 포인터 산술 (시스템 프로그래머)
- 모듈러 산술(modular arithmetic) 구현
- 다중 정밀도(multi-precision) 산술

**Unsigned 사용 주의**:
- 루프 카운터: `for (size_t i = n-1; i >= 0; i--)` → i가 0에서 감소하면 UMax가 됨 → 무한루프!
- 암묵적 signed↔unsigned 변환이 일어나는 비교 연산

## 2.3 정수 산술 (Integer Arithmetic)

컴퓨터의 정수 연산은 **유한 워드 크기**의 제약으로 인해 실제 정수 연산과 다르다.

### 2.3.1 Unsigned 덧셈

w비트 unsigned 덧셈은 $2^w$로 모듈러(modular) 연산:

$$x +^u_w y = \begin{cases} x + y & x + y < 2^w \quad \text{(정상)} \\ x + y - 2^w & 2^w \leq x + y < 2^{w+1} \quad \text{(오버플로)} \end{cases}$$

```
4비트 예시 (UMax = 15):
 9 + 12 = 21 = 10101₂ → 하위 4비트만: 0101 = 5  (오버플로 발생)
```

**오버플로 감지**:
```c
// s = x + y가 오버플로 됐는지 확인
int uadd_ok(unsigned x, unsigned y) {
    unsigned s = x + y;
    return s >= x;  // 오버플로 시 s < x
}
```

**Unsigned 음원(negation)**:

$$-^u_w x = \begin{cases} 0 & x = 0 \\ 2^w - x & x > 0 \end{cases}$$

### 2.3.2 Two's-Complement 덧셈

w비트 two's-complement 덧셈 결과를 다시 w비트 two's-complement로 해석:

$$x +^t_w y = \begin{cases} x + y - 2^w & x + y \geq 2^{w-1} \quad \text{(양수 오버플로)} \\ x + y & -2^{w-1} \leq x + y < 2^{w-1} \quad \text{(정상)} \\ x + y + 2^w & x + y < -2^{w-1} \quad \text{(음수 오버플로)} \end{cases}$$

```c
// 양수 오버플로: 두 양수를 더했는데 음수가 됨
int x = 2147483647;   // TMax
int y = x + 1;        // -2147483648 (TMin) — 양수 오버플로!

// 음수 오버플로: 두 음수를 더했는데 양수가 됨
int a = -2147483648;  // TMin
int b = a + (-1);     // 2147483647 (TMax) — 음수 오버플로!
```

**오버플로 감지** (C에서 signed overflow는 undefined behavior이므로 미리 감지):

```c
// x + y가 오버플로 없이 합산되는지 확인
int tadd_ok(int x, int y) {
    int sum = x + y;
    int neg_over = (x < 0 && y < 0 && sum >= 0);  // 음수 오버플로
    int pos_over = (x >= 0 && y >= 0 && sum < 0);  // 양수 오버플로
    return !neg_over && !pos_over;
}
```

### 2.3.3 Two's-Complement 부정 (Negation)

$$-^t_w x = \begin{cases} TMin_w & x = TMin_w \quad \text{(-TMin은 표현 불가)} \\ -x & x > TMin_w \end{cases}$$

**비트 레벨에서 음수화**: 모든 비트 반전 후 1 더하기  
`-x == ~x + 1` (C에서 성립)

```c
x   = [0101]  =  5
~x  = [1010]  = -6
~x+1= [1011]  = -5  ✓

x   = [1000]  = -8  (TMin, 4비트)
~x  = [0111]  =  7
~x+1= [1000]  = -8  (오버플로, TMin의 음수는 TMin)
```

### 2.3.4~2.3.5 정수 곱셈

**Unsigned 곱셈**: 진짜 결과를 $2^w$로 모듈러

$$x *^u_w y = (x \cdot y) \bmod 2^w$$

**Two's-complement 곱셈**: 진짜 결과의 하위 w비트를 two's complement로 해석  
→ **unsigned와 곱셈 비트 결과는 동일**하다! 단, 해석만 다름.

```c
// 3비트 예시
5 * 3 = 15 = [01111]₂ → 하위 3비트 [111] = 7u = -1t
-3 * 3 = -9 = [10111]₂ → 하위 3비트 [111] = 7u = -1t  // 같은 비트 패턴!
```

### 2.3.6 상수 곱셈 최적화 (Multiplying by Constants)

**정수 곱셈 명령어는 느리다** (Intel Core i7: 3 사이클). 반면 시프트, 덧셈은 1 사이클.  
컴파일러는 상수 곱셈을 **시프트+덧셈 조합**으로 최적화:

**2의 거듭제곱 곱셈**: `x << k` = `x * 2^k`

```c
x * 14  →  14 = 2³ + 2² + 2¹
         →  (x<<3) + (x<<2) + (x<<1)
         →  또는 (x<<4) - (x<<1)   // 16x - 2x
```

| C 식 | 최적화 | 연산수 |
|------|--------|--------|
| `x * 14` | `(x<<3)+(x<<2)+(x<<1)` | 2 add + 3 shift |
| `x * 14` | `(x<<4)-(x<<1)` | 1 sub + 2 shift |

### 2.3.7 2의 거듭제곱 나눗셈 (Dividing by Powers of 2)

**정수 나눗셈은 더 느리다** (30+ 사이클).

- **Unsigned**: `x >> k` = `⌊x / 2^k⌋` (논리 오른쪽 시프트)
- **Signed (양수)**: `x >> k` = `⌊x / 2^k⌋` (산술 오른쪽 시프트)
- **Signed (음수)**: 올림(ceiling) 필요 → `(x + (1<<k) - 1) >> k` = `⌈x / 2^k⌉`

```c
// -7 / 4의 올바른 결과는 -1 (반올림이 0 방향)
int x = -7;
int k = 2;
int result = (x + (1<<k) - 1) >> k;  // (-7 + 3) >> 2 = -4 >> 2 = -1  ✓
// 단순 >> 는 -2를 반환 (바닥 방향 반올림)
```

컴파일러는 `x / 4`를 자동으로 위 패턴으로 최적화한다.

### 2.3.8 정수 산술 최종 정리

Two's-complement 정수는 비트 수준에서 **unsigned와 완전히 동일한 구현**을 공유한다:
- 덧셈, 뺄셈, 곱셈의 비트 결과는 동일 (해석만 다름)
- 이것이 two's-complement를 선택한 핵심 이유

**핵심 함정 요약**:
- signed overflow는 C에서 **undefined behavior** (컴파일러가 없다고 가정하고 최적화)
- unsigned는 항상 모듈러 산술 (오버플로 = well-defined behavior)
- signed ↔ unsigned 묵시적 변환에서 예상치 못한 결과 발생

## 2.4 부동소수점 (Floating Point)

### 2.4.1 분수 이진수 (Fractional Binary Numbers)

이진 소수점 표기: $b_m b_{m-1} \ldots b_1 b_0 . b_{-1} b_{-2} \ldots b_{-n}$

$$\sum_{i=-n}^{m} b_i \cdot 2^i$$

```
101.11₂ = 4 + 0 + 1 + 1/2 + 1/4 = 5.75
0.001₂ = 1/8 = 0.125
```

**문제**: $1/3 = 0.010101..._2$ 처럼 무한히 반복되는 수는 유한 비트로 **정확히 표현 불가** (근사값만 가능).

### 2.4.2 IEEE 754 부동소수점 표현

IEEE 표준 754 (1985년, 현재 대부분의 하드웨어가 구현):

부동소수점 수 = $(-1)^s \times M \times 2^E$

- **s (sign, 1비트)**: 부호 (0=양수, 1=음수)
- **E (exponent)**: 지수 (바이어스 인코딩)
- **M (significand/mantissa)**: 유효수 (1≤M<2 범위)

**형식별 비트 필드:**

```
단정밀도 (single-precision, float, 32비트):
 31  30    23  22              0
  s  [exp:8비트]  [frac:23비트]
  bias = 127 (2^(8-1) - 1)

배정밀도 (double-precision, double, 64비트):
 63  62        52  51                      0
  s  [exp:11비트]  [frac:52비트]
  bias = 1023 (2^(11-1) - 1)
```

### 세 가지 값의 종류 (단정밀도 기준):

#### ① 정규화 수 (Normalized Values)
`exp ≠ 00000000` and `exp ≠ 11111111`

$$E = e - \text{Bias}, \quad M = 1 + f$$

- 지수: $e$는 exp 필드의 unsigned 값, $E = e - 127$ (-126 ~ +127)
- 가수: 암묵적 선행 1 (implied leading 1): $M = 1.f_{22}f_{21}\ldots f_0$  
  → frac=0이면 M=1.0, frac=최댓값이면 M≈2.0

```
예) 단정밀도로 15213.0 표현:
15213 = 11101101101101₂ = 1.1101101101101₂ × 2¹³
s=0, E=13, e=13+127=140=10001100₂, frac=1101101101101 00000000000
→ 0 10001100 11011011011010000000000
```

#### ② 비정규화 수 (Denormalized Values)
`exp = 00000000` (모두 0)

$$E = 1 - \text{Bias}, \quad M = f \quad \text{(암묵적 leading 1 없음)}$$

두 가지 목적:
1. **정확한 0 표현**: frac=0이면 ±0.0 (양수 0과 음수 0이 다른 비트 패턴)
2. **점진적 언더플로(gradual underflow)**: 0에 매우 가까운 수들을 균일한 간격으로 표현

#### ③ 특수 값 (Special Values)
`exp = 11111111` (모두 1)

| frac | 의미 |
|------|------|
| 0 | ±∞ (무한대, 오버플로나 0으로 나눌 때) |
| ≠ 0 | NaN (Not a Number: √-1, ∞-∞ 등) |

#### 수 체계 전체 구조:

```
-∞  음수정규화  음수비정규화  -0  +0  양수비정규화  양수정규화  +∞
←───────────────────────────────────────────────────────────────→
        NaN 영역 (음수/양수 inf 바깥)
```

### 2.4.3 표현 가능한 수 예시

6비트 형식 예(k=3 exp, n=2 frac, bias=3):

```
비트    e   E   f      M      값
000 00   0  -2  0/4  0/4     0
000 01   0  -2  1/4  1/4     1/16 = 0.0625
000 10   0  -2  2/4  2/4     2/16 = 0.125
001 00   1  -2  0/4  4/4     1/4 = 0.25
...
011 00   3   0  0/4  4/4     1.0
011 01   3   0  1/4  5/4     1.25
110 11   6   3  3/4  7/4     14.0  ← 최댓값
111 00   -   -   0    -      +∞
111 01   -   -  ≠0   -      NaN
```

### 2.4.4 반올림 (Rounding)

IEEE 754의 4가지 반올림 모드:

| 모드 | 양수 1.5 | 음수 -1.5 | 특징 |
|------|---------|---------|------|
| **최근접 짝수 (Round-to-even)** | 2 | -2 | 기본 모드, 통계적 편향 없음 |
| 0 방향 (Round-toward-zero) | 1 | -1 | 절삭(truncation) |
| 양의 무한대 방향 (Round-up) | 2 | -1 | 천장(ceiling) |
| 음의 무한대 방향 (Round-down) | 1 | -2 | 바닥(floor) |

**최근접 짝수(Round-to-even)** 가 기본인 이유: 단순 반올림(항상 0.5를 올림)은 계산 반복 시 통계적으로 위쪽으로 편향됨. 짝수 쪽으로 반올림하면 50% 올림, 50% 내림 → 편향 없음.

```
십진수 → 소수점 이하 0자리로 반올림 (최근접 짝수):
1.5 → 2  (1과 2 중 짝수 선택)
2.5 → 2  (2와 3 중 짝수 선택)
7.5 → 8  (7과 8 중 짝수 선택)
-0.5 → 0 (0과 -1 중 짝수 선택)
```

이진수에서도 동일 원칙: LSB가 0(짝수)이 되도록 반올림.

### 2.4.5 부동소수점 연산

IEEE 부동소수점 연산 결과 정의: "정확한 수학 결과에 반올림한 값"

**교환법칙은 성립, 결합법칙과 분배법칙은 성립하지 않음**:

```c
// 결합법칙 위반
(3.14 + 1e10) - 1e10 == 0.0        // 3.14가 소실됨
3.14 + (1e10 - 1e10) == 3.14       // 다른 결과!

// 분배법칙 위반
1e20 * (1e20 - 1e20) == 0.0
1e20 * 1e20 - 1e20 * 1e20 == NaN  // 오버플로 발생
```

**부동소수점 연산의 단조성(monotonicity)**:  
`a ≥ b` 이면 `a + c ≥ b + c` — 부동소수점에서도 성립 (NaN 제외).  
하지만 정수 산술에서는 오버플로로 인해 단조성이 깨질 수 있음.

### 2.4.6 C에서의 부동소수점

**캐스팅 규칙** (C99 기준):

| 변환 | 동작 |
|------|------|
| `int` → `float` | 반올림 가능 (float는 24비트 유효수, int는 32비트) |
| `int`/`float` → `double` | 정확한 보존 (double은 53비트 유효수) |
| `double` → `float` | 오버플로(±∞ 가능), 반올림 |
| `float`/`double` → `int` | 0 방향 잘라냄(truncate). 범위 초과 시 undefined behavior |

```c
(int) 1e10    // 10000000000 → int 범위 초과 → undefined behavior!
(int) 1e7     // 10000000 → 정확히 표현 가능

// 부동소수점 비교 주의
float f = 1e10;
int i = (int) f;
// f와 (float)i가 같을 수도 있고 다를 수도 있음
```

**주요 상수 표현 가능성**:

```c
// float (23비트 frac, 유효 십진수 약 7자리)
float f = 16777216.0f;     // 2^24 → 정확
float g = 16777217.0f;     // 2^24 + 1 → 16777216으로 반올림!

// double (52비트 frac, 유효 십진수 약 15-16자리)
double d = 9007199254740992.0;  // 2^53 → 정확
double e = 9007199254740993.0;  // 2^53 + 1 → 9007199254740992로 반올림!
```

## 2.5 요약

| 표현 방식 | 특징 | 범위 (w비트) |
|---------|------|-------------|
| Unsigned | 자연수, 모듈러 산술 | $0$ ~ $2^w - 1$ |
| Two's-complement | 음수 포함, MSB=부호비트 | $-2^{w-1}$ ~ $2^{w-1}-1$ |
| IEEE 754 | 실수 근사, 특수값 처리 | 매우 넓음 (±∞, NaN 포함) |

**핵심 교훈:**

1. **모든 수치 표현은 유한하다** — 오버플로와 반올림이 언제나 가능
2. **비트 패턴은 컨텍스트에 따라 다른 의미** — 같은 `0xFF`가 255(unsigned)이기도 하고 -1(signed)이기도 함
3. **C의 암묵적 타입 변환이 버그의 근원** — signed ↔ unsigned 혼용 시 특히 위험
4. **부동소수점은 실수가 아니다** — 결합법칙, 분배법칙 불성립; 비교 연산 주의
5. **Two's-complement와 unsigned는 동일한 하드웨어** — 덧셈/곱셈의 비트 결과가 동일

**실무 체크리스트**:
```c
// ✗ 위험한 패턴들
unsigned i;
for (i = n-1; i >= 0; i--)  // i가 unsigned라 항상 참 → 무한루프

int len = strlen(s) - strlen(t);  // unsigned 뺄셈, 음수 결과 없음
if (len > 0) { ... }              // 버그!

float x = 1e20f;
int i = (int) x;  // undefined behavior

// ✓ 안전한 패턴들
for (int i = n-1; i >= 0; i--)   // signed 사용
if (strlen(s) > strlen(t)) { ... }
if (x >= INT_MIN && x <= INT_MAX) { int i = (int)x; }
```
