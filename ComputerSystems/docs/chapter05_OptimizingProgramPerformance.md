# Chapter 5: Optimizing Program Performance

> 프로그램 성능을 극대화하기 위한 체계적 접근법을 다룬다.  
> 컴파일러의 한계를 이해하고, 루프 최적화·메모리 참조 제거·명령어 수준 병렬성(ILP) 활용·루프 언롤링·복수 어큐뮬레이터 기법까지 프로그래머 수준에서 적용 가능한 최적화를 깊이 탐구한다.

## 5.1 컴파일러 최적화의 능력과 한계

컴파일러는 `-O1`, `-O2`, `-O3` 같은 최적화 수준을 통해 코드를 변환하지만, **안전한 최적화(safe optimization)** 만 적용한다. 결과가 달라질 가능성이 있으면 변환하지 않는다.

### 최적화를 막는 두 가지 주요 요인

**① 메모리 별칭(Memory Aliasing)**

서로 다른 두 포인터가 같은 메모리 위치를 가리킬 가능성이 있으면 컴파일러는 최적화를 포기한다.

```c
// 컴파일러가 최적화하지 못하는 경우
void twiddle1(long *xp, long *yp) {
    *xp += *yp;   // 메모리 읽기 2번, 쓰기 1번
    *xp += *yp;   // 메모리 읽기 2번, 쓰기 1번 → 총 6회 메모리 접근
}

// 컴파일러가 twiddle1을 아래로 변환하면 안 됨:
void twiddle2(long *xp, long *yp) {
    *xp += 2 * *yp;  // xp == yp인 경우 결과가 달라짐!
}
```

`xp == yp`이면 `twiddle1`은 값을 4배로, `twiddle2`는 3배로 만든다 → 의미가 다름.

**② 함수 부작용(Function Side Effects)**

```c
// 컴파일러는 f()를 1번으로 줄이지 못함 (f에 부작용이 있을 수 있으므로)
x = f() + f() + f() + f();
```

함수 호출이 전역 상태를 변경하거나 I/O를 수행할 수 있으므로 컴파일러는 보수적으로 동작한다.

### 실용적 결론

컴파일러가 못 하는 최적화는 **프로그래머가 직접** 해야 한다. 특히:
- 루프 내 불변 계산을 루프 밖으로 이동
- 불필요한 메모리 읽기/쓰기 제거
- 포인터 별칭 가능성을 없애는 코드 구조

## 5.2 프로그램 성능 표현: CPE

**CPE (Cycles Per Element)**: 벡터/배열의 원소 하나를 처리하는 데 걸리는 평균 클록 사이클 수.

반복 계산에서 실행 시간이 선형일 때:
$$T = C_0 + \text{CPE} \times n$$

`$C_0$`는 함수 호출·루프 초기화 오버헤드, `$n$`은 원소 수. 큰 `$n$`에서는 CPE가 지배적.

**측정 방법**: 여러 크기 `$n$`에 대해 CPU 사이클 측정 후 기울기 추출.

---

## 5.3 예제 프로그램

이 장의 최적화 예제는 벡터(vector) 자료구조의 **combine** 함수를 기준으로 한다:

```c
typedef struct {
    long   len;
    data_t *data;
} vec_rec, *vec_ptr;

// 기준 함수: 모든 원소를 OP 연산으로 누적
void combine1(vec_ptr v, data_t *dest) {
    long i;
    *dest = IDENT;                              // 초기값 (+ → 0, * → 1)
    for (i = 0; i < vec_length(v); i++) {
        data_t val;
        get_vec_element(v, i, &val);            // 경계 검사 포함
        *dest = *dest OP val;
    }
}
```

`data_t`와 `OP`를 컴파일 타임 상수로 조합해 int/float/double과 덧셈/곱셈을 테스트한다.

## 5.4 루프 비효율 제거 (Code Motion)

### 루프 불변 계산 (Loop-Invariant Computation)

```c
// combine1: 매 반복마다 vec_length() 호출 — 결과가 변하지 않는데도
for (i = 0; i < vec_length(v); i++) { ... }

// combine2: 루프 밖으로 이동 (code motion)
long length = vec_length(v);
for (i = 0; i < length; i++) { ... }
```

| 함수 | 방법 | int + | int * | float + | float * |
|------|------|------:|------:|--------:|--------:|
| combine1 | 기준 (-O1) | 10.12 | 10.12 | 10.17 | 11.14 |
| combine2 | vec_length 이동 | 7.02 | 9.03 | 9.02 | 11.03 |

컴파일러는 `vec_length`에 부작용이 있을 수 있으므로 자동으로 최적화하지 않는다.

**극단적 예시 — O(n²) → O(n)**:
```c
// lower1: strlen()을 루프 조건에 사용 → O(n²)
for (i = 0; i < strlen(s); i++) ...

// lower2: 루프 밖으로 이동 → O(n)
size_t len = strlen(s);
for (i = 0; i < len; i++) ...
```

## 5.5 프로시저 호출 감소

```c
// combine3: get_vec_element 대신 배열 직접 접근
void combine3(vec_ptr v, data_t *dest) {
    long i;
    long length = vec_length(v);
    data_t *data = get_vec_start(v);   // 배열 시작 주소 직접 획득

    *dest = IDENT;
    for (i = 0; i < length; i++) {
        *dest = *dest OP data[i];      // 경계 검사 없음, 직접 접근
    }
}
```

> 놀랍게도 combine3의 CPE는 combine2와 거의 같다 — 아직 **메모리 별칭 문제**가 남아있기 때문.

## 5.6 불필요한 메모리 참조 제거

**combine3의 문제**: 루프마다 `*dest`를 읽고 쓴다. 컴파일러는 `data`와 `dest`가 별칭일 가능성을 배제할 수 없어 최적화 불가.

```asm
# combine3 내부 루프 (double 곱셈)
.L17:
    vmovsd (%rbx), %xmm0       # *dest 읽기   ← 매번 메모리 읽기!
    vmulsd (%rdx), %xmm0, %xmm0
    vmovsd %xmm0, (%rbx)       # *dest 쓰기   ← 매번 메모리 쓰기!
    addq   $8, %rdx
    ...
```

**해결책**: 지역 변수 `acc`로 누적, 루프 종료 후 한 번만 저장:

```c
// combine4: 임시 변수에 누적
void combine4(vec_ptr v, data_t *dest) {
    long i;
    long length = vec_length(v);
    data_t *data = get_vec_start(v);
    data_t acc = IDENT;             // 레지스터에 유지

    for (i = 0; i < length; i++) {
        acc = acc OP data[i];       // 메모리 읽기 1회만
    }
    *dest = acc;                    // 루프 종료 후 1회 쓰기
}
```

```asm
# combine4 내부 루프 (double 곱셈) - 메모리 쓰기 사라짐
.L25:
    vmulsd (%rdx), %xmm0, %xmm0   # acc(xmm0) *= data[i]
    addq   $8, %rdx
    cmpq   %rax, %rdx
    jne    .L25
```

| 함수 | 방법 | int + | int * | float + | float * |
|------|------|------:|------:|--------:|--------:|
| combine3 | 직접 배열 접근 | 7.17 | 9.02 | 9.02 | 11.03 |
| combine4 | 임시 변수 누적 | **1.27** | **3.01** | **3.01** | **5.01** |

## 5.7 현대 프로세서 이해

### 5.7.1 비순서 실행 (Out-of-Order Execution) 프로세서

현대 프로세서는 **슈퍼스칼라(superscalar)** + **비순서 실행(out-of-order)** 구조:

```
┌─────────────────────────────────────────────┐
│         명령어 제어 유닛 (ICU)               │
│  Fetch → Decode → 명령어 큐 (FIFO)          │
│  분기 예측 (Branch Prediction)               │
│  투기적 실행 (Speculative Execution)         │
├─────────────────────────────────────────────┤
│         실행 유닛 (EU)                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│  │정수 ALU ×4│ │FP 덧셈×1 │ │FP 곱셈×1 │     │
│  └──────────┘ └──────────┘ └──────────┘     │
│  ┌──────────┐ ┌──────────┐                  │
│  │로드 유닛×2│ │스토어유닛×1│                │
│  └──────────┘ └──────────┘                  │
├─────────────────────────────────────────────┤
│         퇴직 유닛 (Retirement Unit)           │
│  레지스터 파일, 순차적 의미론 보장           │
└─────────────────────────────────────────────┘
```

**핵심 개념**:
- **레지스터 재명명(Register Renaming)**: 물리 레지스터가 프로그램 레지스터보다 많아, 이름 충돌(WAW, WAR hazard) 없이 여러 명령어 동시 실행
- **투기적 실행**: 분기 결과를 알기 전에 예측하여 실행 시작, 틀리면 취소

### 5.7.2 기능 유닛 성능 (Intel Core i7 Haswell 기준)

| 연산 | 지연(Latency) | 이슈 시간(Issue) | 용량(Capacity) |
|------|:------------:|:---------------:|:--------------:|
| 정수 덧셈 | 1 | 1 | 4 |
| 정수 곱셈 | 3 | 1 | 1 |
| 정수 나눗셈 | 3~30 | 3~30 | 1 |
| FP 덧셈 | 3 | 1 | 1 |
| FP 곱셈 | 5 | 1 | 2 |
| FP 나눗셈 | 15~17 | 15~17 | 1 |

- **지연(Latency)**: 연산 완료까지 걸리는 클록 사이클
- **이슈 시간(Issue time)**: 같은 유닛에서 연속 독립 연산을 시작하는 최소 간격. 이슈 시간 = 1 → **완전 파이프라인(fully pipelined)**
- **용량(Capacity)**: 동시에 수행할 수 있는 기능 유닛 수

**핵심 성능 한계**:

| 바운드 | 설명 | int + | int * | fp + | fp * |
|--------|------|------:|------:|-----:|-----:|
| **지연 바운드(Latency Bound)** | 순차 체인에서 지연 합계 = CPE 하한 | 1.00 | 3.00 | 3.00 | 5.00 |
| **처리량 바운드(Throughput Bound)** | 기능 유닛 처리율 = CPE 절대 하한 | 0.50 | 1.00 | 1.00 | 0.50 |

`combine4`의 CPE ≈ 지연 바운드: 단일 누적 변수 때문에 연산이 직렬로 연결됨.

### 5.7.3 데이터 흐름 그래프 (Data-Flow Graph)

프로그램의 데이터 의존성을 시각화하여 **임계 경로(critical path)** 를 파악한다.

```c
// combine4 내부 루프 (double 곱셈)
// 명령어: vmulsd (%rdx),%xmm0,%xmm0 | addq $8,%rdx | cmpq | jne
```

```
반복 i:
  %xmm0 ──► [mul] ──► %xmm0   (xmm0에 5사이클 의존성 체인)
  %rdx  ──► [add] ──► %rdx    (rdx는 독립적, 1사이클)

반복 i+1:
  이전 xmm0 결과가 완료되어야 mul 시작 가능
  → 임계 경로 = 5사이클/반복 → CPE = 5.00
```

**템플릿 표기법**: 루프 내 연산들의 의존성 패턴을 한 번 그리면, n회 반복 실행 시간 = n × (임계 경로 길이).

## 5.8 루프 언롤링 (Loop Unrolling)

한 루프 반복에서 여러 원소를 처리하여 루프 오버헤드(인덱스 증가, 조건 검사)를 줄인다.

```c
// combine5: 2×1 루프 언롤링
void combine5(vec_ptr v, data_t *dest) {
    long length = vec_length(v);
    long limit  = length - 1;
    data_t *data = get_vec_start(v);
    data_t acc = IDENT;

    for (i = 0; i < limit; i += 2) {
        acc = (acc OP data[i]) OP data[i+1];  // 2원소씩 처리
    }
    for (; i < length; i++)                   // 나머지 처리
        acc = acc OP data[i];
    *dest = acc;
}
```

| 함수 | 방법 | int + | int * | fp + | fp * |
|------|------|------:|------:|-----:|-----:|
| combine4 | 기본 | 1.27 | 3.01 | 3.01 | 5.01 |
| combine5 | 2×1 언롤링 | 1.01 | 3.01 | 3.01 | 5.01 |
| — | 지연 바운드 | 1.00 | 3.00 | 3.00 | 5.00 |

루프 오버헤드는 줄었지만 **단일 누적 변수(`acc`)의 순차 의존성은 그대로** → 지연 바운드에 묶임.

## 5.9 병렬성 향상

지연 바운드를 깨려면 누적 변수 자체의 순차 체인을 끊어야 한다.

### 5.9.1 복수 어큐뮬레이터 (Multiple Accumulators)

짝수/홀수 인덱스 원소를 각각 다른 변수에 누적 → 두 개의 독립 연산 체인:

```c
// combine6: 2×2 루프 언롤링 (2개 어큐뮬레이터)
void combine6(vec_ptr v, data_t *dest) {
    long limit = vec_length(v) - 1;
    data_t *data = get_vec_start(v);
    data_t acc0 = IDENT, acc1 = IDENT;

    for (i = 0; i < limit; i += 2) {
        acc0 = acc0 OP data[i];      // 짝수 인덱스 체인
        acc1 = acc1 OP data[i+1];    // 홀수 인덱스 체인 (독립!)
    }
    for (; i < length; i++)
        acc0 = acc0 OP data[i];
    *dest = acc0 OP acc1;
}
```

**데이터 흐름 그래프**:
```
반복 i:
  acc0 ──► [mul] ──► acc0   ← 체인 1
  acc1 ──► [mul] ──► acc1   ← 체인 2 (체인 1과 독립!)

두 mul은 같은 사이클에 시작 가능 → 유효 CPE ≈ Latency/2
```

| 함수 | 방법 | int + | int * | fp + | fp * |
|------|------|------:|------:|-----:|-----:|
| combine5 | 2×1 언롤링 | 1.01 | 3.01 | 3.01 | 5.01 |
| combine6 | 2×2 언롤링 | **0.81** | **1.51** | **1.51** | **2.51** |
| — | 처리량 바운드 | 0.50 | 1.00 | 1.00 | 0.50 |

처리량 바운드에 접근! k×k 언롤링으로 처리량 바운드까지 도달 가능.

### 5.9.2 재결합 변환 (Reassociation Transformation)

같은 2×1 언롤링이지만 괄호 위치만 바꿔서 의존성 체인을 끊는다:

```c
// combine5 (원본): acc = (acc OP data[i]) OP data[i+1]
//   의존성: acc → [op] → temp → [op] → acc  (순차)

// combine7 (재결합): acc = acc OP (data[i] OP data[i+1])
//   data[i] OP data[i+1]은 acc와 독립적으로 먼저 계산 가능!
for (i = 0; i < limit; i += 2) {
    acc = acc OP (data[i] OP data[i+1]);   // 괄호 이동
}
```

**데이터 흐름**:
```
반복 i:
  data[i] ──► [op] ──► temp ──► [op] ──► acc
  data[i+1] ───────────────────────┘
                ↑ 이 op는 이전 반복의 acc와 무관
                  → 다음 반복과 겹쳐 실행 가능
```

| 함수 | 방법 | int + | int * | fp + | fp * |
|------|------|------:|------:|-----:|-----:|
| combine5 | 2×1 언롤링 | 1.01 | 3.01 | 3.01 | 5.01 |
| combine7 | 2×1a (재결합) | 1.01 | 1.51 | 1.51 | 2.51 |

> **주의**: 부동소수점 연산은 수학적으로 결합 법칙(associative)을 완전히 만족하지 않는다. 재결합 변환은 반올림 오차 패턴을 약간 바꿀 수 있으므로 컴파일러는 자동으로 수행하지 않는다.

## 5.11 제한 요인

### 5.11.1 레지스터 스필링 (Register Spilling)

병렬 어큐뮬레이터 수가 사용 가능한 레지스터 수를 초과하면 컴파일러가 스택에 임시 저장(spill):

```asm
# 10×10 언롤링: acc0은 레지스터 %xmm0에 유지
vmulsd (%rdx), %xmm0, %xmm0      ← 빠름

# 20×20 언롤링: acc0이 스택으로 밀려남
vmovsd 40(%rsp), %xmm0            ← 메모리 로드
vmulsd (%rdx),   %xmm0, %xmm0
vmovsd %xmm0,  40(%rsp)           ← 메모리 저장
```

x86-64는 16개 정수 레지스터 + 16개 YMM 레지스터 → 대부분의 루프는 처리량 한계에 먼저 도달한다.

### 5.11.2 분기 예측 실패 패널티 (Branch Misprediction Penalty)

프로세서는 분기 방향을 예측하여 투기적 실행을 시작하며, 예측 실패 시 파이프라인을 플러시한다.

- Core i7의 분기 예측 실패 패널티: **약 15~20 사이클**
- 루프 종료 조건, 순차 패턴 → 높은 예측 정확도
- 데이터 의존 조건문 (ex. 랜덤 데이터 정렬) → 낮은 예측 정확도

**대응책**: 조건부 이동(conditional move) 사용:

```c
// 조건부 분기 → 예측 실패 위험
if (src1[i1] < src2[i2]) dest[id++] = src1[i1++];
else                      dest[id++] = src2[i2++];

// cmov 스타일로 변환 → 분기 없음
long v1 = src1[i1], v2 = src2[i2];
long take1 = (v1 < v2);
dest[id++] = take1 ? v1 : v2;
// 컴파일러가 cmovXX로 변환 → 분기 예측 패널티 없음
```

## 5.12 메모리 성능 이해

### 5.12.1 로드 성능 (Load Performance)

```
로드 처리량 바운드:
  로드 유닛 2개 × 1회/사이클 → 최대 2로드/사이클
  k개 로드/원소 필요 시: CPE ≥ k/2

로드 지연(L1 캐시 히트): 4사이클 (Core i7 Haswell)
```

**로드 지연이 임계 경로가 되는 예**: 연결 리스트 탐색

```c
// 각 로드의 주소가 이전 로드 결과에 의존 → 포인터 체이싱
long list_len(list_ptr ls) {
    long len = 0;
    while (ls) {
        len++;
        ls = ls->next;   // ls 로드 → 주소 계산 → 다음 ls 로드 (체인!)
    }
    return len;
}
// CPE = 4 (L1 로드 지연 = 병목)
```

### 5.12.2 스토어 성능과 쓰기/읽기 의존성 (Write/Read Dependency)

```c
// write_read: 쓰기 후 읽기 의존성
void write_read(long *src, long *dst, long n) {
    long cnt = n;
    long val = 0;
    while (cnt) {
        *dst = val;           // 스토어
        val = (*src) + 1;     // 로드 — src == dst이면 이전 스토어 결과에 의존!
        cnt--;
    }
}
```

**스토어-로드 포워딩(Store-Load Forwarding)**: 프로세서는 스토어 버퍼에서 아직 캐시에 기록되지 않은 값을 로드에 직접 전달한다. `src ≠ dst`이면 독립적으로 실행되어 CPE ≈ 1.3; `src == dst`이면 포워딩 체인이 형성되어 CPE ≈ 7.3.

## 5.14 성능 병목 식별 및 제거

### 5.14.1 프로파일링 (Profiling) — gprof

```bash
# 1. 프로파일링 플래그로 컴파일
gcc -Og -pg prog.c -o prog

# 2. 실행 (gmon.out 생성)
./prog input.txt

# 3. 프로파일 분석
gprof prog
```

**gprof 출력 형식**:
- **Flat profile**: 각 함수의 CPU 시간 비율, 호출 횟수
- **Call graph**: 함수 호출 관계와 누적 시간

```
  %   cumulative   self              self     total
 time   seconds   seconds    calls  ms/call  ms/call  name
 97.58      8.21     8.21        1  8210.00  8210.00  sort_words
  1.07      8.30     0.09  965027     0.00     0.00   lower1
  0.79      8.37     0.07    3585     0.02     0.02   find_ele_rec
```

→ `sort_words`가 97.58% 시간 차지: 먼저 이 함수를 최적화해야 한다.

### 5.14.2 암달의 법칙 (Amdahl's Law) 재방문

전체 시간의 비율 `α`를 차지하는 부분을 `k`배 빠르게 했을 때의 전체 속도 향상:

$$S = \frac{1}{(1-\alpha) + \alpha/k}$$

```
예: 전체 시간의 80%를 10배 빠르게 → S = 1/(0.2 + 0.08) = 3.57배
   (나머지 20%가 병목이 됨)
```

**교훈**: 프로파일러로 실제 병목을 찾은 후 집중 최적화하라. 전체의 작은 부분을 아무리 최적화해도 전체 성능 향상은 제한적이다.

### 핵심 교훈

- **측정 먼저**: 프로파일 없이 최적화하는 것은 낭비. 실제 병목 함수를 찾아라.
- **CPE와 바운드 이해**: 내 코드가 지연 바운드에 걸렸는지, 처리량 바운드에 걸렸는지 판단하면 다음 최적화 방향이 명확해진다.
- **컴파일러를 도와라**: 별칭 제거, 임시 변수 사용으로 컴파일러가 레지스터에 값을 유지하도록 유도하라.
- **ILP(명령어 수준 병렬성)**: 현대 프로세서는 독립 연산을 동시에 실행할 수 있다. 복수 어큐뮬레이터와 재결합으로 이를 활용하라.
- **SIMD 최후 수단**: AVX 벡터 연산으로 처리량 바운드를 8배 이상 낮출 수 있지만, 복잡도가 크게 증가한다.
