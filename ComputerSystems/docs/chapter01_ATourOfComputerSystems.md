## 개요

컴퓨터 시스템은 하드웨어와 시스템 소프트웨어가 협력하여 애플리케이션 프로그램을 실행하는 구조다. 구체적인 구현은 시대에 따라 변하지만 **근본 개념은 불변**한다. 이 책은 시스템이 어떻게 동작하는지 이해함으로써 더 올바르고 빠른 프로그램을 작성하는 프로그래머를 위한 책이다.

```c
/* code/intro/hello.c */
#include <stdio.h>

int main() {
    printf("hello, world\n");
    return 0;
}
```

이 단순한 프로그램이 실행되기까지 시스템의 모든 핵심 구성 요소가 관여한다.

---

## 1.1 정보는 비트와 컨텍스트다 (Information Is Bits + Context)

`hello.c` 소스 파일은 **비트들의 시퀀스**다. 각 바이트는 ASCII 코드로 하나의 문자를 나타낸다.

```
'#' = 35, 'i' = 105, 'n' = 110, ... '\n' = 10
```

**핵심 원리**: 시스템 내의 모든 정보 — 디스크 파일, 메모리의 프로그램, 네트워크로 전송되는 데이터 — 는 모두 **비트들의 묶음**으로 표현된다. 서로 다른 데이터 객체를 구별하는 유일한 기준은 **컨텍스트(context)**다. 동일한 바이트 시퀀스도 컨텍스트에 따라 정수, 부동소수점, 문자열, 기계어 명령어가 될 수 있다.

> 이는 왜 C에서 타입 캐스팅(type casting)이 위험한지, 왜 정수 오버플로가 예상 밖의 결과를 낳는지를 설명하는 근본 이유다.

## 1.2 컴파일 시스템 (Compilation System)

소스 파일에서 실행 파일로의 변환은 4단계를 거친다:

```
hello.c  →[cpp]→  hello.i  →[cc1]→  hello.s  →[as]→  hello.o  →[ld]→  hello
         전처리기       컴파일러           어셈블러        링커
         (텍스트)       (텍스트)           (바이너리)      (바이너리)
```

```bash
linux> gcc -o hello hello.c
```

### 각 단계 상세

| 단계 | 프로그램 | 입력 | 출력 | 설명 |
|------|---------|------|------|------|
| 전처리 (Preprocessing) | `cpp` | `hello.c` | `hello.i` | `#include`, `#define` 등 지시자 처리. 헤더 파일 인라인 삽입 |
| 컴파일 (Compilation) | `cc1` | `hello.i` | `hello.s` | C → 어셈블리 변환. 어셈블리는 서로 다른 고수준 언어 컴파일러의 공통 출력 포맷 |
| 어셈블 (Assembly) | `as` | `hello.s` | `hello.o` | 어셈블리 → 기계어. **재배치 가능 오브젝트 파일(relocatable object file)** 생성 |
| 링크 (Linking) | `ld` | `hello.o`, `printf.o` | `hello` | 외부 라이브러리 병합. 실행 가능한 **실행 오브젝트 파일(executable object file)** 생성 |

컴파일 단계에서 생성되는 어셈블리 예시:
```asm
main:
    subq  $8, %rsp
    movl  $.LC0, %edi
    call  puts
    movl  $0, %eax
    addq  $8, %rsp
    ret
```

## 1.3 컴파일 시스템을 이해해야 하는 이유

단순 프로그램은 컴파일러에 맡겨도 되지만, 전문 프로그래머라면 반드시 이해해야 한다:

**1. 프로그램 성능 최적화**  
- `switch` vs `if-else` 중 어느 쪽이 더 효율적인가?  
- 함수 호출 오버헤드는 얼마인가?  
- `while`과 `for` 루프의 기계어 차이는?  
- 포인터 참조 vs 배열 인덱싱의 비용 차이?  
- 로컬 변수에 누적하는 것이 왜 참조 인자보다 빠른가?  
→ 챕터 3, 5, 6에서 심층 탐구

**2. 링크 타임 오류 이해**  
- `unresolved reference` 오류의 원인  
- static 변수 vs global 변수의 차이  
- 서로 다른 C 파일에서 같은 이름의 전역 변수 정의 시 발생하는 일  
- 정적 라이브러리 vs 동적 라이브러리  
- 라이브러리 커맨드라인 순서의 중요성  
→ 챕터 7에서 심층 탐구

**3. 보안 취약점 방지**  
- 버퍼 오버플로우(buffer overflow) 취약점은 수십 년간 네트워크/인터넷 서버의 주요 보안 구멍  
- 스택 규율(stack discipline)과 메모리 레이아웃 이해가 보안 프로그래밍의 첫걸음  
→ 챕터 3에서 상세 다룸

## 1.4 프로세서의 하드웨어 구조

### 시스템 하드웨어 구성

```
┌─────────────────────────────────────────────────────────────┐
│  CPU                                                        │
│  ┌──────────┐    ┌──────┐                                   │
│  │Register  │    │      │                                   │
│  │  file    │←──→│ ALU  │                                   │
│  └──────────┘    └──────┘                                   │
│        │  PC                                                │
│  ──────┼──────── System bus ──── Memory bus ───────         │
│        │                                                    │
│  ┌─────┴──────┐        ┌─────────────────┐                 │
│  │Bus         │        │  Main Memory    │                 │
│  │Interface   │        │  (DRAM)         │                 │
│  └────────────┘        └─────────────────┘                 │
│        │                                                    │
│  ──────┴──────────── I/O bus ───────────────────           │
│        │                                                    │
│  ┌─────┴──┐  ┌──────┐  ┌──────┐  ┌──────────┐            │
│  │  USB   │  │Graph.│  │Disk  │  │ Network  │            │
│  │ ctrl   │  │adapt.│  │ctrl  │  │ adapter  │            │
│  └────────┘  └──────┘  └──────┘  └──────────┘            │
└─────────────────────────────────────────────────────────────┘
```

### 핵심 구성 요소

**버스(Buses)**  
컴포넌트 간 바이트 블록을 전송하는 전기적 도관. 고정 크기의 청크 단위(**word**)로 전송. 현대 시스템은 주로 4바이트(32비트) 또는 8바이트(64비트) word 크기.

**I/O 디바이스**  
시스템의 외부 세계 연결점. 각 디바이스는 컨트롤러나 어댑터를 통해 I/O 버스에 연결.

**메인 메모리(Main Memory)**  
프로세서가 프로그램을 실행하는 동안 코드와 데이터를 보관하는 임시 저장 장치. 물리적으로 DRAM(Dynamic Random Access Memory) 칩들로 구성. 논리적으로는 연속된 바이트 배열로 각 바이트에 고유한 주소가 있음.

**프로세서(Processor / CPU)**  
메인 메모리에 저장된 명령어를 읽고 실행하는 엔진. 핵심은 **PC(Program Counter)** — 현재 실행 중인 명령어의 메인 메모리 주소를 가리키는 word 크기 레지스터.

프로세서의 반복 동작 (instruction execution cycle):
1. PC가 가리키는 메모리에서 명령어 읽기
2. 명령어 해석(decode)
3. 명령어가 지시하는 연산 수행
4. PC를 다음 명령어로 갱신 (연속 또는 비연속)

기본 연산 유형:
- **Load**: 메모리 → 레지스터
- **Store**: 레지스터 → 메모리
- **Operate**: ALU를 통한 두 레지스터 값 연산 → 레지스터
- **Jump**: PC를 특정 주소로 업데이트

### `hello` 프로그램 실행 흐름

1. 쉘 프롬프트에서 `./hello` 입력
2. 쉘이 실행 파일을 디스크 → 메인 메모리로 로드 (DMA 활용)
3. CPU가 `main` 루틴의 기계어 명령어 실행
4. `"hello, world\n"` 문자열을 메모리 → 레지스터 → 디스플레이 디바이스로 복사

## 1.5 캐시가 중요한 이유 (Caches Matter)

### 프로세서-메모리 속도 격차 문제

물리 법칙으로 인해 **큰 저장 장치는 느리다**. 빠른 장치는 제조 비용이 비싸다.

| 저장 장치 | 크기 | 속도 (상대적) |
|---------|------|-------------|
| 레지스터 파일 | 수백 바이트 | 1× (기준) |
| L1 캐시 | 수만 바이트 | ~4× 느림 |
| L2 캐시 | 수십~수백만 바이트 | ~5× L1보다 느림 |
| 메인 메모리 | 수십억 바이트 (GB) | ~100× 레지스터보다 느림 |
| 디스크 | 수백~수천 GB | ~10,000,000× 메모리보다 느림 |

반도체 기술 발전으로 이 **프로세서-메모리 갭(processor-memory gap)은 계속 확대**된다.

### 캐시 메모리 구조

```
┌──────────────────────────────────┐
│  CPU Chip                        │
│  ┌──────────┐                    │
│  │ Register │                    │
│  │  file    │                    │
│  └────┬─────┘                    │
│       │                          │
│  ┌────▼─────┐   L1 cache         │
│  │   ALU    │   (on-chip, ~수십KB)│
│  └──────────┘                    │
│  ┌──────────────────────────┐    │
│  │       L2 cache           │    │
│  │  (특수 버스, 수백KB~수MB)  │    │
│  └──────────────────────────┘    │
└──────────────────────────────────┘
          │
    Main Memory (DRAM)
```

**SRAM(Static RAM)**: L1, L2 캐시 구현에 사용. DRAM보다 빠르지만 비쌈.  
최신 시스템은 L3 캐시까지 3단계 캐시 구조를 갖는다.

**지역성(Locality)**: 캐시가 효과적인 이유. 프로그램은 최근 접근한 데이터/코드 주변을 반복적으로 접근하는 경향이 있다. 이 원리를 이해하고 활용하면 **프로그램 성능을 한 자릿수 이상 향상**시킬 수 있다 (챕터 6).

## 1.6 저장 장치 계층 구조 (Storage Hierarchy)

"작고 빠른 저장 장치를 크고 느린 저장 장치와 프로세서 사이에 삽입"하는 개념의 일반화:

```
L0  ┌─────────────────────────────────┐  ← Registers (CPU)
    │  빠름, 작음, 고가 (per byte)    │
L1  ├─────────────────────────────────┤  ← L1 cache (SRAM)
    │                                 │
L2  ├─────────────────────────────────┤  ← L2 cache (SRAM)
    │                                 │
L3  ├─────────────────────────────────┤  ← L3 cache (SRAM)
    │                                 │
L4  ├─────────────────────────────────┤  ← Main memory (DRAM)
    │                                 │
L5  ├─────────────────────────────────┤  ← Local disk (HDD/SSD)
    │  느림, 큼, 저가 (per byte)      │
L6  └─────────────────────────────────┘  ← Remote storage (NFS, Web)
```

**핵심 원리**: 각 레벨의 저장 장치는 그 아래 레벨 저장 장치의 **캐시**로 동작한다.
- 레지스터 파일 → L1 캐시의 캐시
- L1, L2 → L2, L3의 캐시
- L3 → 메인 메모리의 캐시
- 메인 메모리 → 디스크의 캐시
- 로컬 디스크 → 원격 스토리지의 캐시 (분산 파일 시스템)

## 1.7 운영체제의 역할: 하드웨어 관리

`hello` 프로그램이 실행될 때, 프로그램은 키보드나 디스크를 **직접 접근하지 않는다**. 운영체제(OS)가 제공하는 서비스를 통해 간접 접근한다.

```
┌─────────────────────────┐
│   Application programs  │  ← hello, shell, ...
├─────────────────────────┤
│    Operating System     │  ← 하드웨어 접근의 중개자
├─────────────────────────┤
│ Processor  Mem  I/O dev │  ← Hardware
└─────────────────────────┘
```

OS의 두 가지 목적:
1. **하드웨어 오용 방지**: 폭주(runaway) 애플리케이션으로부터 하드웨어 보호
2. **하드웨어 추상화**: 복잡하고 제각각인 저수준 하드웨어 장치를 단순하고 일관된 인터페이스로 제공

OS는 세 가지 핵심 추상화(abstraction)를 통해 이를 달성한다:

```
Processes   → 프로세서 + 메모리 + I/O 디바이스의 추상화
Virtual Mem → 메인 메모리 + 디스크 I/O의 추상화
Files       → I/O 디바이스의 추상화
```

### 1.7.1 프로세스 (Processes)

**프로세스**는 실행 중인 프로그램에 대한 OS의 추상화. 시스템이 프로그램에게 제공하는 환상:
- 프로세서를 독점 사용한다
- 메인 메모리를 독점 사용한다
- I/O 디바이스를 독점 사용한다

여러 프로세스가 단일 시스템에서 동시에(concurrently) 실행될 수 있다. OS는 **컨텍스트 스위칭(context switching)**으로 이를 구현한다.

**컨텍스트(context)**: 프로세스 실행에 필요한 모든 상태 정보
- 현재 PC 값
- 레지스터 파일 내용
- 메인 메모리 내용

```
Process A               Process B
                Time
  User code       ─────────────────────────────▶
  ┌───────┐
  │       │  ← 실행 중
  └───┬───┘
      │ (system call)
  ┌───▼───┐
  │Kernel │  ← 컨텍스트 저장 → B 컨텍스트 복원
  └───────┘
                          ┌───────┐
                          │       │  ← B 실행
                          └───┬───┘
                              │ (disk interrupt)
                          ┌───▼───┐
                          │Kernel │  ← 컨텍스트 저장 → A 컨텍스트 복원
                          └───────┘
  ┌───────┐
  │       │  ← A 재개
  └───────┘
```

**커널(Kernel)**: 항상 메모리에 상주하는 OS 코드. 별도의 프로세스가 아닌, 모든 프로세스를 관리하는 코드/데이터 구조의 집합. 애플리케이션이 시스템 호출(system call) 명령어로 커널에 제어를 넘긴다.

### 1.7.2 스레드 (Threads)

프로세스는 실제로 **여러 실행 단위(스레드)**로 구성될 수 있다. 각 스레드는 프로세스의 컨텍스트 내에서 실행되며, 동일한 코드와 전역 데이터를 공유한다.

스레드가 중요한 이유:
- 네트워크 서버에서 동시성(concurrency) 요구사항 충족
- 프로세스 간 데이터 공유보다 스레드 간 데이터 공유가 더 쉽고 효율적
- 멀티프로세서 활용 시 성능 향상

### 1.7.3 가상 메모리 (Virtual Memory)

각 프로세스가 메인 메모리를 **독점 사용한다는 환상**을 제공하는 추상화. 모든 프로세스는 동일한 균일한 메모리 뷰(**가상 주소 공간(virtual address space)**)를 갖는다.

Linux 프로세스의 가상 주소 공간 구조 (주소 낮음 → 높음):

```
0x0000000000000000
      ┌─────────────────────────────┐
      │  Program code & data        │  ← 실행 파일에서 직접 초기화
      │  (text + initialized data)  │
      ├─────────────────────────────┤
      │  Heap                       │  ← malloc/free로 동적 확장/축소
      │  (run-time heap)            │
      │         ↓                   │
      ├─────────────────────────────┤
      │  Shared libraries           │  ← libc, libm 등 (코드+데이터 공유)
      ├─────────────────────────────┤
      │         ↑                   │
      │  User stack                 │  ← 함수 호출마다 확장/축소
      │  (at run time)              │
      ├─────────────────────────────┤
      │  Kernel virtual memory      │  ← 사용자 코드 접근 불가
      │  (invisible to user code)   │
      └─────────────────────────────┘
0xFFFFFFFFFFFFFFFF
```

가상 메모리 동작의 핵심 아이디어: **프로세스의 가상 메모리 내용을 디스크에 저장하고, 메인 메모리를 디스크의 캐시로 사용**. 이를 위해 프로세서가 생성하는 모든 주소에 대한 하드웨어 번역(address translation)이 필요하다 (챕터 9).

### 1.7.4 파일 (Files)

파일은 **바이트의 시퀀스**, 그 이상도 이하도 아니다.  

- 모든 I/O 디바이스(디스크, 키보드, 디스플레이, 네트워크 포함)는 파일로 모델링된다
- 시스템의 모든 입출력은 **Unix I/O**라는 소수의 시스템 콜로 파일을 읽고 쓰는 방식으로 수행된다
- 이 추상화로 애플리케이션은 디스크 기술의 세부 사항에 무관하게 작동할 수 있다

## 1.8 네트워크 — 또 하나의 I/O 디바이스

시스템 관점에서 **네트워크는 또 다른 I/O 디바이스**에 불과하다. 네트워크 어댑터는 I/O 버스에 연결되며, 시스템은 메인 메모리 → 네트워크 어댑터로 바이트를 복사하고, 반대 방향도 마찬가지다.

```
로컬 시스템                           원격 시스템
┌─────────────────┐   네트워크   ┌─────────────────┐
│  telnet client  │ ──────────▶ │  telnet server  │
│                 │ ◀────────── │  + shell        │
│  (사용자 입력)   │             │  + hello program│
└─────────────────┘             └─────────────────┘
```

`hello` 원격 실행 5단계:
1. 사용자가 로컬 텔넷 클라이언트에 "hello" 타이핑
2. 클라이언트가 "hello" 문자열을 원격 텔넷 서버로 전송
3. 서버가 문자열을 원격 쉘에 전달
4. 원격 쉘이 `hello` 프로그램 실행 후 출력을 텔넷 서버에 반환
5. 텔넷 서버가 출력 문자열을 로컬 클라이언트로 전달 → 화면 출력

## 1.9 중요 주제들 (Important Themes)

### 1.9.1 암달의 법칙 (Amdahl's Law)

시스템 일부를 개선할 때의 전체 효과에 관한 법칙:

> 전체 시스템 성능은 **가장 느린 부분의 속도**와 **그 부분이 전체에서 차지하는 비율**에 의해 제한된다.

**수식:**

전체 실행 시간 $T_{old}$ 중 비율 $\alpha$인 부분을 $k$배 향상시켰을 때, 새 실행 시간:

$$T_{new} = (1 - \alpha) T_{old} + (\alpha T_{old}) / k = T_{old} \left[(1 - \alpha) + \alpha/k\right]$$

스피드업(Speedup):

$$S = \frac{T_{old}}{T_{new}} = \frac{1}{(1 - \alpha) + \alpha/k}$$

**예시**: 전체의 60%($\alpha = 0.6$)를 3배($k = 3$) 향상

$$S = \frac{1}{0.4 + 0.6/3} = \frac{1}{0.6} \approx 1.67\times$$

전체 성능 향상은 개선된 부분의 향상보다 훨씬 작다.

**$k \to \infty$ 극단적 경우** (해당 부분을 완벽히 없앤다면):

$$S_\infty = \frac{1}{1 - \alpha}$$

→ 전체의 60%를 아무리 개선해도 최대 $1/0.4 = 2.5\times$ 향상이 한계.

**함의**: 시스템을 *유의미하게* 빠르게 하려면 **대부분의 구성 요소**를 개선해야 한다.

### 1.9.2 동시성과 병렬성 (Concurrency and Parallelism)

**동시성(Concurrency)**: 동시에 여러 활동이 있는 시스템의 일반적 개념  
**병렬성(Parallelism)**: 동시성을 활용하여 시스템을 더 빠르게 만드는 것

세 가지 수준의 병렬성:

#### ① 스레드 레벨 동시성 (Thread-Level Concurrency)

```
┌───────────────────────────────────────────────────┐
│                  All Processors                    │
│  ┌──────────────┐     ┌────────────────────────┐  │
│  │  Uniprocessor│     │     Multiprocessors    │  │
│  └──────────────┘     │  ┌──────┐  ┌─────────┐│  │
│                        │  │Multi-│  │Hyper-   ││  │
│                        │  │core  │  │threaded ││  │
│                        │  └──────┘  └─────────┘│  │
│                        └────────────────────────┘  │
└───────────────────────────────────────────────────┘
```

**멀티코어 프로세서**: 여러 CPU 코어를 단일 IC 칩에 집적. 각 코어는 독립된 L1, L2 캐시를 가지며, L3 캐시와 메인 메모리 인터페이스는 공유.

**하이퍼스레딩(Hyperthreading)** = Simultaneous Multi-Threading:
- 일부 CPU 하드웨어(PC, 레지스터 파일 등)를 복수로 갖되, 부동소수점 연산 유닛 등은 공유
- 한 스레드가 캐시 미스 대기 중일 때 다른 스레드 실행으로 CPU 활용률 극대화
- 기존 프로세서: 스레드 전환에 ~20,000 클럭 사이클 필요 → 하이퍼스레드 프로세서: 매 사이클 결정
- 예: Intel Core i7 — 각 코어가 2개 스레드 → 4코어 시스템에서 8스레드 병렬 실행

#### ② 명령어 레벨 병렬성 (Instruction-Level Parallelism)

현대 프로세서는 단일 클럭 사이클에 여러 명령어를 실행한다:
- 초기 마이크로프로세서 (Intel 8086, 1978): 명령어당 3~10 사이클 필요
- 현대 프로세서: 클럭당 2~4 명령어 처리

**파이프라이닝(Pipelining)**: 명령어 실행에 필요한 동작을 여러 단계로 분리, 각 단계를 파이프라인으로 병렬 처리. 단순한 하드웨어로 클럭당 1 명령어에 근접한 처리율 달성 (챕터 4).

**슈퍼스칼라(Superscalar) 프로세서**: 클럭당 1개 이상의 명령어를 처리하는 프로세서. 현대 대부분의 프로세서가 해당.

#### ③ SIMD 병렬성 (Single-Instruction, Multiple-Data)

단일 명령어로 여러 데이터에 동일 연산을 병렬 수행:
- 예: Intel/AMD 프로세서의 AVX 명령어 — 단일 명령어로 8쌍의 단정밀도(float) 덧셈 병렬 처리
- 주로 이미지, 사운드, 비디오 처리에 활용
- gcc 등의 컴파일러에서 자동 추출을 시도하거나, 벡터 데이터 타입으로 명시적 사용 가능

### 1.9.3 추상화의 중요성 (Importance of Abstractions)

추상화(abstraction)는 컴퓨터 과학에서 가장 중요한 개념 중 하나다.

```
┌──────────────────────────────────────────────────────┐
│               Virtual Machine                        │
│  ┌──────────────────────────────────────────────┐   │
│  │            Operating System                  │   │
│  │  ┌────────┐  ┌──────────┐  ┌──────────────┐ │   │
│  │  │Processes│  │Virtual  │  │    Files     │ │   │
│  │  │        │  │ Memory  │  │              │ │   │
│  │  └────────┘  └──────────┘  └──────────────┘ │   │
│  └──────────────────────────────────────────────┘   │
│  ┌────────────────────────────────────────────────┐  │
│  │  Instruction Set Architecture (ISA)            │  │
│  │  ┌──────────┐  ┌────────────┐  ┌───────────┐  │  │
│  │  │Processor │  │Main Memory │  │I/O devices│  │  │
│  │  └──────────┘  └────────────┘  └───────────┘  │  │
│  └────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

| 추상화 | 추상화 대상 | 핵심 |
|-------|-----------|------|
| **ISA** (Instruction Set Architecture) | 실제 프로세서 하드웨어 | 기계어 프로그램은 명령어를 순차 실행하는 것처럼 동작. 실제로는 병렬 실행하지만 순차 모델 보장 |
| **파일** | I/O 디바이스 | 모든 I/O 장치를 바이트 시퀀스로 통일 |
| **가상 메모리** | 메인 메모리 + 디스크 | 프로세스마다 독립된 균일한 메모리 뷰 제공 |
| **프로세스** | CPU + 메모리 + I/O | 실행 중인 프로그램의 독점 사용 환상 |
| **가상 머신** | 전체 컴퓨터 (OS + 프로세서 + 프로그램) | 여러 OS를 단일 물리 머신에서 실행 가능 (VMware, VirtualBox 등) |

ISA의 강력함: 동일한 기계어를 전혀 다른 내부 구현을 가진 프로세서들이 실행할 수 있다. 이를 통해 비용과 성능의 스펙트럼에서 다양한 구현이 가능하다.
