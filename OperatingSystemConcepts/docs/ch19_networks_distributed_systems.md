# Chapter 19: Networks and Distributed Systems

## 개요

분산 시스템(Distributed System)은 메모리나 클록을 공유하지 않는 프로세서들의 집합이다. 각 노드(node)는 자신의 로컬 메모리를 가지며, 통신 네트워크를 통해 상호작용한다. 인터넷이 가장 대표적인 분산 시스템이다.

---

## 19.1 분산 시스템의 장점

### 자원 공유 (Resource Sharing)
- 사이트 A의 사용자가 사이트 B의 데이터베이스 조회 가능
- 원격 특수 하드웨어(GPU, 슈퍼컴퓨터) 접근
- 원격 파일·프린터 공유

### 계산 속도 향상 (Computation Speedup)
- 대규모 연산을 서브 연산으로 분할 → 여러 사이트에서 병렬 실행
- **부하 분산(Load Balancing)**: 과부하 사이트의 작업을 가벼운 사이트로 이전

### 신뢰성 (Reliability)
- 하나의 사이트가 실패해도 나머지 사이트가 계속 운영
- 충분한 하드웨어·데이터 중복성 → 부분 장애에도 서비스 지속

## 19.2 네트워크 구조 (Network Structure)

### 19.2.1 근거리 통신망 (LAN)

```
                  Router   WAN Link
┌──────┐   ┌──────┐  │  ┌──────┐
│ Host │───│  SW  │──┼──│Router│───(WAN)
└──────┘   └──────┘  │  └──────┘
┌──────┐      │
│ Host │─── Wireless
└──────┘    AP
```

- **범위**: 단일 건물 또는 인접 건물
- **특성**: 높은 속도, 낮은 오류율

| 기술 | 표준 | 속도 |
|------|------|------|
| Ethernet | IEEE 802.3 | 10Mbps ~ 100Gbps+ (광케이블) |
| WiFi | IEEE 802.11 | 11Mbps ~ 400Mbps+ |

- Ethernet: 중앙 컨트롤러 없는 **멀티 액세스 버스** → 새 호스트 추가 용이
- WiFi: 물리 케이블 없이 무선 송수신기로 참여

### 19.2.2 광역 통신망 (WAN)

- **범위**: 국가·대륙 규모
- **링크**: 전화선, 전용 회선, 광케이블, 마이크로웨이브, 위성
- 라우터(Router)가 트래픽 방향 결정·전달
- 백본(Backbone) WAN: 40Gbps ~ 100Gbps 광섬유

**ARPANET → Internet 진화**:
- 1968년 4개 사이트 → 수백만 컴퓨터 시스템의 전 세계 네트워크

## 19.3 통신 구조 (Communication Structure)

### 19.3.1 이름 해석 (Naming and Name Resolution)

원격 프로세스는 `<hostname, identifier>` 쌍으로 식별된다.

**DNS (Domain Name System)** 계층 구조:
```
eric.cs.yale.edu 해석 과정:

1. 로컬 호스트 → edu 네임서버 질의: "yale.edu 네임서버 주소는?"
2. edu 네임서버 → yale.edu 네임서버 주소 반환
3. 로컬 호스트 → yale.edu 네임서버 질의: "cs.yale.edu는?"
4. cs.yale.edu 네임서버 → eric의 IP 주소 반환 (예: 128.148.31.100)
```

- **캐싱**: 이미 해석한 IP를 로컬 캐시에 저장 → 속도 향상 (단, TTL로 갱신)
- **고가용성**: Primary + Secondary 네임서버로 단일 장애점(SPoF) 방지
- **도메인 위임**: 각 도메인이 자체 호스트 정보 관리 (Yale이 yale.edu 관리)

### 19.3.2 통신 프로토콜 (Communication Protocols)

**OSI 7계층 모델**:

```
┌─────────────────────────────────────────────────────┐
│  Layer 7: Application   │ 사용자 직접 상호작용        │
│  (HTTP, FTP, SMTP, DNS) │ 파일 전송, 원격 로그인     │
├─────────────────────────┼─────────────────────────┤
│  Layer 6: Presentation  │ 형식 변환, 문자 코드 변환  │
│                         │ 암호화/복호화              │
├─────────────────────────┼─────────────────────────┤
│  Layer 5: Session       │ 프로세스 간 세션 관리      │
│                         │ 연결 설정·유지·종료         │
├─────────────────────────┼─────────────────────────┤
│  Layer 4: Transport     │ 메시지 전송, 패킷 순서     │
│  (TCP, UDP)             │ 흐름 제어, 혼잡 제어       │
├─────────────────────────┼─────────────────────────┤
│  Layer 3: Network       │ 패킷 분할, 논리 주소       │
│  (IP)                   │ 라우팅, 라우터 동작 수준   │
├─────────────────────────┼─────────────────────────┤
│  Layer 2: Data Link     │ 프레임, 물리 주소          │
│                         │ 오류 감지·복구             │
├─────────────────────────┼─────────────────────────┤
│  Layer 1: Physical      │ 비트 스트림, 전기 신호     │
│                         │ 하드웨어 구현              │
└─────────────────────────────────────────────────────┘
```

**TCP/IP 스택과 OSI 매핑**:

```
OSI           TCP/IP
Application ─ HTTP, DNS, SMTP, FTP, Telnet
Presentation ─ (정의 없음)
Session      ─ (정의 없음)
Transport    ─ TCP, UDP
Network      ─ IP
Data Link    ─ (정의 없음, 물리 네트워크 위에서 실행)
Physical     ─ (정의 없음)
```

**OSI 메시지 구조** (캡슐화):
```
[data-link header][network header][transport header][session header][data][data-link trailer]
```
각 레이어는 자신의 헤더를 추가하고, 하위로 내려가며 캡슐화한다.

### 19.3.3 TCP/IP 예시 — Ethernet에서 패킷 전송

**ARP (Address Resolution Protocol)**:
```
IP 주소 → MAC 주소 변환 과정:

1. 송신자가 목적지 IP 포함한 ARP 패킷을 LAN 전체 브로드캐스트
2. 해당 IP의 호스트만 응답 → 자신의 MAC 주소 반환
3. 송신자가 IP-MAC 쌍을 ARP 캐시에 저장 (타임아웃으로 갱신)
```

**Ethernet 패킷 구조**:
```
┌──────────┬──────────────────┬──────────────────┬─────────┬─────────────┬──────────┐
│ preamble │ destination MAC  │  source MAC      │ length  │   data      │checksum  │
│  (7+1B)  │   (2 or 6 B)     │   (2 or 6 B)     │  (2B)   │  (0-1500B)  │  (4B)    │
└──────────┴──────────────────┴──────────────────┴─────────┴─────────────┴──────────┘
```

**라우팅**:
- 동일 LAN → ARP 캐시에서 MAC 찾아 직접 전송
- 다른 LAN → 라우터로 전송 → WAN을 통해 목적지 네트워크의 라우터 → 최종 호스트

### 19.3.4 전송 프로토콜: UDP와 TCP

#### UDP (User Datagram Protocol)

- **무연결(Connectionless)**: 연결 설정 없이 즉시 전송
- **비신뢰적**: 패킷 손실·순서 보장 없음
- 헤더: 4개 필드만 (source port, destination port, length, checksum)
- 응용 레이어가 오류 처리 담당

```
UDP 통신 예시 (패킷 손실)

client                    server
  │──── initial request ────▶│
  │                          │─── data (seq 1) ──▶
  │◀─── data (seq 1) ────────│
  │                          │─── data (seq 2) ──▶ (손실!)
  │◀─── data (seq 3) ────────│
  │◀─── data (seq 4) ────────│
  │                          │
  │ (seq 2 누락, 애플리케이션이 처리 결정)
```

#### TCP (Transmission Control Protocol)

- **연결 지향(Connection-Oriented)**: 3-way handshake로 연결 설정
- **신뢰적**: ACK, 시퀀스 번호, 재전송으로 순서·무손실 보장

```
TCP 주요 메커니즘:

1. ACK (확인 응답)
   송신 ──패킷──▶ 수신
   송신 ◀──ACK── 수신
   타이머 만료 시 재전송

2. 시퀀스 번호 (Sequence Number)
   - 수신측이 패킷을 올바른 순서로 재조합
   - 누락 패킷 탐지

3. 3-way Handshake
   client ──SYN──▶ server
   client ◀──SYN-ACK── server
   client ──ACK──▶ server
   (연결 확립)

4. 흐름 제어 (Flow Control)
   수신측이 ACK에 수신 가능 윈도우 크기 포함 → 송신 속도 조절

5. 혼잡 제어 (Congestion Control)
   패킷 손실(ACK 타임아웃) 발생 시 전송 속도 감소 → 공정성 유지
```

**포트 번호**: 동일 호스트에서 복수 서버 프로세스 구분

| 서비스 | 포트 |
|--------|------|
| FTP | 21 |
| SSH | 22 |
| SMTP | 25 |
| HTTP | 80 |
| HTTPS | 443 |

> 잘 알려진 포트 목록: `/etc/services` 참조

## 19.4 네트워크 및 분산 운영체제

### 19.4.1 네트워크 운영체제 (Network OS)

사용자가 원격 자원에 명시적으로 접근 — 현재 모든 범용 OS 해당

**원격 로그인 (Remote Login)**:
```bash
ssh kristen.cs.yale.edu
# → 암호화 소켓 연결 수립 → 원격 쉘 세션
```

**원격 파일 전송 (Remote File Transfer)**:
```bash
sftp owen@kzoo.edu
# 명령어: get, put, ls/dir, cd
```

**클라우드 스토리지**: Dropbox, Google Drive — FTP/SFTP와 유사하나 GUI 기반

**한계**: 사용자가 **패러다임 전환** 필요 (로컬 명령 vs. 원격 명령 vs. 클라우드 UI)

### 19.4.2 분산 운영체제 (Distributed OS)

원격 자원을 로컬 자원처럼 투명하게 접근 — 사용자가 위치를 의식하지 않음

#### 데이터 마이그레이션 (Data Migration)

| 방식 | 설명 | 단점 |
|------|------|------|
| 전체 파일 전송 | 파일 전체를 사이트 A로 복사 | 대용량 파일에서 비효율 (Andrew FS, 실패 사례) |
| 블록 전송 | 필요한 블록만 전송 (수요 페이징 방식) | 구현 복잡도 ↑ |

현대 분산 시스템은 블록 전송 방식 채택

#### 계산 마이그레이션 (Computation Migration)

- 데이터가 있는 사이트에서 연산을 실행 → 결과만 반환
- 구현 방법:
  1. **RPC(Remote Procedure Call)**: 프로세스 P가 사이트 A의 미리 정의된 프로시저 호출 → 결과 반환
  2. **메시지 패싱**: 사이트 A에서 새 프로세스 Q 생성 → Q가 작업 수행 → 결과 메시지 전송

```
판단 기준:
데이터 전송 시간 > 원격 실행 시간
    → 계산 마이그레이션이 유리
```

#### 프로세스 마이그레이션 (Process Migration)

프로세스 전체(또는 일부)를 다른 사이트로 이동하여 실행:

| 이유 | 설명 |
|------|------|
| 부하 분산 | 사이트 간 작업 부하 균등화 |
| 계산 속도 향상 | 서브프로세스 병렬 실행 |
| 하드웨어 선호 | 특수 프로세서(GPU 등) 활용 |
| 소프트웨어 선호 | 특정 사이트에만 있는 소프트웨어 의존 |
| 데이터 접근 | 대용량 DB 서버 근처에서 실행 효율 |

- **투명한 마이그레이션**: 사용자 개입 없음 (부하 분산, 계산 속도 향상에 활용)
- **명시적 마이그레이션**: 사용자 지정 (하드웨어/소프트웨어 선호 충족)

## 19.5 분산 시스템 설계 이슈 (Design Issues)

### 19.5.1 견고성 (Robustness)

#### 장애 유형
- 링크 장애, 호스트 장애, 사이트 장애, 메시지 손실

#### 장애 감지 (Failure Detection)

**Heartbeat 절차**:
```
사이트 A <──I-am-up(주기적)──> 사이트 B

사이트 A가 메시지 미수신 시:
1. 타임아웃 대기 후 Are-you-up? 전송
2. 다른 경로로 Are-you-up? 전송:
   - 응답 있음 → 직접 링크 장애
   - 응답 없음 → 사이트 B 장애 or 모든 경로 장애
   (메모리 공유 없으므로 정확한 원인 구분 불가)
```

#### 재구성 (Reconfiguration)

- 링크 장애 감지 → 라우팅 테이블 브로드캐스트 갱신
- 사이트 장애 감지 → 코디네이터 선출(단, 실제 장애 vs. 네트워크 분할 구분 필요)
  - 네트워크 파티션 → 두 코디네이터 → 상호배제 충돌 위험

#### 장애 복구 (Recovery from Failure)

- 링크 복구 → 양쪽 사이트 알림 (heartbeat 재개)
- 사이트 복구 → 다른 사이트 알림 → 라우팅 테이블, 미전달 메시지, 트랜잭션 로그 업데이트

### 19.5.2 투명성 (Transparency)

- **목표**: 로컬·원격 자원을 동일하게 접근 (위치 노출 없음)
- **사용자 이동성**: 어느 머신에서든 로그인 시 홈 환경 자동 제공
  - LDAP: 원격·이동 사용자 인증
  - 데스크탑 가상화: 원격 데스크탑 세션 접근

### 19.5.3 확장성 (Scalability)

- **목표**: 부하 증가에 우아하게 대응 (성능 저하 최소화, 포화 지연)
- 과부하 컴포넌트 = 장애 컴포넌트처럼 동작 → 확장성과 장애 허용성 연관

**스토리지 효율화**:
- **압축(Compression)**: 무손실 압축으로 파일 크기 감소 (zip 등)
- **중복 제거(Deduplication)**: 시스템 전체에서 동일 데이터의 단일 인스턴스만 저장
  - 파일 레벨 또는 블록 레벨 적용 가능
  - 클라우드 스토리지에서 대규모 절약 효과

## 19.6 분산 파일 시스템 (Distributed File System, DFS)

### 기본 개념

| 용어 | 정의 |
|------|------|
| Service | 특정 기능을 제공하는 소프트웨어 엔티티 |
| Server | 단일 머신에서 실행되는 서비스 소프트웨어 |
| Client | 서비스를 호출하는 프로세스 |
| DFS | 클라이언트·서버·스토리지가 분산된 파일 시스템 |

이상적 DFS: 사용자에게 단일 중앙화 파일 시스템처럼 보임

### 19.6.1 클라이언트-서버 모델

```
┌────────┐    파일 요청    ┌────────┐
│ client │ ─────────────▶ │ server │
│        │ ◀───────────── │        │
│        │    파일 전달    │  disk  │
└────────┘               └────────┘
```

**NFS (Network File System)**:
- Sun Microsystems 개발, 오픈 프로토콜
- **무상태(Stateless)**: 서버가 클라이언트 상태 추적 안 함
- **멱등성(Idempotent)**: 동일 연산 반복 실행 가능 (서버 crash 후 재발행 안전)
- 목표: **빠른 장애 복구** — 클라이언트가 상태(파일 포인터 등) 유지 → 서버 재시작 후 재발행

**OpenAFS (Andrew File System)**:
- Carnegie Mellon University 개발
- **확장성** 목표: 서버가 최대한 많은 클라이언트 지원
- **전체 파일 캐싱**: 파일 요청 시 전체 내용을 클라이언트 로컬에 다운로드 → 수정 후 close 시 서버에 업로드
- NFS보다 서버 통신 횟수 대폭 감소

**단점**: 단일 서버 = 단일 장애점(SPoF), 성능 병목

### 19.6.2 클러스터 기반 DFS (Cluster-Based DFS)

```
┌────────┐                    ┌───────────────┐
│ client │──┐                 │ Metadata      │
└────────┘  │   ┌─────────┐  │ Server        │
            ├──▶│ network │──▶│(디렉토리+청크맵)│
┌────────┐  │   └─────────┘  └───────────────┘
│ client │──┘         │
└────────┘            │      ┌───────────────┐
                      ├─────▶│ Data Server 1 │ (chunk A, chunk B)
                      │      └───────────────┘
                      │      ┌───────────────┐
                      ├─────▶│ Data Server 2 │ (chunk A복제, chunk C)
                      │      └───────────────┘
                      │      ┌───────────────┐
                      └─────▶│ Data Server 3 │ (chunk B복제, chunk C복제)
                             └───────────────┘
```

**동작 흐름**:
1. 클라이언트 → 메타데이터 서버: 파일 접근 요청
2. 메타데이터 서버 → 클라이언트: 청크 보유 데이터 서버 목록 반환
3. 클라이언트 → 데이터 서버(들): 직접 청크 읽기/쓰기 (병렬 가능)

**청크 복제**: 보통 3벌 복제 → 장애 허용 + 읽기 성능 향상

**GFS (Google File System, 2003)**:
설계 4대 관찰:
1. 하드웨어 장애는 **예외가 아니라 정상** — 상시 장애 대응
2. 저장 파일은 **매우 크다** (GB ~ TB)
3. 파일 수정은 대부분 **끝에 추가(Append)** — 덮어쓰기 드묾
4. **API 재설계**가 시스템 유연성 향상

GFS → **MapReduce** (병렬 대규모 연산 레이어) 추가 개발

**Hadoop Distributed File System (HDFS)**:
- GFS/MapReduce에서 파생 → 오픈소스 구현
- **Hadoop 생태계**: HDFS + MapReduce + 그 위의 모듈 스택
- 빅데이터(소셜미디어 분석, 과학 데이터 포인트 트렌드 분석) 처리 목적

## 19.7 DFS 이름 지정과 투명성 (DFS Naming and Transparency)

### 위치 투명성 vs. 위치 독립성

| 개념 | 정의 | 강도 |
|------|------|------|
| 위치 투명성 (Location Transparency) | 파일 이름이 물리적 위치 정보를 노출하지 않음 | 낮음 (정적 매핑) |
| 위치 독립성 (Location Independence) | 파일의 물리적 위치가 변경되어도 이름은 변경 불필요 | 높음 (동적 매핑) |

```
위치 독립성 ⊃ 위치 투명성 (더 강한 속성)
```

**주요 구현 사례**:
- **대부분의 현대 DFS**: 정적 위치 투명 매핑 제공
- **OpenAFS**: 위치 독립성 + 파일 이동 지원
- **HDFS**: POSIX 표준 없이 위치 독립성 구현 → 자체 자동 튜닝
- **Amazon S3**: 블록 단위 on-demand 스토리지, 내부 위치 자동 관리

**위치 독립성의 이점**:
- 파일 이름이 **내용**을 나타냄 (위치가 아님)
- 스토리지 공간 자체를 공유하는 개념 → 전체 저장소가 단일 가상 자원처럼 동작
- 네이밍 계층과 스토리지 장치 계층의 분리 → 아키텍처 유연성

### 명명 체계 (Naming Schemes)

| 접근법 | 예시 | 투명성 |
|--------|------|--------|
| host:local-name | `ystem.yale.edu:/files/doc.txt`, URL | 없음 |
| 마운트 포인트 | NFS 마운트 (`/remote/files/`) | 정적 위치 투명성 |
| 전역 이름 공간 | OpenAFS (`/afs/yale.edu/files/doc.txt`) | 위치 독립성 |

### 디스크리스 클라이언트 (Diskless Client)

- 클라이언트가 디스크 없음 → 부팅 시 ROM의 특수 프로토콜로 커널 네트워크 로드
- 장점: 낮은 비용, OS 업그레이드 시 서버만 수정
- 단점: 부팅 프로토콜 복잡, 로컬 디스크 대비 성능 손실

## 19.8 요약

```
분산 시스템의 핵심 개념 맵

네트워크 구조
├── LAN (Ethernet/WiFi, 고속, 저오류)
└── WAN (인터넷, 라우터, 광케이블 백본)

통신 구조
├── 이름 해석: DNS 계층 (edu→yale.edu→cs.yale.edu→host)
├── OSI 7계층 → TCP/IP 4계층으로 실용화
├── ARP: IP → MAC 변환 (LAN 내)
├── UDP: 비신뢰, 고속, connectionless
└── TCP: 신뢰, ACK+시퀀스+혼잡제어, connection-oriented

네트워크 OS vs 분산 OS
├── 네트워크 OS: SSH, FTP/SFTP, Cloud Storage (명시적 원격 접근)
└── 분산 OS: 투명한 접근 (데이터/계산/프로세스 마이그레이션)

분산 설계 이슈
├── 견고성: heartbeat 장애 감지, 재구성, 복구
├── 투명성: 위치 노출 없음, 사용자 이동성 (LDAP)
└── 확장성: 우아한 성능 저하, 압축·중복제거

분산 파일 시스템 (DFS)
├── 클라이언트-서버: NFS(무상태), OpenAFS(전체 파일 캐싱)
└── 클러스터 기반: GFS/HDFS (메타데이터 서버 + 데이터 서버 + 청크 3중 복제)
```

| 시스템 | 유형 | 핵심 특성 |
|--------|------|-----------|
| NFS | 클라이언트-서버 | 무상태, 멱등성, 빠른 장애 복구 |
| OpenAFS | 클라이언트-서버 | 확장성, 전체 파일 캐싱, 위치 독립성 |
| GFS | 클러스터 | 대용량 파일, Append-only 최적화, MapReduce |
| HDFS | 클러스터 | GFS 오픈소스 구현, Hadoop 생태계, 빅데이터 |
| Amazon S3 | 클라우드 | On-demand 블록, 위치 자동 관리 |
