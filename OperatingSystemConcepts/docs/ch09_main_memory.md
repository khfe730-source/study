## 1. 배경

### 1.1 기본 하드웨어

CPU가 직접 접근 가능한 저장소는 **레지스터(register)**와 **메인 메모리(main memory)** 뿐이다. 디스크 주소를 인자로 받는 기계 명령어는 존재하지 않으므로, 실행 중인 명령어와 데이터는 반드시 메모리 또는 레지스터에 있어야 한다.

**레지스터 vs 메모리 접근 비용**
- 레지스터: CPU 클록 1사이클 이내
- 메모리: 메모리 버스 트랜잭션 → 수 클록 사이클 → CPU 스톨(stall) 발생
- 해결책: CPU와 메모리 사이에 **캐시(cache)** 삽입

**메모리 보호 (base/limit 레지스터)**

```
물리 메모리 레이아웃:

  0         base     base+limit
  ├─────────┼────────┼──────────────▶
  │   OS    │ 프로세스 │   다른 프로세스
  └─────────┴────────┴──────────────
                │
               base register = 300040
              limit register =  120900
              → 합법 주소: 300040 ~ 420939

  CPU 생성 주소 → HW 비교:
    address < base      → trap (OS: 접근 오류)
    address >= base+limit → trap (OS: 접근 오류)
    정상 범위            → 메모리 접근 허용
```

base/limit 레지스터는 커널 모드의 특권 명령어(privileged instruction)로만 적재 가능.

---

### 1.2 주소 바인딩 (Address Binding)

소스 코드의 심볼 주소(symbolic address)가 실제 물리 주소에 연결되는 시점:

```
소스 프로그램
    │ 컴파일러(compiler)
    ▼
  오브젝트 파일 (재배치 가능 주소, relocatable address)
    │ 링커(linker)
    ▼
  실행 파일 (절대 주소 또는 재배치 코드)
    │ 로더(loader)
    ▼
  메모리 내 프로그램 (물리 주소)
```

| 바인딩 시점 | 설명 | 특징 |
|------------|------|------|
| **컴파일 시간 (Compile time)** | 물리 주소가 고정됨 | 주소 변경 시 재컴파일 필요 |
| **적재 시간 (Load time)** | 컴파일러가 재배치 코드 생성, 로드 시 절대 주소 확정 | 주소 변경 시 재적재 필요 |
| **실행 시간 (Execution time)** | 실행 중 이동 가능, HW 지원 필요 | 현대 OS 대부분이 채택 |

---

### 1.3 논리 주소 vs 물리 주소 (Logical vs Physical Address Space)

- **논리 주소(logical address)**: CPU가 생성하는 주소 (= **가상 주소(virtual address)**)
- **물리 주소(physical address)**: 메모리 장치(MAR)에 실제로 전달되는 주소

컴파일/적재 시간 바인딩: 논리 = 물리  
실행 시간 바인딩: 논리 ≠ 물리 → **MMU(Memory Management Unit)**가 변환

```
CPU ──논리주소──▶ [MMU] ──물리주소──▶ 물리 메모리

단순 재배치 레지스터 방식:
  논리 주소 346 + relocation register(14000) = 물리 주소 14346
```

논리 주소 공간(0 ~ max)과 물리 주소 공간(R ~ R+max)은 분리되어 있으며, 이 분리가 올바른 메모리 관리의 핵심이다.

---

### 1.4 동적 적재 (Dynamic Loading)

루틴(routine)이 호출될 때까지 디스크에 재배치 가능 형식(relocatable load format)으로 유지. 호출 시점에 재배치 링킹 로더(relocatable linking loader)가 해당 루틴을 메모리에 적재하고 주소 테이블 갱신.

- OS 특별 지원 불필요 — 프로그래머 책임
- 드물게 호출되는 오류 처리 루틴(error routine)처럼 대용량이지만 사용 빈도가 낮은 코드에 유용

---

### 1.5 동적 링킹과 공유 라이브러리 (DLL)

**정적 링킹(static linking)**: 라이브러리를 실행 파일에 직접 포함 → 파일 크기 증가, 메모리 낭비

**동적 링킹(dynamic linking)**: 실행 시간까지 라이브러리 연결 지연  
- DLL(Dynamically Linked Libraries) = **공유 라이브러리(shared library)**
- 물리 메모리에 라이브러리 단 하나의 인스턴스만 존재 → 여러 프로세스가 공유
- 버전 정보로 호환성 관리 (마이너 변경 → 동일 버전 번호, 메이저 변경 → 번호 증가)
- OS 지원 필요: 프로세스 간 메모리 보호 환경에서 동일 메모리 접근 허용

## 2. 연속 메모리 할당

### 2.1 메모리 보호 (relocation + limit 레지스터)

```
논리 주소 → limit 레지스터 범위 검사 → relocation 레지스터 더하기 → 물리 주소

CPU ──논리주소──▶ [< limit?] ──No──▶ 트랩(trap): addressing error
                       │ Yes
                       ▼
                  [+relocation] ──▶ 물리 주소 → 메모리
```

컨텍스트 스위치 시 디스패처(dispatcher)가 relocation/limit 레지스터를 새 프로세스 값으로 교체.

---

### 2.2 메모리 할당 전략

**가변 파티션(variable-partition) 방식**: 가용 메모리 공간 = **홀(hole)** 집합

프로세스 도착 시 홀에서 공간 할당, 종료 시 해제 후 인접 홀과 합병(merge).

| 전략 | 방법 | 특성 |
|------|------|------|
| **First Fit (최초 적합)** | 충분히 큰 첫 번째 홀 선택 | 가장 빠름 |
| **Best Fit (최적 적합)** | 충분히 큰 홀 중 가장 작은 것 선택 | 가장 작은 잔여 홀 생성 |
| **Worst Fit (최악 적합)** | 가장 큰 홀 선택 | 가장 큰 잔여 홀 생성 |

> 시뮬레이션 결과: First Fit과 Best Fit이 Worst Fit보다 우수. First Fit vs Best Fit은 상황에 따라 다르지만 First Fit이 일반적으로 더 빠름.

---

### 2.3 단편화 (Fragmentation)

**외부 단편화(External Fragmentation)**  
- 총 가용 메모리는 충분하지만 **비연속**으로 흩어져 있어 할당 불가
- **50% 규칙**: N개 블록 할당 시 약 0.5N 블록이 단편화로 손실 → 메모리 1/3이 사용 불가

**내부 단편화(Internal Fragmentation)**  
- 할당된 메모리가 실제 요구보다 약간 큼 (블록 단위 할당 때문)
- 파티션 내부의 사용되지 않는 공간

```
외부 단편화 예시:

  [P1: 50KB][hole: 10KB][P2: 40KB][hole: 15KB][P3: 30KB]
                                                         ^
  25KB 프로세스 요청 → 총 가용 25KB지만 연속 홀 없음 → 실패
```

**해결책**:
- **압축(Compaction)**: 모든 프로세스를 한쪽으로 이동 → 하나의 큰 홀 생성. 실행 시간 바인딩에서만 가능, 비용 큼
- **페이징(Paging)**: 비연속 물리 주소 공간 허용 → 외부 단편화 원천 제거 (다음 절)

## 3. 페이징

### 3.1 기본 방법

물리 메모리를 고정 크기 **프레임(frame)**, 논리 메모리를 동일 크기 **페이지(page)**로 분할. 외부 단편화 없음, 내부 단편화만 발생 (평균 페이지의 절반).

**주소 변환 구조**

```
논리 주소: [ 페이지 번호 p | 페이지 오프셋 d ]
           |←── m-n bits ──→|←── n bits ──→|
               (페이지 테이블 인덱스)  (프레임 내 변위)

변환 과정:
  p → 페이지 테이블 조회 → 프레임 번호 f
  물리 주소 = f × (페이지 크기) + d
```

**구체적 예시** (페이지 크기 4B, 물리 메모리 32B = 8프레임)

```
논리 주소 13 → page 3, offset 1
페이지 테이블: page 3 → frame 2
물리 주소 = 2 × 4 + 1 = 9

논리 메모리    페이지 테이블    물리 메모리
┌──────────┐  ┌─────┬──────┐  ┌──────────┐
│ page 0   │  │  0  │  5   │  │ frame 0  │
├──────────┤  ├─────┼──────┤  ├──────────┤
│ page 1   │  │  1  │  6   │  │ frame 1  │
├──────────┤  ├─────┼──────┤  ├──────────┤
│ page 2   │  │  2  │  1   │  │ page 1   │← frame 2
├──────────┤  ├─────┼──────┤  ├──────────┤
│ page 3   │  │  3  │  2   │  │ page 3   │← frame 3
└──────────┘  └─────┴──────┘  ├──────────┤
                               │ frame 4  │
                               ├──────────┤
                               │ page 0   │← frame 5
                               ├──────────┤
                               │ page 2   │← frame 6
                               └──────────┘
```

**페이지 크기 트레이드오프**
- 작은 페이지: 내부 단편화 감소, 페이지 테이블 항목 수 증가
- 큰 페이지: 페이지 테이블 크기 감소, 디스크 I/O 효율 증가
- 현대 시스템: 4KB~8KB (일부 시스템은 2MB, 1GB 휴지 페이지(huge page) 지원)

---

### 3.2 하드웨어 지원

**PTBR (Page-Table Base Register)**: 현재 프로세스의 페이지 테이블 위치 저장. 컨텍스트 스위치 시 이 레지스터만 교체하면 됨.

#### TLB (Translation Look-Aside Buffer)

메모리에 페이지 테이블을 두면 모든 메모리 접근이 2회 필요(페이지 테이블 1회 + 실제 데이터 1회). TLB는 이를 해결하는 **연관 캐시(associative cache)**.

```
TLB를 사용한 주소 변환:

CPU → 논리주소 (p, d)
         │
         ▼
   ┌─────────────┐
   │  TLB 검색   │ (병렬 비교, 파이프라인 내 수행)
   └──────┬──────┘
     TLB  │  TLB
     hit  │  miss
     │    │
     │    ▼
     │  페이지 테이블 (메인 메모리) → 프레임 번호 f
     │  + TLB 갱신
     │
     ▼
  물리 주소 (f, d) → 메모리 접근
```

**유효 접근 시간(EAT, Effective Access Time)** 계산

```
메모리 접근 시간 = 10ns, 히트율(hit ratio) = α

EAT = α × 10 + (1-α) × (10 + 10)   [페이지 테이블 접근 + 실제 접근]
    = 10 + 10(1-α)

히트율 80%: EAT = 0.80×10 + 0.20×20 = 12ns  (+20% 오버헤드)
히트율 99%: EAT = 0.99×10 + 0.01×20 = 10.1ns (+1% 오버헤드)
```

**Intel Core i7 TLB 구조**
- L1 명령어 TLB: 128 엔트리
- L1 데이터 TLB: 64 엔트리
- L2 TLB: 512 엔트리 (L1 미스 시 6사이클)
- L2 미스: 수백 사이클의 페이지 테이블 워크(page-table walk) 필요

**ASID (Address-Space Identifier)**: TLB 항목에 ASID를 저장해 여러 프로세스의 TLB 항목을 동시 유지. ASID 미지원 시 컨텍스트 스위치마다 TLB 전체 플러시(flush) 필요.

**Wired 항목**: 커널 핵심 코드의 TLB 항목은 제거 불가로 고정.

---

### 3.3 보호 (Protection)

페이지 테이블 각 항목에 보호 비트(protection bit) 포함:

| 비트 | 의미 |
|------|------|
| 읽기-쓰기 / 읽기 전용 | 쓰기 시도 → 하드웨어 트랩 |
| 실행 전용(execute-only) | 코드 페이지 보호 |
| **유효-무효 비트 (valid-invalid bit)** | 프로세스 주소 공간 여부 |

**PTLR (Page-Table Length Register)**: 페이지 테이블 크기를 저장해 모든 논리 주소를 유효 범위와 비교.

```
valid-invalid 비트 예시 (14비트 주소, 2KB 페이지):
  프로그램 실제 사용: 0~10468

  페이지 0~5: valid (v)
  페이지 6~7: invalid (i) → 접근 시 트랩

  주의: 페이지 5는 valid이지만 10469~12287은 미사용
  → 내부 단편화의 결과
```

---

### 3.4 공유 페이지 (Shared Pages)

**재진입 코드(reentrant code = non-self-modifying code)**: 실행 중 코드가 변경되지 않으므로 여러 프로세스가 동일한 물리 프레임을 공유 가능.

```
libc 공유 예시 (libc = 2MB):

  정적 링킹: 프로세스 40개 × 2MB = 80MB 필요
  공유 페이지:  단 2MB만 물리 메모리에 적재
              각 프로세스의 페이지 테이블이 동일 프레임을 가리킴

  P1 페이지 테이블   물리 메모리       P2 페이지 테이블
  ┌───────────┐    ┌──────────┐    ┌───────────┐
  │libc page1 │→4  │ frame 4  │ ←│ libc page1│
  │libc page2 │→1  │ frame 1  │ ←│ libc page2│
  │libc page3 │→6  │ frame 6  │ ←│ libc page3│
  │  data     │→3  │ frame 3  │   │  data     │→7 (별도)
  └───────────┘    └──────────┘    └───────────┘
```

OS가 읽기 전용 강제 적용 필수. 공유 라이브러리(DLL)가 대표적 구현 사례.

## 4. 페이지 테이블 구조

### 4.1 계층적 페이징 (Hierarchical Paging)

32비트 주소 공간 + 4KB 페이지 → 페이지 테이블 항목 2²⁰개 = 4MB (프로세스당). 메모리 낭비 심각.

**해결책**: 페이지 테이블 자체를 다시 페이지(page the page table) → 다단계 페이지 테이블

**2단계 페이징 (32비트 시스템)**

```
논리 주소: [ p1 | p2 | d ]
           [10비트|10비트|12비트]

변환 과정:
  CR3 → 외부 페이지 테이블[p1] → 내부 페이지 테이블[p2] → 프레임 번호 → +d = 물리 주소

  외부 페이지 테이블(outer page table)
  ┌────┐
  │    │ p1=0 → 내부 페이지 테이블 A
  ├────┤
  │    │ p1=1 → 내부 페이지 테이블 B
  ├────┤
    ...

  내부 페이지 테이블 A
  ┌────┐
  │ f0 │ p2=0 → 프레임 f0
  ├────┤
  │ f1 │ p2=1 → 프레임 f1
    ...
```

**64비트 주소 공간의 문제**: 2단계로는 outer 페이지 테이블이 2⁴² 항목 = 244B. 3단계는 outer outer가 2³⁴B(16GB). 7단계까지 필요할 수 있어 접근 비용 급증 → 64비트에서는 계층적 페이징이 부적합.

---

### 4.2 해시 페이지 테이블 (Hashed Page Tables)

32비트 초과 주소 공간 처리. 가상 페이지 번호를 해시 함수에 통과시켜 해시 테이블 탐색.

```
가상 주소 (p, d)
    │
    ▼ hash(p)
  해시 테이블[i]
    │
    ▼
  연결 리스트 (충돌 해결):
  [VPN | 프레임 번호 | next ptr] → [VPN | 프레임 | next] → ...
    │
  p와 VPN 일치 → 프레임 번호 사용 → 물리 주소 (프레임, d)
```

**클러스터 페이지 테이블(Clustered Page Table)**: 64비트용 변형. 해시 테이블 각 항목이 단일 페이지 대신 여러 페이지(예: 16개)의 매핑 저장. 듬성듬성한(sparse) 주소 공간에 효율적.

---

### 4.3 역 페이지 테이블 (Inverted Page Table)

기존: 프로세스마다 페이지 테이블 → 가상 페이지 수만큼 항목  
역 페이지 테이블: **물리 프레임당 하나의 항목** → 시스템 전체에 단 하나의 페이지 테이블

```
구조: 인덱스 = 프레임 번호 i
  항목 = <pid, page-number>

탐색:
  가상 주소 (pid, p, d) 입력
  역 페이지 테이블에서 (pid, p) 매칭 항목 i 탐색
  물리 주소 = (i, d)

  ┌─────┬──────────────┐
  │ i=0 │ (PID_A, p3)  │
  ├─────┼──────────────┤
  │ i=1 │ (PID_B, p0)  │
  ├─────┼──────────────┤
  │ i=2 │ (PID_A, p1)  │
  └─────┴──────────────┘
```

**장점**: 물리 메모리 크기에 비례하는 단 하나의 테이블 → 메모리 절약  
**단점**:
- 가상 주소로 물리 순서 정렬된 테이블 탐색 → 전체 탐색 필요 → 해시 테이블 병용으로 해결
- **공유 메모리 지원 어려움**: 하나의 물리 프레임에 하나의 가상 주소 매핑만 가능

사용 예: IBM PowerPC, 64비트 UltraSPARC

---

### 4.4 Oracle SPARC Solaris

64비트 OS의 가상 메모리 최적화:
- 커널용/사용자 프로세스용 **두 개의 해시 테이블**
- 각 해시 항목이 단일 페이지 아닌 **연속 가상 메모리 영역(base + span)** 표현 → 효율적
- **TTEs (Translation Table Entries)**: CPU TLB에 캐시
- **TSB (Translation Storage Buffer)**: 최근 접근 페이지의 TTE 인메모리 캐시
- 변환 순서: TLB → TSB → 해시 테이블(커널 인터럽트) → TLB 갱신

## 5. 스와핑 (Swapping)

프로세스 또는 페이지를 메모리에서 **백킹 스토어(backing store)**(고속 보조 저장장치)로 임시 이동 → 다중 프로그래밍 정도(degree of multiprogramming) 향상.

### 5.1 표준 스와핑 (Standard Swapping)

프로세스 전체를 메모리 ↔ 백킹 스토어 이동. 멀티스레드 프로세스는 모든 스레드 데이터 구조도 함께 스왑.

```
메인 메모리:
  ┌────────┐
  │   OS   │
  ├────────┤ ◀─swap out── 프로세스 P2
  │   P1   │ ──swap in──▶ 프로세스 P2
  └────────┘

백킹 스토어: [P2 이미지]
```

단점: 전체 프로세스 이동 시간 크다 → 현대 OS(Solaris 제외)에서는 사실상 미사용.

---

### 5.2 페이징과 함께하는 스와핑

현대 Linux, Windows가 채택. 프로세스 전체가 아닌 **페이지 단위** 스왑.

- **Page out**: 메모리 → 백킹 스토어
- **Page in**: 백킹 스토어 → 메모리

→ Chapter 10 가상 메모리와 결합해 효과 극대화

---

### 5.3 모바일 시스템에서의 스와핑

모바일 기기는 스와핑을 지원하지 않는다.

**이유**:
- 플래시 메모리(flash memory)의 쓰기 횟수 제한
- 플래시와 메모리 간 처리량 낮음
- 저장 공간 부족

**iOS 대처 방식**:
- 가용 메모리 부족 시 앱에 자발적 메모리 해제 요청
- **읽기 전용 데이터(코드)**: 메모리에서 제거 후 플래시에서 재적재
- **수정된 데이터(스택)**: 절대 제거하지 않음
- 메모리 해제 실패 앱 강제 종료

**Android 대처 방식**:
- 메모리 부족 시 프로세스 종료
- 종료 전 **앱 상태를 플래시에 저장** → 빠른 재시작 가능

---

## 6. Intel 32/64비트 아키텍처 예시

### 6.1 IA-32 아키텍처

**세그먼테이션(segmentation) + 페이징** 두 단계 변환:

```
CPU → 논리 주소 → [세그멘테이션 유닛] → 선형 주소 → [페이징 유닛] → 물리 주소
```

#### IA-32 세그멘테이션

- 세그먼트 최대 4GB, 프로세스당 최대 16K 세그먼트
- **LDT(Local Descriptor Table)**: 프로세스 전용 세그먼트 (최대 8K)
- **GDT(Global Descriptor Table)**: 공유 세그먼트 (최대 8K)
- 각 항목: 8바이트 세그먼트 디스크립터 (base + limit 포함)

```
논리 주소 형식: (selector, offset)
  selector [16비트]: s[13비트] | g[1비트] | p[2비트]
                    (세그먼트#)  (GDT/LDT)  (보호)
  offset   [32비트]: 세그먼트 내 바이트 위치

변환: selector → LDT/GDT → base + limit 로드
     offset < limit? → 아니면 폴트
     선형 주소 = base + offset
```

#### IA-32 페이징

4KB 페이지 기준, 2단계 페이징:

```
선형 주소: [ p1 | p2 | d ]
           [10비트|10비트|12비트]

CR3 → 페이지 디렉토리[p1] → 페이지 테이블[p2] → 프레임 → +d = 물리 주소

4MB 페이지: 페이지 디렉토리 항목의 Page Size 플래그 설정 시
  페이지 디렉토리 → 4MB 프레임 직접 (페이지 테이블 우회)
  오프셋 = 하위 22비트
```

#### PAE (Page Address Extension)

- IA-32에서 4GB 초과 물리 메모리 지원
- 2단계 → **3단계** 페이징 (page directory pointer table 추가)
- 페이지 테이블/디렉토리 항목: 32비트 → **64비트** (base 주소 24비트)
- 지원 물리 주소 공간: 36비트 → **64GB**

```
PAE 구조 (4KB 페이지):
  [ PDPT 인덱스| PD 인덱스 | PT 인덱스 | 오프셋 ]
  [   2비트    |  9비트   |   9비트   |  12비트 ]

CR3 → PDPT[2비트] → PD → PT → 프레임 → 물리 주소
```

---

### 6.2 x86-64 아키텍처

- AMD가 설계, Intel이 채택 (AMD64 / Intel 64 → 통칭 x86-64)
- **48비트 가상 주소**, **52비트 물리 주소** (PAE 연장)
- 4단계 페이징 계층:

```
가상 주소 [63~0]:
  [사용 안 함|PML4 인덱스|PDPT 인덱스|PD 인덱스|PT 인덱스|오프셋]
     16비트     9비트       9비트      9비트     9비트    12비트

CR3 → PML4[9비트] → PDPT[9비트] → PD[9비트] → PT[9비트] → 프레임 → +오프셋

지원 페이지 크기: 4KB / 2MB / 1GB
```

---

## 7. ARMv8 아키텍처 예시

ARM은 칩을 설계하고 제조사에 라이센스. 전 세계 1000억 개 이상 생산. iOS, Android 대부분이 ARM 기반.

**3가지 변환 단위(translation granule)**:

| 단위 크기 | 페이지 크기 | 리전 크기 |
|----------|-----------|---------|
| 4KB | 4KB | 2MB, 1GB |
| 16KB | 16KB | 32MB |
| 64KB | 64KB | 512MB |

**4KB 단위 4단계 계층 페이징**:

```
가상 주소 (48비트 사용, 64비트 아키텍처):
  [미사용|L0 인덱스|L1 인덱스|L2 인덱스|L3 인덱스|오프셋]
   16비트   9비트     9비트     9비트     9비트    12비트

TTBR(Translation Table Base Register) → L0 테이블 → L1 → L2 → L3 → 4KB 페이지
                                                          ↘ 2MB 리전 (L2 직접)
                                               ↘ 1GB 리전 (L1 직접)
```

**2단계 TLB 구조**:
```
주소 변환 순서:
  1. 마이크로 TLB (micro TLB) 검색
     - 명령어용 TLB + 데이터용 TLB (분리)
     - ASID 지원
  2. 마이크로 TLB 미스 → 메인 TLB (main TLB) 검색
  3. 메인 TLB 미스 → 하드웨어 페이지 테이블 워크(page table walk) 수행
```

## 메모리 관리 기법 비교

| 기법 | 외부 단편화 | 내부 단편화 | 코드 공유 | 복잡도 |
|------|-----------|-----------|----------|-------|
| 연속 할당 (first/best fit) | 심각 (50% 규칙) | 없음 | 어려움 | 낮음 |
| 페이징 | 없음 | 평균 1/2 페이지 | 용이 (reentrant) | 중간 |
| 세그멘테이션 | 있음 | 없음 | 세그먼트 단위 | 중간 |
| 세그멘테이션+페이징 (IA-32) | 없음 | 있음 | 용이 | 높음 |

## 핵심 정리

- **메모리 보호**: base/limit 레지스터로 프로세스 간 격리, 커널 모드만 변경 가능
- **주소 바인딩**: 컴파일/적재/실행 시간 중 현대 OS는 **실행 시간** 바인딩 채택
- **MMU**: 논리→물리 주소 변환 전담 하드웨어; 재배치 레지스터 기반
- **페이징**: 프레임(물리)/페이지(논리) 분할, 외부 단편화 제거, 페이지 테이블로 매핑
- **TLB**: 페이지 테이블 캐시; 히트율 99% 시 사실상 성능 저하 없음; ASID로 컨텍스트 스위치 최적화
- **페이지 테이블 구조**: 2단계(32비트), 해시(>32비트 희소 공간), 역(메모리 절약, 공유 제한)
- **스와핑**: 현대 시스템은 페이지 단위(page out/in); 모바일은 flash 제약으로 미지원
- **IA-32**: 세그멘테이션 + 2단계 페이징; PAE로 64GB 물리 메모리 지원
- **x86-64**: 4단계 페이징, 48비트 가상/52비트 물리 주소
- **ARMv8**: 4단계 페이징, 3가지 변환 단위, 2단계 TLB(마이크로+메인)
