## 1.1 운영체제란 무엇인가

### 시스템 관점의 계층 구조

```
┌─────────────────────────────┐
│         User(s)             │
├─────────────────────────────┤
│    Application Programs     │
│  (컴파일러, 브라우저, DB...)  │
├─────────────────────────────┤
│      Operating System       │  ← 자원 할당자 / 제어 프로그램
├─────────────────────────────┤
│    Computer Hardware        │
│   (CPU, Memory, I/O, ...)   │
└─────────────────────────────┘
```

OS는 크게 두 가지 관점으로 정의된다.

**사용자 관점 (User View)**: 편의성 극대화. 데스크톱/랩톱은 단일 사용자가 자원을 독점하므로 사용 편의와 성능에 초점. 임베디드 시스템처럼 사용자 인터페이스가 없는 경우, OS는 개입 없이 자율적으로 작동한다.

**시스템 관점 (System View)**: OS는 **자원 할당자(resource allocator)**이자 **제어 프로그램(control program)**이다. CPU 시간, 메모리, I/O 장치 등 희소 자원에 대한 충돌 요청을 조정하고, 잘못된 프로그램이 시스템에 해를 끼치지 못하도록 제어한다.

### 커널(Kernel)의 정의

가장 보편적으로 사용되는 OS의 정의: **항상 실행 중인 하나의 프로그램**. 이것이 커널이다.

- **커널**: 항시 실행되는 핵심 프로그램
- **시스템 프로그램 (system programs)**: 커널과 연관되어 있지만 커널의 일부는 아님
- **응용 프로그램 (application programs)**: 시스템 동작과 관련 없는 나머지 모든 프로그램

모바일 OS(iOS, Android)의 경우 커널 외에 **미들웨어(middleware)** — 데이터베이스, 멀티미디어, 그래픽 지원 프레임워크 — 도 OS의 일부로 본다.

## 1.2 컴퓨터 시스템 구성

현대 컴퓨터는 하나 이상의 CPU와 여러 **장치 컨트롤러(device controller)**가 공통 버스로 연결되어 있으며, 공유 메모리에 접근한다. 각 컨트롤러는 로컬 버퍼와 특수 목적 레지스터를 유지하고, OS는 각 컨트롤러마다 **장치 드라이버(device driver)**를 보유한다.

```
CPU ──┐
      │
GPU ──┤
      ├── System Bus ── Memory Controller ── RAM
USB ──┤
      │
Disk──┘
```

### 1.2.1 인터럽트 (Interrupts)

인터럽트는 하드웨어와 OS가 상호작용하는 핵심 메커니즘이다.

**동작 흐름**:
```
1. device driver가 device controller의 레지스터에 명령 로드
2. device controller가 동작 수행 (예: 키보드 문자 읽기)
3. 완료 시 device controller → CPU에 인터럽트 신호 전송
4. CPU는 현재 명령 실행 후 interrupt-request line 감지
5. 인터럽트 번호를 인덱스로 인터럽트 벡터(interrupt vector) 참조
6. 해당 인터럽트 서비스 루틴(ISR) 실행
7. 완료 후 중단된 계산 재개
```

**인터럽트 벡터(interrupt vector)**: 인터럽트 서비스 루틴의 주소 배열. 낮은 메모리 주소에 저장되며, 고유 인터럽트 번호로 인덱싱된다.

**인터럽트 체이닝(interrupt chaining)**: 실제 시스템에서는 장치 수가 벡터 크기를 초과하므로, 벡터의 각 원소가 핸들러 리스트의 헤드를 가리키는 구조를 사용한다.

**Intel 인터럽트 벡터 예시**:
| 벡터 번호 | 설명 |
|-----------|------|
| 0 | Divide error |
| 2 | NMI (Nonmaskable Interrupt) |
| 14 | Page fault |
| 32–255 | Maskable interrupts (장치 인터럽트) |

**인터럽트 종류**:
- **하드웨어 인터럽트**: 장치 컨트롤러가 system bus를 통해 CPU에 신호 전송
- **소프트웨어 인터럽트 (트랩, trap)**: 오류(division by zero, invalid memory access) 또는 **시스템 콜(system call)**에 의해 발생

**마스킹(masking)**: CPU는 2개의 인터럽트 요청 라인을 가진다.
- **비마스크형(NMI)**: 복구 불가능한 메모리 오류 등 예약
- **마스크형**: 임계 구역 실행 중 CPU가 끌 수 있음. 장치 컨트롤러가 서비스 요청에 사용

**인터럽트 우선순위**: 현대 CPU는 인터럽트 우선순위 레벨 시스템을 구현하여, 낮은 우선순위 인터럽트를 지연시키거나 높은 우선순위 인터럽트가 낮은 우선순위 처리를 선점할 수 있다.

### 1.2.2 저장 구조 (Storage Structure)

CPU는 메모리에서만 명령을 로드할 수 있다. 부트스트랩 프로그램은 휘발성인 RAM에 저장할 수 없어 **EEPROM** 같은 펌웨어에 저장된다.

**저장 장치 계층 (Storage Hierarchy)**:

```
  빠름 / 작음 / 비쌈
┌────────────────────┐  ← Registers (< 1KB, ~0.25ns)
├────────────────────┤  ← L1/L2/L3 Cache (< 16MB, ~0.5-25ns)
├────────────────────┤  ← Main Memory / RAM (< 64GB, ~80-250ns)
├────────────────────┤  ← NVM / SSD (< 1TB, ~25,000-50,000ns)
├────────────────────┤  ← HDD (< 10TB, ~5,000,000ns)
└────────────────────┘  ← Magnetic Tape (무제한, 매우 느림)
  느림 / 큼 / 쌈
```

- **Volatile storage**: 전원이 끊기면 내용 소실 → 단순히 "메모리"라 칭함
- **NVS (Nonvolatile Storage)**:
  - **Mechanical**: HDD, 광학 디스크, 자기 테이프 — 크고 저렴
  - **Electrical (NVM)**: Flash memory, SSD, FRAM, NRAM — 빠르고 비쌈

**폰 노이만 명령 실행 사이클**: fetch → decode → execute → store. 메모리 유닛은 주소만 보고 그 의미(명령인지 데이터인지)는 알 수 없다.

### 1.2.3 I/O 구조 (I/O Structure)

인터럽트 구동 I/O는 소량의 데이터 이동에는 적합하지만, 대용량 NVS I/O에는 오버헤드가 크다. 이를 해결하기 위해 **DMA(Direct Memory Access)**를 사용한다.

**DMA 동작**:
- 버퍼, 포인터, 카운터 설정 후 device controller가 CPU 개입 없이 데이터 블록 전체를 직접 메모리로 전송
- 블록당 인터럽트 1회 발생 (바이트당 1회 대신)
- CPU는 DMA 수행 중 다른 작업 가능

## 1.3 컴퓨터 시스템 아키텍처

### 1.3.1 단일 프로세서 시스템 (Single-Processor)

하나의 범용 CPU 코어를 가진 시스템. 키보드, 디스크, 그래픽 컨트롤러 같은 **특수 목적 프로세서**도 존재하지만, 이들은 제한된 명령어 집합만 실행하고 프로세스는 실행하지 않으므로 단일 프로세서 시스템에 해당한다.

### 1.3.2 멀티프로세서 시스템 (Multiprocessor)

현대 컴퓨터의 지배적 아키텍처.

**SMP (Symmetric Multiprocessing)**:
```
CPU0          CPU1
 ├─ Registers  ├─ Registers
 └─ L1 Cache   └─ L1 Cache
      └─────────────┘
            │
     Shared L2/L3 Cache
            │
      Shared Memory (via System Bus)
```
- 각 CPU가 동등한 역할(OS 기능 + 사용자 프로세스 모두 수행)
- N개 CPU → N개 프로세스 병렬 실행 가능
- 단점: 하나가 idle한 동안 다른 하나가 과부하 → 동적 자원 공유로 해결

**멀티코어 (Multicore)**: 단일 칩 위에 여러 코어. 칩 간 통신보다 칩 내 통신이 빠르고 전력 효율도 우수.

```
   Processor Chip
┌─────────────────────┐
│  Core0  │  Core1   │
│ ┌─────┐ │ ┌─────┐  │
│ │ L1  │ │ │ L1  │  │
│ └─────┘ │ └─────┘  │
│    └────┴────┘      │
│       Shared L2     │
└─────────────────────┘
```

**NUMA (Non-Uniform Memory Access)**:
```
CPU0─Memory0    CPU1─Memory1
    └─────────────────┘
         Interconnect
```
- 각 CPU가 자신의 로컬 메모리를 빠른 로컬 버스로 접근
- 원격 메모리 접근 시 interconnect를 거쳐 지연(NUMA penalty) 발생
- OS의 CPU 스케줄링과 메모리 관리가 NUMA penalty 최소화에 중요 (5.5.2절, 10.5.4절)
- 대규모 서버와 HPC 시스템에서 인기

### 1.3.3 클러스터 시스템 (Clustered Systems)

둘 이상의 독립된 시스템(노드)이 연결된 **느슨하게 결합된(loosely coupled)** 시스템. 각 노드는 일반적으로 멀티코어 시스템.

**목적**: 고가용성(high availability) — 하나의 노드 실패 시에도 서비스 지속.

- **비대칭 클러스터링(asymmetric)**: 하나는 active, 다른 하나는 hot-standby로 모니터링
- **대칭 클러스터링(symmetric)**: 모두 active 상태로 서로 모니터링. 자원 활용도 높음

**graceful degradation**: 살아있는 하드웨어 수준에 비례한 서비스 제공 능력.  
**fault tolerant**: 단일 컴포넌트 실패에도 완전한 동작 지속.

**DLM (Distributed Lock Manager)**: 병렬 클러스터에서 다수의 호스트가 동일 데이터에 동시 접근할 때 충돌 방지를 위한 접근 제어 및 잠금 관리.  
**SAN (Storage-Area Network)**: 여러 시스템이 스토리지 풀에 붙는 구조. 노드 장애 시 다른 노드가 즉시 인계 가능.

## 1.4 운영체제 동작 원리

OS는 이벤트 기반으로 동작한다. 이벤트는 항상 **인터럽트(하드웨어) 또는 트랩(소프트웨어)**으로 신호된다.

**부트 순서**:
```
1. 전원 인가 → 펌웨어의 Bootstrap program 실행
2. CPU 레지스터, 장치 컨트롤러, 메모리 초기화
3. OS 커널 로케이트 및 메모리 로드
4. 커널 실행 시작 → 시스템 데몬(daemon) 시작
   (Linux: systemd → 기타 데몬들)
5. 이벤트 대기 (완전히 부팅됨)
```

### 1.4.1 멀티프로그래밍과 멀티태스킹

**멀티프로그래밍(Multiprogramming)**: 메모리에 여러 프로세스를 동시에 유지하여 CPU 유휴 상태를 방지.

```
프로세스 A: [실행 → I/O 대기           → 실행]
프로세스 B:          [실행 → I/O 대기        → 실행]
프로세스 C:                    [실행 → 실행       ]
CPU:        [  A  ][  B  ][  C  ][  A  ][  B  ][  C  ]
```

**멀티태스킹(Multitasking)**: 멀티프로그래밍의 논리적 확장. CPU가 프로세스 간에 매우 빈번하게 전환하여 빠른 응답 시간 제공. 이것이 우리가 흔히 아는 **시분할(time-sharing)** 시스템의 기반.

멀티태스킹이 요구하는 OS 기능들:
- **메모리 관리**: 여러 프로세스를 메모리에 유지 (9, 10장)
- **CPU 스케줄링**: 어떤 프로세스를 실행할지 결정 (5장)
- **가상 메모리(virtual memory)**: 물리 메모리보다 큰 프로그램 실행 가능 (10장)
- **파일 시스템**: 정보의 영구 저장 (13, 14, 15장)
- **프로세스 동기화/통신**: (6, 7장)
- **데드락 방지**: (8장)

### 1.4.2 이중 모드와 멀티모드 동작 (Dual-Mode and Multimode Operation)

잘못된 프로그램으로부터 OS 자신과 다른 프로그램을 보호하기 위해 **모드 비트(mode bit)**로 실행 컨텍스트를 구분한다.

```
User mode (mode bit = 1)         Kernel mode (mode bit = 0)
┌───────────────────────┐        ┌──────────────────────────┐
│  User Application     │        │  OS Code                 │
│  (제한된 명령어만 실행) │        │  (모든 명령어 실행 가능)  │
└───────────┬───────────┘        └──────────────┬───────────┘
            │ system call / trap                 │
            └──────────────────►─────────────────┘
                                 ↑ mode bit = 0
            ◄─────────────────────────────────────
             return              mode bit = 1
```

**전환 흐름**:
1. 부팅 시: 하드웨어는 커널 모드로 시작
2. OS가 사용자 앱에 제어권을 줄 때: 모드 비트 → 1 (user mode)
3. 인터럽트/트랩/시스템 콜 발생: 자동으로 커널 모드 전환
4. 서비스 완료 후: 모드 비트 → 1 (user mode)로 복귀

**특권 명령어(privileged instructions)**: I/O 제어, 타이머 관리, 인터럽트 관리 등. 커널 모드에서만 실행 가능. 유저 모드에서 시도 시 하드웨어가 OS로 트랩.

**멀티레벨 보호링**:
- Intel: 4개 링 (Ring 0 = kernel, Ring 3 = user)
- ARMv8: 7개 모드
- 가상화 지원 CPU: VMM(Virtual Machine Manager)을 위한 별도 모드 (user < VMM < kernel)

**시스템 콜(system call)**: 사용자 프로그램이 OS에게 특권 서비스를 요청하는 메커니즘. 인터럽트 벡터의 특정 위치로 트랩 실행 → 소프트웨어 인터럽트로 처리.

### 1.4.3 타이머 (Timer)

OS가 CPU 제어권을 영구적으로 잃지 않도록 보장하는 메커니즘.

- 하드웨어 타이머가 지정된 주기(고정 또는 가변)마다 인터럽트 발생
- **가변 타이머**: 고정 클럭 + 카운터 조합으로 구현. OS가 카운터 설정 → 클럭 틱마다 카운터 감소 → 0이 되면 인터럽트

> **Linux HZ 파라미터**: 타이머 인터럽트 빈도 설정. `HZ=250`이면 4ms마다 인터럽트. `jiffies` 변수는 부팅 이후 누적 타이머 인터럽트 수를 추적.

타이머 카운터 수정 명령어는 **특권 명령어**이므로 유저 모드에서 변경 불가.

## 1.5 자원 관리

### 1.5.1 프로세스 관리 (Process Management)

**프로세스(process)**: 실행 중인 프로그램. 프로그램은 디스크에 저장된 수동적 개체(passive entity)이고, 프로세스는 능동적 개체(active entity)다.

- **단일 스레드 프로세스**: 단일 프로그램 카운터(PC)로 순차 실행
- **멀티스레드 프로세스**: 스레드당 별도의 PC 보유

OS의 프로세스 관리 책임:
- 사용자/시스템 프로세스 생성 및 삭제
- CPU에서의 프로세스 및 스레드 스케줄링
- 프로세스 일시 정지 및 재개
- 프로세스 동기화 메커니즘 제공
- 프로세스 간 통신(IPC) 메커니즘 제공

### 1.5.2 메모리 관리 (Memory Management)

CPU는 메모리에서만 직접 데이터를 읽고 쓸 수 있다. 프로그램 실행을 위해 절대 주소로 매핑 후 메모리에 로드되어야 한다.

OS의 메모리 관리 책임:
- 어떤 메모리 영역이 어떤 프로세스에 의해 사용 중인지 추적
- 메모리 공간 할당 및 해제
- 어떤 프로세스/데이터를 메모리에 올리고 내릴지 결정

### 1.5.3 파일 시스템 관리 (File-System Management)

OS는 물리적 저장 장치의 다양성을 추상화하여 논리적 저장 단위인 **파일(file)**을 제공한다.

OS의 파일 시스템 책임:
- 파일 및 디렉터리 생성/삭제
- 파일·디렉터리 조작 기본 연산 지원
- 파일을 저장 매체에 매핑
- 비휘발성 저장 매체에 백업

### 1.5.4 대용량 저장소 관리 (Mass-Storage Management)

HDD, NVM(SSD)이 주요 온라인 저장 매체. OS 책임: 마운트/언마운트, 자유 공간 관리, 저장 할당, 디스크 스케줄링, 파티셔닝, 보호.

2차 저장소보다 느리고 저렴한 3차 저장소(자기 테이프, 광학 디스크)는 백업용. 일부 OS는 3차 저장소 관리도 담당(마운트, 장치 독점 할당, 2차→3차 데이터 마이그레이션).

### 1.5.5 캐시 관리 (Cache Management)

**캐싱의 원리**: 필요한 정보를 더 빠른 저장소(캐시)로 임시 복사. 접근 시 캐시를 먼저 확인(hit) → 없으면 소스에서 가져와 캐시에 복사(miss).

**데이터 이동 예시 (integer A 증가 연산)**:
```
HDD → Main Memory → Cache → Register (읽기)
Register → Cache → Main Memory → HDD (쓰기)
```

**캐시 일관성(cache coherency)**: 멀티프로세서 환경에서 동일 데이터의 여러 캐시 사본이 최신 값을 가지도록 보장하는 문제. 일반적으로 하드웨어 수준에서 처리.

캐시 크기와 교체 정책(replacement policy)의 선택이 성능에 결정적 영향을 미친다 (10장).

계층별 저장소 특성 요약:
| 레벨 | 이름 | 크기 | 접근 시간 | 관리 주체 |
|------|------|------|----------|---------|
| 1 | Registers | < 1KB | 0.25-0.5ns | 컴파일러 |
| 2 | Cache | < 16MB | 0.5-25ns | 하드웨어 |
| 3 | Main Memory | < 64GB | 80-250ns | OS |
| 4 | SSD (NVM) | < 1TB | 25,000-50,000ns | OS |
| 5 | HDD | < 10TB | ~5,000,000ns | OS |

### 1.5.6 I/O 시스템 관리 (I/O System Management)

I/O의 특수성을 사용자로부터 숨기는 것이 OS의 목적 중 하나. Unix에서는 **I/O 서브시스템**이 I/O 장치의 특수성을 OS 대부분으로부터 격리한다.

I/O 서브시스템 구성:
- **메모리 관리 컴포넌트**: 버퍼링(buffering), 캐싱(caching), 스풀링(spooling)
- **일반 장치 드라이버 인터페이스**: 통일된 인터페이스 제공
- **특정 하드웨어 장치 드라이버**: 장치별 특수성 캡슐화

---

## 1.6 보안과 보호

**보호(Protection)**: 시스템이 정의한 자원에 대한 프로세스/사용자 접근을 제어하는 메커니즘. 인터페이스 오류를 조기에 검출하고 서브시스템 오염을 방지.

**보안(Security)**: 외부/내부 공격으로부터 시스템 방어. 바이러스, 웜, DoS(Denial-of-Service) 공격, 신원 도용 등 포함.

**사용자 식별**:
- **UID (User ID)** / Windows: **SID (Security ID)**: 사용자마다 고유한 숫자 ID. 모든 프로세스와 스레드에 연관됨
- **GID (Group ID)**: 사용자 집합에 대한 권한 부여
- **특권 상승(privilege escalation)**: Unix의 `setuid` 속성 → 파일 소유자의 UID로 프로그램 실행

## 1.7 가상화 (Virtualization)

가상화는 단일 컴퓨터의 하드웨어를 여러 실행 환경으로 추상화하여 각각이 독립된 컴퓨터에서 실행되는 것처럼 보이게 한다.

```
┌──────────┬──────────┬──────────┐
│  Guest   │  Guest   │  Guest   │
│  OS (W)  │  OS (L)  │  OS (M)  │
├──────────┴──────────┴──────────┤
│    VMM (Virtual Machine Mgr)   │
├────────────────────────────────┤
│         Host Hardware          │
└────────────────────────────────┘
```

**에뮬레이션(emulation)** vs **가상화(virtualization)**:
- 에뮬레이션: 소스 CPU 타입 ≠ 타겟 CPU 타입 (예: Apple의 Rosetta). 모든 명령어를 변환해야 하므로 성능 오버헤드가 큼
- 가상화: 소스 CPU = 타겟 CPU. 네이티브 컴파일된 OS가 다른 네이티브 OS 위에서 실행

**VMM (Virtual Machine Manager)**: 게스트 OS 실행, 자원 사용 관리, 게스트 간 격리. 현대 데이터센터에서는 VMware ESX, Citrix XenServer처럼 VMM 자체가 호스트 OS 역할.

---

## 1.8 분산 시스템 (Distributed Systems)

물리적으로 분리된 이기종 컴퓨터 시스템의 집합이 네트워크로 연결된 구조. 공유 자원에 대한 접근이 계산 속도, 기능, 데이터 가용성, 신뢰성을 높인다.

**네트워크 분류**:
- **LAN (Local-Area Network)**: 방, 건물, 캠퍼스 단위
- **WAN (Wide-Area Network)**: 도시, 국가 단위
- **MAN (Metropolitan-Area Network)**: 도시 내 건물 연결
- **PAN (Personal-Area Network)**: Bluetooth, 802.11 등 단거리 무선

**OS 관점의 분산 시스템**:
- **네트워크 OS**: 파일 공유 등 네트워크 기능 제공. 각 컴퓨터는 자율적으로 동작
- **분산 OS**: 단일 OS가 네트워크 전체를 제어하는 것처럼 보이는 덜 자율적인 환경

## 1.9 커널 자료 구조 (Kernel Data Structures)

### 1.9.1 리스트, 스택, 큐 (Lists, Stacks, Queues)

배열은 직접 접근이 가능하지만, 크기가 가변적이거나 순서를 유지하면서 삽입/삭제가 필요할 때는 **연결 리스트(linked list)**가 사용된다.

- **단일 연결 리스트 (singly linked list)**: 각 항목이 후계자를 가리킴
- **이중 연결 리스트 (doubly linked list)**: 이전/다음 모두 참조 가능
- **원형 연결 리스트 (circularly linked list)**: 마지막 원소가 첫 원소를 가리킴

연결 리스트 접근: O(n). 이를 기반으로 스택, 큐 등 고급 자료구조 구축.

**스택(Stack)**: LIFO. 함수 호출 시 파라미터, 지역 변수, 반환 주소 push → 반환 시 pop.

**큐(Queue)**: FIFO. 프린터 작업, CPU 스케줄링 대기 큐 등에 활용.

### 1.9.2 트리 (Trees)

계층적 데이터 표현. 커널에서 특히 중요한 것은 **레드-블랙 트리(red-black tree)**:
- Linux의 CPU 스케줄링 알고리즘(CFS)에서 사용 (5.7.1절)
- 자기 균형 이진 탐색 트리 → 최악의 경우에도 O(log n) 보장

### 1.9.3 해시 함수와 맵 (Hash Functions and Maps)

데이터를 입력받아 숫자 값으로 변환 → 테이블 인덱스로 사용. 이상적 경우 O(1) 접근.

**해시 충돌(hash collision)**: 서로 다른 입력이 같은 해시 값을 가질 때. 해당 위치에 연결 리스트를 두는 **체이닝(chaining)**으로 해결.

**해시 맵(hash map)**: [key:value] 쌍을 해시 함수로 연결. 사용 예: 사용자 이름 → 패스워드 매핑을 통한 인증.

### 1.9.4 비트맵 (Bitmaps)

n비트 문자열로 n개 자원의 상태를 표현. 공간 효율성이 뛰어남 (Boolean 배열 대비 1/8).

```
비트맵: 0 0 1 0 1 1 1 0 1
인덱스: 0 1 2 3 4 5 6 7 8
```
→ 자원 2, 4, 5, 6, 8은 사용 불가 / 0, 1, 3, 7은 가용

디스크 블록 가용성 관리 등에 활용.

> **Linux 커널 자료구조 구현 위치**:
> - 연결 리스트: `<linux/list.h>`
> - 큐 (kfifo): `kfifo.c`
> - 레드-블랙 트리: `<linux/rbtree.h>`

## 1.10 컴퓨팅 환경 (Computing Environments)

### 1.10.1 전통적 컴퓨팅 (Traditional Computing)

배치 시스템(batch)과 인터랙티브 시스템의 역사적 구분이 시분할(time-sharing)로 통합. 현재는 단일 사용자의 모든 프로세스가 시분할로 실행됨. 방화벽(firewall)은 네트워크 장치 간 통신을 제한.

### 1.10.2 모바일 컴퓨팅 (Mobile Computing)

스마트폰, 태블릿. 제한된 메모리, 프로세서, 전력이 특징. GPS, 가속도계, 자이로스코프 같은 고유 하드웨어 활용.

양대 모바일 OS: **Apple iOS** (iPhone/iPad), **Google Android** (다양한 제조사).

### 1.10.3 클라이언트-서버 컴퓨팅 (Client-Server)

- **컴퓨트 서버**: 클라이언트 요청에 따라 작업 실행 후 결과 반환 (예: DB 서버)
- **파일 서버**: 파일 시스템 인터페이스 제공 (예: 웹 서버)

### 1.10.4 P2P 컴퓨팅 (Peer-to-Peer)

클라이언트/서버 구분 없이 모든 노드가 동등한 peer. 서버 병목을 제거하고 여러 노드에 서비스를 분산.

서비스 발견 방식:
1. **중앙 집중형 룩업**: 중앙 서버에 서비스 등록 → 클라이언트가 서버에 질의
2. **브로드캐스트 방식**: 중앙 서버 없이 요청 브로드캐스트 → 해당 노드가 응답

### 1.10.5 클라우드 컴퓨팅 (Cloud Computing)

가상화를 기반으로 컴퓨팅, 저장소, 애플리케이션을 네트워크 서비스로 제공.

| 유형 | 설명 |
|------|------|
| Public Cloud | 인터넷을 통해 누구나 유료로 사용 (예: AWS EC2) |
| Private Cloud | 회사 내부용 |
| Hybrid Cloud | Public + Private 혼합 |
| SaaS | 소프트웨어 서비스 (워드프로세서, 스프레드시트) |
| PaaS | 플랫폼 서비스 (DB 서버 등) |
| IaaS | 인프라 서비스 (서버, 스토리지) |

### 1.10.6 실시간 임베디드 시스템 (Real-Time Embedded)

가장 광범위한 컴퓨팅 환경. 자동차, 가전, 제조 시스템 등. 정해진 시간 내 결과 보장이 필수적인 **실시간 OS(RTOS)**.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| **인터럽트(Interrupt)** | 하드웨어와 OS의 상호작용 핵심 메커니즘. 벡터 테이블로 핸들러 디스패치 |
| **DMA** | CPU 개입 없는 대용량 데이터 전송. 블록당 인터럽트 1회 |
| **이중 모드(Dual-mode)** | 모드 비트로 user/kernel 구분. 특권 명령어는 kernel mode만 실행 가능 |
| **시스템 콜(System call)** | 사용자 프로그램이 OS 서비스를 요청하는 트랩 메커니즘 |
| **SMP** | 대칭 멀티프로세싱. 모든 CPU가 동등하게 OS/사용자 작업 수행 |
| **NUMA** | 비균일 메모리 접근. 로컬 메모리 접근은 빠르나 원격 접근은 느림 |
| **멀티프로그래밍** | CPU 유휴 방지를 위해 메모리에 여러 프로세스 유지 |
| **캐시 일관성** | 멀티프로세서에서 여러 캐시의 동일 데이터 일관성 보장 (주로 HW 처리) |
| **가상화** | 단일 HW를 여러 실행 환경으로 추상화. VMM이 게스트 OS 관리 |
